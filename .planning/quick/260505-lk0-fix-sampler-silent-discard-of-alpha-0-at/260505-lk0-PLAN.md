<objective>
Fix sampler silent-discard of skin-declared attachments that no slot binding ever activates.

**TRUE bug shape (empirically confirmed against `fixtures/SAMPLER_ALPHA_ZERO/TOPSCREEN_ANIMATION_JOKER.json`):**

When a Spine animator declares an attachment in a skin's manifest but turns "visibility off" in setup pose (no `attachment` field on the slot def) AND never animates it, Spine encodes this as `slot.attachment = null` at setup with no animation timeline raising it. The skin's manifest still declares the binding (e.g. default skin: `JOKER-BG → JOKER/BG`, `JOKER-FRAME → JOKER/FRAME`) and the attachment is exported into the atlas with real source dims. But `slot.getAttachment()` returns `null` for these slots forever — because no setup-pose binding and no animation timeline ever activates them.

The current sampler walks `skeleton.slots` and skips any slot where `slot.getAttachment() === null` (sampler.ts:285). This guard is correct in isolation — slots with a genuinely-null binding at this frame have nothing to measure. But it misses skin-declared attachments that no slot binding ever activates. Those end up absent from `globalPeaks`, absent from `setupPosePeaks`, absent from the export plan, and absent from the optimized images folder. Cross-load asymmetry confirmed: same JSON loaded against atlas-source vs an optimized images folder reports different attachment sets.

**The user's locked principle:** Visibility (slot binding null OR alpha 0) is a runtime-mutable concern. Dev code, player actions, equipment swaps can bind any skin-declared attachment to any slot at any frame. Therefore any attachment the skin manifest declares must be measured for peak scale, optimization, and export — independent of whether any setup-pose binding or animation timeline activates it during sampling.

**Falsified prior plan:** The previous PLAN.md (which this overwrites) targeted the alpha gate at sampler.ts:291. Empirical probe of the JOKER fixture falsified that bug model — both `JOKER-BG` and `JOKER-FRAME` slot defs have NO `color` field (default alpha 1.0), so the alpha gate never fires. Removing it would change nothing. The real bug is upstream of slot iteration: the sampler never iterates skin-declared attachments that no slot binds.

**Fix shape:** Add a "skin-manifest pass" after the existing setup-pose snapshot, per skin. Iterate `skin.getAttachments()` (returns `SkinEntry { slotIndex, name, attachment }`); for each entry, if `globalPeaks` already has the key (existing path measured it via the live binding), defer; else measure the attachment using the existing `attachmentWorldAABB(slot, entry.attachment)` + `computeRenderScale(slot, entry.attachment)` helpers (both accept arbitrary attachment params per bounds.ts:59-92 and bounds.ts:148-184) and emit a `PeakRecord` keyed by `${skinName}/${slotName}/${entry.name}` with `animationName === SETUP_POSE_LABEL` and `isSetupPosePeak: true`. No slot mutation (`slot.setAttachment`); the helpers consume the passed attachment directly.

Purpose: restore the invariant that any attachment referenced by any skin manifest gets measured, optimized, and exported — independent of whether any animation or setup-pose binding makes it visible.

Output:
- A failing-then-passing regression spec under `tests/core/sampler-skin-defined-unbound-attachment.spec.ts` that asserts both `JOKER-BG/JOKER/BG` and `JOKER-FRAME/JOKER/FRAME` appear in `globalPeaks` and `setupPosePeaks` with correct dims and finite positive scale.
- A targeted source change in `src/core/sampler.ts` that adds the skin-manifest pass without touching the existing setup-pose pass, animation pass, or downstream consumers.
- All existing tests still pass; cross-load symmetry restored.
</objective>

Pseudocode (the executor refines the exact insertion point and naming):

```typescript
// === Pass 1.5 (per skin) — skin-manifest coverage ===
// Fixes silent-discard of skin-declared attachments that no slot binding ever
// activates (e.g. JOKER-BG / JOKER-FRAME in fixtures/SAMPLER_ALPHA_ZERO).
// See Plan 260505-lk0 (2026-05-05) — `slot.getAttachment() === null` skip at
// line 285 is correct for the live-binding pass; this pass fills the gap.
for (const entry of skin.getAttachments()) {
  const slot = skeleton.slots[entry.slotIndex];
  if (slot === undefined) continue;       // defensive (skin.bones may diverge from skeleton.slots in pathological exports)

  const attachment = entry.attachment;
  if (attachment === null) continue;      // defensive (SkinEntry shouldn't carry null, but guard anyway)

  const key = `${skin.name}/${slot.data.name}/${entry.name}`;
  if (globalPeaks.has(key)) continue;     // existing setup-pose pass already measured this binding — defer

  // attachmentWorldAABB / computeRenderScale accept arbitrary attachment params
  // (bounds.ts:59-92, bounds.ts:148-184). slot.bone world transforms are
  // already current from the Physics.pose call above.
  const aabb = attachmentWorldAABB(slot, attachment);
  if (aabb === null) continue;             // BoundingBox / Path / Point / Clipping — no source texture

  // Region-name resolution mirrors snapshotFrame's logic (sampler.ts:309-310):
  // prefer attachment.region.name (path indirection), fall back to entry.name.
  const regionName = (attachment as { region?: { name?: string } }).region?.name;
  const sd = load.sourceDims.get(regionName ?? entry.name);
  if (sd === undefined || sd.w <= 0 || sd.h <= 0) continue;

  const rs = computeRenderScale(slot, attachment);
  if (rs === null) continue;
  const { scale: peakScale, scaleX: peakScaleX, scaleY: peakScaleY } = rs;
  const worldW = aabb.maxX - aabb.minX;
  const worldH = aabb.maxY - aabb.minY;

  const record: PeakRecord = {
    attachmentKey: key,
    skinName: skin.name,
    slotName: slot.data.name,
    attachmentName: entry.name,
    animationName: SETUP_POSE_LABEL,
    time: 0,
    frame: 0,
    peakScaleX,
    peakScaleY,
    peakScale,
    worldW,
    worldH,
    sourceW: sd.w,
    sourceH: sd.h,
    isSetupPosePeak: true,
  };
  globalPeaks.set(key, record);
  setupPosePeaks.set(key, record);          // mirror snapshotFrame's setup-pose-pass dual write
  setupPoseBaseline.set(key, peakScale);    // unblocks the per-animation D-54 affected check
                                            // for any animation that later binds this attachment
}
// === end Pass 1.5 ===
```

**Insertion point:** Inside the per-skin loop at `for (const skin of load.skeletonData.skins)` (sampler.ts:164), AFTER the `snapshotFrame(...)` Pass 1 call (which currently runs at sampler.ts:173-186) and BEFORE the `for (const anim of ...)` Pass 2 loop (sampler.ts:189). The world transforms set up by the Physics.pose call at sampler.ts:172 are still current when the new pass runs.

**Defer rule (no double-emit):** The `if (globalPeaks.has(key)) continue;` check is the only deduplication. The existing setup-pose pass populates `globalPeaks` with the live binding (whatever `slot.getAttachment()` returned at setup); the new pass only emits for keys absent from `globalPeaks`, which is exactly the set of skin-declared attachments no setup-pose binding activated. This is structurally correct because Pass 1.5 runs strictly after Pass 1 within the same skin scope.

**Why setupPosePeaks dual-write?** The existing `snapshotFrame` writes to BOTH `globalPeaks` AND `setupPosePeaks` during Pass 1 (sampler.ts:328-370). The new pass mirrors this so the breakdown analyzer's D-60 "Setup Pose (Default)" top card includes the unbound attachments — they ARE visible only in the setup-pose semantic sense (they're declared in the manifest), and the user's locked principle says they belong on that row.

**Why `setupPoseBaseline` write?** If a future animation rebinds one of these attachments to its slot via `AttachmentTimeline`, the per-animation Pass 2 affected-check (sampler.ts:374-403) computes `scaleDelta = abs(peakScale - baseline)`. Without a baseline entry, that delta starts at `peakScale - 0 = peakScale` and the attachment becomes "affected by every animation" — false positive. Writing the baseline here ensures the affected-check works correctly for unbound attachments that animations later bind.

**What NOT to do (constraints from the planning context):**
- Do NOT remove the `slot.color.a <= 0` gate at sampler.ts:291 — the falsified prior plan targeted that line; the JOKER fixture has default white color, so the gate is irrelevant to this bug.
- Do NOT remove the `slot.getAttachment() === null` skip at sampler.ts:285 — it is correct for the live-binding pass.
- Do NOT modify `core/usage.ts`, `core/export.ts`, `main/summary.ts`, or any renderer code. The fix is sampler-only.
- Do NOT temporarily bind via `slot.setAttachment(attachment)` and restore — the helpers accept the attachment as a parameter; no slot mutation needed.
- Do NOT add migration logic. Fix forward; users re-optimize manually.
- Do NOT modify the per-animation Pass 2 — animation timelines that genuinely bind attachments still flow through the existing `slot.getAttachment()` path correctly.

**Performance:** Adds one `skin.getAttachments()` iteration per skin per `sampleSkeleton` call. For typical rigs (50-200 attachments) and the existing 120 Hz sampling cost (millions of `computeWorldVertices` calls), this is unmeasurable overhead.
