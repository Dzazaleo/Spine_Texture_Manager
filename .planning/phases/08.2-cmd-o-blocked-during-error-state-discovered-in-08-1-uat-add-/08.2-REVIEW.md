# Phase 08.2: Code Review Report

**Depth:** standard

## Summary

Phase 08.2 ships the native Electron application Menu, Open Recent persistence,
the `menu:notify-state` IPC handler with trust-boundary validation, and the
canonical 08.1 UAT regression fix (Cmd+O on the recovery banner). The primary
risk surfaces called out in the brief are well-handled:

- **D-181 trust boundary (`menu:notify-state`)**: src/main/ipc.ts:568-588
  validates payload as object, then field-by-field `typeof === 'boolean'` for
  all three fields before any mutation. Pattern parallels validateExportPlan
  and handleProjectSave field-validation idiom; silent rejection is correct
  for a one-way channel. `await import('./index.js')` deferral resolves
  the cycle cleanly.
- **Pitfall 9 listener-identity discipline (preload)**: All five new
  subscription bridges (onMenuOpen/onMenuOpenRecent/onMenuSave/onMenuSaveAs
  + the existing onCheckDirtyBeforeQuit/onExportProgress patterns) capture
  `wrapped` in a const before `ipcRenderer.on` and reference the same const
  in the unsubscribe closure. No anonymous-wrapper leaks.
- **Pitfall 15 cleanup discipline (App.tsx)**: All four menu-event subs
  return their unsubs from the cleanup function (App.tsx:259-264). The
  effect's dep array `[handleBeforeDrop, handleProjectLoad]` correctly
  triggers re-subscription when the bound deps change.
- **D-156 atomic-write + hand-rolled validator (recent.ts)**: validateRecentFile
  follows the version-FIRST gate idiom from validateProjectFile; atomic-write
  uses same-directory `.tmp` + `rename` to avoid EXDEV.
- **D-176 keydown listener deletion (AppShell.tsx)**: Confirmed gone end-to-end.
  AppShell.tsx:668-679 carries the explanatory comment block; no `useEffect`
  registers a `keydown` handler. test 8.2-MENU-06 locks the inverse.

The findings below are warnings around defense-in-depth gaps, pre-existing
file content (a `console.log` at App.tsx:211), and informational items
around code readability. No critical security or correctness issues found.

## Warnings

### WR-01: `addRecent` does not validate input is a `.stmproj` path

**File:** `src/main/recent.ts:128-134`
**Issue:** `addRecent(absolutePath)` accepts any string and persists it
verbatim to recent.json. The JSDoc explicitly defers validation to callers
("This module is a dumb store. See `<threat_model>` T-08.2-01-06"), and the
two production callers in project-io.ts (handleProjectSaveAs:177 and
handleProjectOpenFromPath:480) only fire on the `ok:true` arm of writes
that already passed `.endsWith('.stmproj')` checks. However, there is no
defense-in-depth check at the dumb-store boundary itself: if a future
caller in another module forgets the precondition, recent.json silently
fills with non-`.stmproj` entries which would then surface as broken
"Open Recent" menu items. Mirrors the trust-boundary discipline in
ipc.ts:568-576 (which DOES validate even though preload is "trusted").
**Fix:** Add a one-line guard to defend the on-disk format invariant:
```typescript
export async function addRecent(absolutePath: string): Promise<string[]> {
  if (typeof absolutePath !== 'string' || !absolutePath.endsWith('.stmproj')) {
    // Silent reject — caller contract violation; mirrors recent.json's
    // load-time silent failure (D-177 non-critical UX state).
    return loadRecent();
  }
  const current = await loadRecent();
  // ... rest unchanged
}
```

### WR-02: `clearRecent` callback inside `buildAppMenu` captures stale `mainWindowRef` indirectly

**File:** `src/main/index.ts:189-194`
**Issue:** The "Clear Menu" click handler in `buildAppMenu` calls
`void applyMenu(currentMenuState, mainWindowRef)` — it reads the
module-level `mainWindowRef` and `currentMenuState` at click time, which
is correct for the window handle but means the rebuilt menu uses
`currentMenuState` rather than the `state` argument that was passed
to this `buildAppMenu` invocation. In practice these are the same value
(the `applyMenu` caller always passes `currentMenuState`), so the rebuild
is consistent. However, `buildAppMenu` is exported and tests invoke it
directly with arbitrary state (menu.spec.ts cases a-g), and a future
caller passing a different `state` would see the rebuilt menu silently
fall back to module state — a subtle action-at-a-distance bug.
**Fix:** Use the closure-captured `state` argument so the rebuilt menu
reflects the same intent:
```typescript
click: async () => {
  await clearRecent();
  // Use the caller's state, not module state — keeps buildAppMenu's
  // contract consistent for direct callers (tests + future surfaces).
  void applyMenu(state, mainWindowRef);
},
```

### WR-03: AppShell `onClickOpen` / `onClickLocateSkeleton` no-op silently when other error kinds occur

**File:** `src/renderer/src/components/AppShell.tsx:598-630`, `643-666`
**Issue:** In `onClickOpen` (lines 624-627) and `onClickLocateSkeleton`
(lines 656-664, partial), error kinds OTHER than
`'SkeletonNotFoundOnLoadError'` are dropped on the floor with a silent
`return` and a comment "surface via existing error UI. The inline banner
pattern is the standard." But there is no inline banner state for these
generic errors — `skeletonNotFoundError` is the only inline banner state
slot, and it requires the kind to be `SkeletonNotFoundOnLoadError`. So a
real `ProjectFileParseError` or `ProjectFileVersionTooNewError` from a
toolbar Open click is silently swallowed. The user clicks Open, picks a
broken `.stmproj`, and nothing happens. The drag-drop path routes through
App.tsx which has the generic `error` AppState branch, so this is a
toolbar-only gap.
**Fix:** Either route the error through a transient toast/banner, or
re-throw to a generic error state. Quick fix using `console.error` plus
a TODO for Phase 9:
```typescript
if (resp.error.kind === 'SkeletonNotFoundOnLoadError') {
  setSkeletonNotFoundError({ /* ... */ });
  return;
}
// TODO Phase 9: surface non-recovery error kinds via a banner state.
// For now, log + alert so the user isn't left wondering why nothing
// happened.
console.error('[Open] failed:', resp.error);
window.alert(`${resp.error.kind}: ${resp.error.message}`);
return;
```

### IN-02: Three duplicated "extract dirname / extract basename" splits inline regex

**File:** `src/renderer/src/components/AppShell.tsx:523, 525, 557, 559, 795, 1020`; `src/main/index.ts:180`; `src/renderer/src/App.tsx:248`
**Issue:** The pattern `path.replace(/[\\/][^\\/]+$/, '') || '.'` and
`path.split(/[\\/]/).pop() ?? path` is repeated 6+ times across the
renderer and once in the main process. Cross-platform path-segment
handling is reasonable to inline once or twice, but at this density a
small renderer-side helper (`lib/path-display.ts` with `dirOf(p)` and
`basenameOf(p)`) would consolidate the regex literal and make the
"empty-segment fallback" rule explicit (`|| '.'` exists in some sites,
not in others — minor inconsistency).
**Fix:** Extract to `src/renderer/src/lib/path-display.ts`:
```typescript
export const basenameOf = (p: string): string => p.split(/[\\/]/).pop() ?? p;
export const dirOf = (p: string): string => p.replace(/[\\/][^\\/]+$/, '') || '.';
```

### IN-03: `getMainWindow` / `getCurrentMenuState` / `setCurrentMenuState` exported from `index.ts` purely for `import('./index.js')` cycle deferral

**File:** `src/main/index.ts:67-75`
**Issue:** Three exported accessors with single-line bodies exist purely
to give project-io.ts and ipc.ts something to dynamic-import without
pulling in BrowserWindow/Menu/etc. transitively. The pattern is correctly
documented (ipc.ts:55-73 carries the cycle-deferral rationale), but
extracting these three accessors plus the `MenuState` type into a tiny
`src/main/menu-state.ts` module would let project-io.ts and ipc.ts use
static imports, eliminate the dynamic-import error-swallow `try/catch`
blocks (project-io.ts:181-186 and 484-489), and improve testability —
the dynamic-import dance is the reason vi.mock blocks in test files
need both the recent.ts and index.ts mocks (project-io.spec.ts:38-53).
**Fix:** Extract module:
```typescript
// src/main/menu-state.ts
import type { BrowserWindow } from 'electron';

export type MenuState = {
  canSave: boolean;
  canSaveAs: boolean;
  modalOpen: boolean;
};

let currentMenuState: MenuState = {
  canSave: false,
  canSaveAs: false,
  modalOpen: false,
};
let mainWindowRef: BrowserWindow | null = null;
export const getMainWindow = (): BrowserWindow | null => mainWindowRef;
export const setMainWindow = (w: BrowserWindow | null): void => { mainWindowRef = w; };
export const getCurrentMenuState = (): MenuState => currentMenuState;
export const setCurrentMenuState = (next: MenuState): void => { currentMenuState = next; };
```
Then `applyMenu` lives in `index.ts` (still has Menu/BrowserWindow deps),
and project-io/ipc statically import the four accessors.

### IN-04: `BrowserWindow.getFocusedWindow()` may diverge from `mainWindowRef`

**File:** `src/main/project-io.ts:148, 266, 512`; `src/main/ipc.ts:309`
**Issue:** Save-As / Open / Locate-Skeleton dialogs use
`BrowserWindow.getFocusedWindow()` to attach as modal parent, while
menu click handlers route through `mainWindowRef`. In the single-window
app this is identical, but if a DevTools window steals focus, the dialog
might attach to DevTools rather than the main window. Low impact (DevTools
on its own pane is unusual to leave focused), but inconsistent with the
new `mainWindowRef` discipline this phase introduces.
**Fix:** Phase 8.2 introduces `getMainWindow()`; consider migrating the
three dialog sites in a follow-up:
```typescript
const win = getMainWindow() ?? BrowserWindow.getFocusedWindow();
```

### IN-05: `validateRecentFile` reuses input array reference rather than rebuilding

**File:** `src/main/recent.ts:70`
**Issue:** Returns `{ ok: true, file: { version: 1, paths: obj.paths as string[] } }` — the `paths` array is the same reference as the parsed JSON's array. Since `JSON.parse` produces fresh objects this is safe (callers can mutate without affecting on-disk state until the next `addRecent`). Documenting precedent: `validateProjectFile` likely does the same thing per the JSDoc claim of "mirrors validateProjectFile (D-156)". No bug, but a `paths.slice()` would be a tiny defense-in-depth against a future caller passing in a synthesized object whose `paths` array is shared with another consumer.
**Fix:** Optional hardening:
```typescript
return { ok: true, file: { version: 1, paths: [...obj.paths as string[]] } };
```
