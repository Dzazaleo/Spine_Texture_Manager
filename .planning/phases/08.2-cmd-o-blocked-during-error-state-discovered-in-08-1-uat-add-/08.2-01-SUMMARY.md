# Phase 08.2 Plan 01: Open Recent Persistence (recent.ts) Summary

A 145-line pure-main module that manages `app.getPath('userData')/recent.json` for the File → Open Recent submenu, lifting the atomic-write idiom from `project-io.ts` and the hand-rolled validator from `project-file.ts` (D-156).

## What was built

`src/main/recent.ts` — a tiny, robust JSON store for the Open Recent affordance, with five exports:

1. **`loadRecent(): Promise<string[]>`** — reads `recent.json`, returns `[]` on any failure (missing file, malformed JSON, unknown/newer version, non-array paths, non-string elements). Silent per D-177 — must not block app launch.
2. **`addRecent(absolutePath): Promise<string[]>`** — dedupes existing copy → unshifts new path to front → caps at 10 entries (D-178) → atomic write (D-180). Returns the new array so callers can refresh menu without a second read.
3. **`clearRecent(): Promise<void>`** — writes `{ version: 1, paths: [] }` atomically. Wired by Plan 02 to the File → Open Recent → Clear Menu item.
4. **`validateRecentFile(input): { ok: true; file: RecentFileV1 } | { ok: false }`** — hand-rolled type guard mirroring `validateProjectFile` (D-156). Version-FIRST gate (D-151 discipline) — unknown OR newer versions both return `{ ok: false }`.
5. **`type RecentFileV1`** — `{ version: 1; paths: string[] }` on-disk schema.

`tests/main/recent.spec.ts` — 9 cases across 5 describe blocks driving RED → GREEN through Tasks 1 and 2:

- **loadRecent (D-177)**: cases (a) ENOENT → `[]`, (b) malformed JSON → `[]`, (c) valid v1 → paths.
- **addRecent (D-178, D-180)**: (d) push-to-front + dedupe, (e) cap at 10 / oldest evicted, (f) atomic-write `<path>.tmp` + `rename`.
- **clearRecent**: (g) writes `{ version: 1, paths: [] }`.
- **validateRecentFile**: (h) rejects missing version / `version: 2` / non-array paths / non-string elements / accepts valid v1.
- **hygiene**: (i) reads `src/main/recent.ts` at test time and asserts only `app` is named-imported from `'electron'`.
