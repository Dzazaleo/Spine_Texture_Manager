## Menu strategy / Cmd+O fix

### Q1 — Where does the Cmd+O / Cmd+S accelerator actually live?

| Option | Description | Selected |
|--------|-------------|----------|
| Native Electron Menu (Recommended) | Menu.buildFromTemplate in src/main/index.ts; accelerator fires regardless of renderer state | ✓ |
| Renderer keydown only, lifted to App.tsx | Move AppShell listener up; defers File menu to later phase | |
| Both — native menu + App.tsx keydown | Defense in depth | |

**User's choice:** Native Electron Menu (Recommended) — codified as **D-173**.

### Q2 — How rich should the application Menu template be?

| Option | Description | Selected |
|--------|-------------|----------|
| Full standard macOS template (Recommended) | App / File / Edit / View / Window / Help with role-based items | ✓ |
| File menu only | Replaces default; loses Cmd+C/V in inputs, no Quit/About surface | |
| File + Edit only | Minimal subset for input ergonomics | |

**User's choice:** Full standard macOS template (Recommended) — codified as **D-174**.

### Q3 — How does a menu click reach the actual handler?

| Option | Description | Selected |
|--------|-------------|----------|
| Menu → IPC to renderer → renderer triggers existing flow (Recommended) | Reuses existing renderer state machine; menu is just another trigger | ✓ |
| Menu → main calls handler directly → push state to renderer | Distributed logic; harder to share dirty-guard | |

**User's choice:** Menu → IPC to renderer (Recommended) — codified as **D-175**.

### Q4 — What happens to the existing AppShell.tsx:626-643 keydown listener?

| Option | Description | Selected |
|--------|-------------|----------|
| Remove it — menu is single source of truth (Recommended) | Delete the keydown block; modal-suppression moves to notifyMenuState | ✓ |
| Keep it as a fallback | Two paths to same action; risk of double-firing | |

**User's choice:** Remove it (Recommended) — codified as **D-176**.

## Menu item enabled/disabled rules + dirty-guard

### Q1 — How does main learn renderer state to enable/disable menu items?

| Option | Description | Selected |
|--------|-------------|----------|
| Renderer pushes menu-state IPC on change (Recommended) | window.api.notifyMenuState({canSave, canSaveAs, modalOpen}) on change | ✓ |
| Always enabled; renderer validates and no-ops if invalid | Visually misleading on macOS | |
| Two static menu templates: 'no project' and 'project loaded' | Less granular; can't capture modalOpen | |

**User's choice:** Renderer pushes menu-state IPC (Recommended) — codified as **D-181**.

### Q2 — Exact rules for Save / Save As enabled state?

| Option | Description | Selected |
|--------|-------------|----------|
| Save enabled when summary loaded (no dirty check); Save As enabled when summary loaded (Recommended) | Save when clean is no-op (Photoshop/Pages parity); avoids per-override IPC churn | ✓ |
| Save enabled only when isDirty AND summary loaded; Save As enabled when summary loaded | Cleaner visual but extra IPC | |
| Both always enabled when summary loaded; Save As also enabled with no project | Edge case | |

**User's choice:** Save enabled when summary loaded; no dirty check (Recommended) — codified as **D-182**.

### Q3 — When Cmd+O / File→Open / File→Open Recent fires while session is dirty, what happens?

| Option | Description | Selected |
|--------|-------------|----------|
| Same SaveQuitDialog gate as drag-drop/toolbar Open (Recommended) | Reuses Phase 8 D-143 + Phase 8.1 D-163 onBeforeDropRef | ✓ |
| No gate — menu Open just opens | Data-loss footgun; diverges | |
| Gate only Cmd+O; toolbar/menu Open skip | Inconsistent across entry points | |

**User's choice:** Same SaveQuitDialog gate (Recommended) — codified as **D-183**.

### Q4 — When a modal is open, how should menu items behave?

| Option | Description | Selected |
|--------|-------------|----------|
| Disable Open / Open Recent / Save / Save As while any modal is open (Recommended) | modalOpen flag in notifyMenuState; Edit menu stays enabled | ✓ |
| Keep menu items enabled; modals defensively no-op | Visually misrepresents | |
| Suppress only Save/Save As during modals; allow Open | Risks losing unsubmitted modal input | |

**User's choice:** Disable all four while modal open (Recommended) — codified as **D-184**.

## Save As shortcut + Win/Linux visibility

### Q1 — Save As keyboard shortcut?

| Option | Description | Selected |

**User's choice:** Cmd+Shift+S (Recommended) — codified as **D-185**.

### Q2 — On Windows/Linux, autoHideMenuBar (currently true)?

| Option | Description | Selected |
|--------|-------------|----------|
| Keep autoHideMenuBar: true — Alt-key reveals menu (Recommended) | Cleaner chrome; toolbar already covers visible affordances | ✓ |

**User's choice:** Keep autoHideMenuBar: true (Recommended) — codified as **D-186**.

### Q3 — When the app is in projectLoadFailed (App.tsx recovery banner), what menu state applies?

| Option | Description | Selected |
|--------|-------------|----------|
| Open / Open Recent enabled; Save / Save As disabled (Recommended) | Recovery banner is the bug Phase 08.2 fixes; Cmd+O must work | ✓ |
| Same as 'no project loaded' state | Equivalent | |
| Disable everything until banner is dismissed | Contradicts phase goal | |

**User's choice:** Open enabled, Save disabled (Recommended) — codified as **D-187**.

### Q4 — Should this phase also touch Edit menu shortcuts (Cut/Copy/Paste/Undo/Redo)?

| Option | Description | Selected |
|--------|-------------|----------|
| Use Electron's role-based Edit menu — ships standard shortcuts for free (Recommended) | { role: 'editMenu' }; zero extra code; auto-platform-correct | ✓ |

**User's choice:** Role-based Edit menu (Recommended) — codified as **D-188**.
