## Summary

This phase delivers UI polish: a new cool blue-dark color scheme, full-width panel layout, toolbar height harmonization, danger-themed headers on problem-zone panels, and an atlas-less chip summary. The CSS and most JSX changes are straightforward. One pre-existing logic bug was exposed by the review: `onClickOptimize` and `onConflictPickDifferent` in `AppShell.tsx` use the stale `summary` prop instead of `effectiveSummary` when building the export plan, meaning after a Settings-driven resample the Optimize dialog operates on outdated peak data. Several dead-code and quality issues are also present.

## Critical Issues

### CR-01: Optimize dialog builds export plan from stale `summary` prop instead of `effectiveSummary`

**File:** `src/renderer/src/components/AppShell.tsx:512` and `:623`

**Issue:** `onClickOptimize` (line 512) and `onConflictPickDifferent` (line 623) both call `buildExportPlan(summary, overrides)` using the `summary` prop. `effectiveSummary` is defined at line 239 as `localSummary ?? summary`, where `localSummary` is set to the fresh `SkeletonSummary` after a Settings-driven resample (line 1128). When the user changes the sampling rate in Settings and then clicks Optimize Assets, `effectiveSummary` has the new peaks but `onClickOptimize` still builds the plan from the stale prop peaks. The resulting Optimize dialog shows wrong source dims, wrong peak dims, and incorrect pixel-area savings. `onConflictPickDifferent` (line 623) has the identical bug for the re-pick flow. By contrast, `savingsPctMemo` (line 704) correctly uses `effectiveSummary` — the inconsistency is clear evidence of the bug.

**Fix:**
```tsx
// AppShell.tsx line 511-514
const onClickOptimize = useCallback(async () => {
  const plan = buildExportPlan(effectiveSummary, overrides);  // was: summary
  setExportDialogState({ plan, outDir: lastOutDir });
}, [effectiveSummary, overrides, lastOutDir]);               // was: summary

// AppShell.tsx line 622-626
const plan = buildExportPlan(effectiveSummary, overrides);   // was: summary
setExportDialogState({ plan, outDir: newOutDir });
setLastOutDir(newOutDir);
// in the useCallback deps array at line 626: effectiveSummary  (was: summary)
```

// DELETE this line:

### WR-02: `closeBothDialogs` useCallback is defined but never invoked

**File:** `src/renderer/src/components/AppShell.tsx:577-631`

**Issue:** `closeBothDialogs` is created via `useCallback` (line 577-581) and allocates a new closure on every render. The only reference to it in the module body is `void closeBothDialogs;` at line 631, which is a bare void-expression that reads the variable to suppress a lint warning — the function is never actually called. The comment at line 627 explains the original intent but the function was superseded by `onConflictCancel` and `onConflictOverwrite`. Dead callbacks created with `useCallback` are not free — React still runs the memoization bookkeeping on every render.

**Fix:** Remove the function and the `void` guard:
```tsx
// DELETE lines 577-631 (closeBothDialogs useCallback + void reference)
// The ConflictDialog's three handlers (onConflictCancel, onConflictPickDifferent,
// onConflictOverwrite) fully cover the close-both-dialogs logic.
```

### WR-03: `unused` RowState variant is dead code in both table panels

**File:** `src/renderer/src/panels/GlobalMaxRenderPanel.tsx:172-179`, `src/renderer/src/panels/AnimationBreakdownPanel.tsx:174-182`

**Issue:** The `RowState` union type includes `'unused'`, `rowState()` checks `if (isUnused) return 'unused'`, and both the left-accent bar `<span>` and the tinted ratio `<td>` have `state === 'unused' && 'bg-danger'` / `'bg-danger/10 text-danger'` branches — but every callsite passes `isUnused=false` unconditionally:

- `GlobalMaxRenderPanel.tsx:949`: `rowState(row.effectiveScale, false, row.isMissing)`
- `GlobalMaxRenderPanel.tsx:1064`: `rowState(row.effectiveScale, false, row.isMissing)`
- `AnimationBreakdownPanel.tsx:845`: `rowState(row.effectiveScale, false, row.isMissing)`
- `AnimationBreakdownPanel.tsx:890`: `rowState(row.effectiveScale, false, row.isMissing)`

The `'unused'` row-level accent color was designed to flag attachments that appear in the rig's attachments map but are not referenced by any animation (the "images-folder-vs-JSON orphan" concept). That cross-reference is never computed and passed into `rowState()`, so the visual branch never fires. The `UnusedAssetsPanel` is the current dedicated surface for orphaned files.

**Fix (option A — remove dead code):** Delete the `'unused'` variant from `RowState`, the `if (isUnused)` branch from `rowState()`, and the two `state === 'unused'` clsx branches in each panel if the row-level visual is not planned.

**Fix (option B — wire it up):** If row-level "unused" coloring in the Global panel is intended, pass `orphanedFiles` to `GlobalMaxRenderPanel`, build a `Set<string>` of orphaned filenames, and call `rowState(row.effectiveScale, orphanedSet.has(row.attachmentName), row.isMissing)`.

### WR-04: `onQueryChange` prop is received by `AnimationBreakdownPanel` but never called

**File:** `src/renderer/src/panels/AnimationBreakdownPanel.tsx:120,305,316`

**Issue:** `onQueryChange: (q: string) => void` is declared in `AnimationBreakdownPanelProps` (line 120), destructured in the component signature (line 305), and the comment at line 316 notes it "is never invoked from this panel." The prop exists solely to satisfy the contract shared with `GlobalMaxRenderPanel`, but it is never called — any caller who passes a non-trivial `onQueryChange` will silently receive no callbacks from this panel. If the panel's intent is to never call it, the prop should either be removed from the interface or marked with a comment at the props level (not just in the function body) so callers are not misled.

**Fix:** Either remove the prop from the interface (AppShell at line 1570 already passes `setQuery` and the panel ignores it — no runtime breakage from removal), or document it explicitly in the interface JSDoc:
```tsx
/**
 * ...
 * NOTE: This panel never calls onQueryChange — the sticky-bar SearchBar in
 * AppShell is the sole write path. The prop is retained for interface symmetry
 * with GlobalMaxRenderPanel only.
 */
onQueryChange: (q: string) => void;
```

## Info

### IN-01: `key={f.filename}` in UnusedAssetsPanel uses basename, not a unique path

**File:** `src/renderer/src/panels/UnusedAssetsPanel.tsx:115`

**Issue:** The table row key is `f.filename` — defined in `OrphanedFile` as "PNG basename without the .png extension." If the images folder has subdirectories, two files in different subdirectories can share the same basename (e.g. `characters/hero.png` and `props/hero.png` both yield `filename: "hero"`). React would emit a duplicate-key warning and may render only one of the two rows. In practice Spine's `images/` folder is flat, but the type does not guarantee uniqueness.

**Fix:** Use the full path if available, or derive a synthetic key:
```tsx
// Option: use index as tie-breaker (only safe because this list is not reordered)
filteredOrphans.map((f, idx) => (
  <tr key={`${f.filename}-${idx}`} className="border-b border-border">
```
Or, if `OrphanedFile` can carry a `fullPath` field in the future, use that instead.

### IN-02: `role="alert"` on static-content warning panels interrupts screen readers

**File:** `src/renderer/src/panels/UnusedAssetsPanel.tsx:51`, `src/renderer/src/panels/MissingAttachmentsPanel.tsx:57`

**Issue:** Both panels use `role="alert"` on their outermost `<div>`. ARIA `alert` is a live region that causes screen readers to interrupt whatever they are currently reading and announce the element's content immediately on mount. These panels render at skeleton-load time alongside a full screen of other content, so the alert interruption will trample over the primary load announcement (skeleton filename, peak count). `role="status"` (polite live region) or no live-region role would be more appropriate for informational warnings that do not require immediate action.

**Fix:**
```tsx
// MissingAttachmentsPanel.tsx line 57 — change role to status:
<div
  role="status"
  aria-label="Missing attachment PNGs"
  ...
>

// UnusedAssetsPanel.tsx line 51 — same change:
<div
  role="status"
  aria-label="Orphaned image files"
  ...
>
```

### IN-03: Tab strip does not satisfy the WAI-ARIA Tabs pattern

**File:** `src/renderer/src/components/AppShell.tsx:1327-1340`, `:1538-1573`

**Issue:** `TabButton` has `role="tab"` and `aria-selected`, and the `<nav>` has `role="tablist"`. However the tab panels (the conditional `GlobalMaxRenderPanel` / `AnimationBreakdownPanel` blocks) have no `role="tabpanel"`, and the tabs have no `aria-controls` attribute linking them to their panels. Screen reader users cannot navigate from a tab button to its panel content via standard keyboard shortcuts (Arrow + Tab key in AT virtual-mode) because the semantic link is missing.

**Fix:** Add an `id` to each tab button and a matching `role="tabpanel"` + `aria-labelledby` to each content container:
```tsx
// TabButton — add id prop and aria-controls:
<button
  id={`tab-${isActive ? 'global' : 'animation'}`}
  aria-controls={`panel-${...}`}
  role="tab"
  ...
>

// Panel wrappers — add role and aria-labelledby:
{activeTab === 'global' && (
  <div role="tabpanel" id="panel-global" aria-labelledby="tab-global">
    <GlobalMaxRenderPanel ... />
  </div>
)}
```
