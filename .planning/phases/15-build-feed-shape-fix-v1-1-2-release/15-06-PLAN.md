**The new URL-resolution invariant (load-bearing pre-flight gate):**

Plan 15-04 Task 5 verified sha512+size byte-for-byte against local files but did NOT HEAD/GET the constructed URL against GitHub's stored asset name — that's the verification gap D-15-LIVE-1 surfaced. Plan 15-06 Task 1 (D-07 Gate 1, extended) adds:

```bash
# For each url in release/latest-mac.yml, verify the dotted URL resolves
# to an existing local file (the spaces↔dots reverse-mapping):
node -e "
const yaml = require('js-yaml');
const fs = require('fs');
const y = yaml.load(fs.readFileSync('release/latest-mac.yml', 'utf8'));
for (const f of y.files) {
  // Reverse the sanitizeAssetUrl substitution: dots → spaces in the productName segment.
  // The url has dots; the local file has spaces. The transformation is a 1:1 bijection,
  // so reversing it must produce a file that exists in release/.
  const localCandidate = f.url.replace(/\\./g, ' ').replace(/(\\.zip|\\.dmg|\\.exe|\\.AppImage)\$/, m => m.replace(/ /g, '.'));
  // Better: directly check that some file in release/ when sanitized matches f.url:
  const releaseFiles = fs.readdirSync('release/');
  const matchingLocal = releaseFiles.find(name => name.replace(/ /g, '.') === f.url);
  if (!matchingLocal) {
    console.error('FAIL: emitted url', f.url, 'has NO corresponding local file in release/ (after space→dot map)');
    process.exit(1);
  }
  console.log('OK:', f.url, '↔', matchingLocal);
}
"
```

(Exact JavaScript may be cleaner — see Task 1 action; the principle is: every emitted url MUST round-trip back to a real file via the inverse sanitization.)

CI dry run extends this with a HEAD against published assets to catch any GitHub-side rename divergence we haven't predicted.

Plan 15-06's release-body authoring or any future plan could conceivably add commits after; using main HEAD is robust).

**For v1.1.3 specifically:** the chore(15-05) commit (Plan 15-05 Task 3) is currently the last commit on main. So tagging at `main` and tagging at chore(15-05) would produce the same SHA in this case. BUT: the plan must use `git rev-parse main`, not a hardcoded SHA, because:
1. If anything additional gets committed to main between Plan 15-05 close and Plan 15-06 tag (e.g., the user lands an unrelated docs commit, or a Plan 15-05 SUMMARY commit), the hardcoded SHA would exclude it.
2. `git rev-parse main` is self-documenting and reproducible.
3. AP-1 codifies this as the universal rule for future Phase-15-style release plans.

The published `latest-mac.yml` in v1.1.2 advertised installer urls with literal spaces; GitHub Releases stores assets with the spaces replaced by dots; electron-updater 6.x sanitized spaces to dashes when constructing the request URL — three different forms, none of them agreeing. v1.1.2's installed mac client cannot auto-update to v1.1.3 (its broken url construction will still 404). v1.1.0 / v1.1.1 final mac clients auto-update to v1.1.3 cleanly.

## New in this version

- **macOS auto-update fix:** synthesizer (`scripts/emit-latest-yml.mjs`) now emits installer urls in the GitHub-canonical dotted form, agreeing byte-for-byte with the asset names GitHub stores at upload time. Squirrel.Mac swap on Download & Restart succeeds end-to-end.
- **Pre-flight URL-resolution invariant gate:** every emitted `files[].url` is now round-trip-verified against the local `release/` filesystem before tag push, AND verified against the actually-stored GitHub asset names after CI publish. The gate that would have caught v1.1.2's defect is now in place.
- (No user-facing UI changes — this is a release-engineering hotfix.)

## Known issues

### Stranded testers — manual download required

If you installed `v1.1.0-rc1`, `-rc2`, or `-rc3`: please download `v1.1.3` manually from the assets list below — after upgrading, all future auto-updates work normally. (Root cause: electron-updater 6.x channel-name comparison treats `rc1`/`rc2`/`rc3` as opaque tokens because they lack a dot before the number. Convention documented in [CLAUDE.md `## Release tag conventions`](https://github.com/Dzazaleo/Spine_Texture_Manager/blob/main/CLAUDE.md#release-tag-conventions).)

If you installed `v1.1.2` on macOS: please download `v1.1.3` manually from the assets list below — v1.1.2's auto-update was broken (HTTP 404 on the .zip download due to a 3-way filename mismatch between local artifact, GitHub-stored asset, and electron-updater request URL). v1.1.2's installed client cannot reach v1.1.3 automatically. After manually upgrading to v1.1.3, all future auto-updates work normally. (Root cause: D-15-LIVE-1; fix landed in v1.1.3 via `scripts/emit-latest-yml.mjs` `sanitizeAssetUrl()` — synthesizer now emits feed urls in the GitHub-canonical dotted form.)
