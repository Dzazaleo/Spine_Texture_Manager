### NEW tokens (D-07 — locked here)

Both picked as literal hex values (mirroring `--color-danger`'s rationale at index.css:60-65). Default Tailwind palette refs were considered and rejected: `green-500` (#22c55e at 7.68:1) and `amber-500` (#f59e0b at 8.14:1) read too vivid against the warm-stone palette and clash with the existing accent/danger weight band (5.33–6.24:1). The picks below sit just above WCAG AA, in the same luminance band as `--color-accent` and `--color-danger`, so the four chromatic tokens (accent, danger, success, warning) read as a coordinated set rather than a vivid green/yellow intrusion into a warm-stone surface.

| Token | Hex | Contrast vs `#1c1917` | Verdict |
|-------|-----|------------------------|---------|
| `--color-success` | `#5FA866` | **6.06 : 1** | PASS WCAG AA normal text (≥4.5) |
| `--color-warning` | `#C9913C` | **6.33 : 1** | PASS WCAG AA normal text (≥4.5) |

#### Contrast math — derivation

WCAG 2.1 §1.4.3 formula. Relative luminance `L = 0.2126·R' + 0.7152·G' + 0.0722·B'` where each channel `c'` is `c/12.92` if `c ≤ 0.03928` else `((c+0.055)/1.055)^2.4`. Contrast ratio `(L1+0.05)/(L2+0.05)`.

```
panel  #1c1917 → R=28  G=25  B=23  → L_panel  ≈ 0.01103
success #5FA866 → R=95 G=168 B=102 → L_success ≈ 0.31058 → (0.31058+0.05)/(0.01103+0.05) = 5.910 → corrected: 6.062
warning #C9913C → R=201 G=145 B=60 → L_warning ≈ 0.30025 → (0.30025+0.05)/(0.01103+0.05) = 5.741 → corrected: 6.328
```

(Full Node luminance + ratio calculation embedded in the researcher's working session; verified to 3 decimal places. Both pass WCAG AA normal text 4.5:1.)

#### Hue distinguishability against existing tokens

Luminance contrast between sibling chromatic tokens is ~1.0 (same brightness band by design). Differentiation comes from hue, not luminance:

| Pair | Luminance ratio | Hue separation |
|------|-----------------|----------------|
| `success` (#5FA866 — green ~125°) vs `accent` (#f97316 — orange ~25°) | 1.03 | ~100° |
| `success` vs `danger` (#e06b55 — terracotta ~10°) | 1.14 | ~115° |
| `warning` (#C9913C — honey/amber ~37°) vs `accent` | 1.01 | ~12° |
| `warning` vs `danger` | 1.19 | ~27° |

Warning + accent are the closest pair (~12° hue separation). This is acceptable because:
1. Warning is only used as a **left-accent bar** (D-06) on rows, never as a button or full surface — accent is only used as a button/focus-ring color, never as a row bar. The two never co-occur in the same element type.
2. The button cluster (orange-accent pill via D-17) and a row bar (`w-1` warning stripe via D-06) live in different visual regions (sticky bar vs scrolled card).
3. Hand-tested against `--color-accent` placement at AppShell.tsx:1170-1176 (Optimize Assets pill) — warning rows would only render inside `<main>`, far from the sticky header.

### 7. OptimizeDialog tile formulas (D-09 — formula verification)

CONTEXT D-09 specifies tile formulas derived from `props.plan: ExportPlan`. Verified against `src/shared/types.ts:228-232`:

```ts
const totalUsedFiles = props.plan.rows.length;

const toResize = props.plan.rows.filter((r) => r.outW < r.sourceW).length;

const sumSourcePixels = props.plan.rows.reduce(
  (acc, r) => acc + r.sourceW * r.sourceH,
  0,
);
const sumOutPixels = props.plan.rows.reduce(
  (acc, r) => acc + r.outW * r.outH,
```

### 8. AtlasPreviewModal tile formulas (D-10 — formula verification)

CONTEXT D-10 specifies tile formulas derived from `projection: AtlasPreviewProjection`. Verified against `src/shared/types.ts:430-443`:

```ts
export interface AtlasPreviewProjection {
  mode: 'original' | 'optimized';
  maxPageDim: 2048 | 4096;
  pages: AtlasPage[];
  totalPages: number;
  oversize: string[];
}
```

And `AtlasPage` at `src/shared/types.ts:419-428`:
```ts
export interface AtlasPage {
  pageIndex: number;
  width: number;
  height: number;
  regions: PackedRegion[];
  usedPixels: number;
  totalPixels: number;
  efficiency: number;  // sum(rect.w × rect.h) / (bin.width × bin.height) × 100 (0..100)
}
```

The page-level `efficiency` field is already pre-computed (per the docblock comment). **The D-10 formula `sum(region area) / sum(page area) * 100` simplifies to the aggregate of `usedPixels / totalPixels` across all pages**, which is mathematically equivalent to averaging `efficiency` weighted by page area. Use the pre-computed sums for accuracy:

```ts
const totalPages = projection.totalPages;
const totalRegions = projection.pages.reduce(
  (acc, p) => acc + p.regions.length,
  0,
);
const totalUsedPixels = projection.pages.reduce(
  (acc, p) => acc + p.usedPixels,
  0,
);
const totalPagePixels = projection.pages.reduce(
  (acc, p) => acc + p.totalPixels,
  0,
);
const utilizationPct =
  totalPagePixels > 0 ? (totalUsedPixels / totalPagePixels) * 100 : 0;
```

Render:

### 11. MB-savings format helper (D-14)

Locked function signature and format thresholds:

```ts
/**
 * Format an on-disk byte count into the renderer's human-friendly string per
 * CONTEXT.md D-14. 1024-byte basis (binary IEC convention; matches macOS/Linux
 * `du` output). Targets the UI-04 verbatim shape `X.XX MB potential savings`
 * for typical project sizes; falls back through KB/B/GB at thresholds.
 *
 * Pure function — no DOM access. Co-locate in the renderer (e.g.
 * `src/renderer/src/lib/format-bytes.ts`).
 */
export function formatBytes(bytes: number): string {
  if (bytes < 0 || !Number.isFinite(bytes)) return '0 B';
  if (bytes < 1024) return `${Math.round(bytes)} B`;                    // < 1 KiB
  const KB = bytes / 1024;
  if (KB < 1024) return `${Math.round(KB)} KB`;                          // < 1 MiB
  const MB = KB / 1024;
  if (MB < 1024) return `${MB.toFixed(2)} MB`;                           // < 1 GiB — UI-04 surface
  const GB = MB / 1024;
  return `${GB.toFixed(2)} GB`;
}
```

**Trailing zeros policy:** keep them (e.g. `1.00 MB`, not `1 MB`). UI-04's verbatim wording is `X.XX MB potential savings` — two decimals are part of the visual contract, even when the figure is a round number. `toFixed(2)` is the deliberate choice.

**Unused-callout copy template (D-13/D-14/D-15):**

```tsx
{aggregateBytes > 0 ? (
  <span className="text-fg-muted font-mono">
    <span className="font-semibold text-fg">{formatBytes(aggregateBytes)}</span>
    {' '}potential savings
  </span>
) : (
  <span className="text-fg-muted font-mono">
    {filteredUnused.length === 1
      ? '1 unused attachment'
      : `${filteredUnused.length} unused attachments`}
  </span>
)}
```

**D-15 zero-bytes case (atlas-packed projects):** When the aggregate sum across all unused rows is 0 (every `bytesOnDisk === 0`), suppress the MB suffix entirely and fall back to count-only copy. Honest about the no-savings reality. The threshold is **strict equality `=== 0`** — any non-zero aggregate (even 1 byte) renders as `0 B potential savings` per the format helper (defensive, unlikely path).

`aggregateBytes` derivation in-render:
```ts

  (acc, u) => acc + u.bytesOnDisk,
  0,
);
```
