<objective>
Restructure the partition logic in `buildExportPlan` (`src/core/export.ts` + its byte-identical mirror `src/renderer/src/lib/export-view.ts`) so the passthrough/resize decision is made AFTER override resolution and AFTER cap math, using the simple post-evaluation predicate `outW === sourceW AND outH === sourceH`. Delete the vestigial Phase 22 D-04 REVISED `peakAlreadyAtOrBelowSource` branch (now dead code after plan 22.1-01's unified actualSource model). Add an `isCapped?: boolean` field to ExportRow to surface the cap-binding signal to the OptimizeDialog row (consumed by plan 22.1-05).

Purpose: This closes G-07 (BLOCKER) — currently the partition decision is computed in the dedup accumulator BEFORE final outW/outH (and BEFORE override resolution), so a 50% user override on a passthrough row is silently dropped. Animators routinely override glow/blend-mode attachments well below the calculated optimum; that workflow is currently broken at the export layer. It also closes G-04 — the predicate generalizes to TRIANGLE-style no-drift peakScale=1.0× rows that are currently Lanczos-resampled wastefully.

Output:
- src/core/export.ts: Acc interface trimmed (drops isPassthrough; gains isCapped); emit loop computes isPassthrough from outW/outH; vestigial cap branch deleted with comment; ExportRow.isCapped emitted.
- src/renderer/src/lib/export-view.ts: Byte-identical mirror of all the above.
- src/shared/types.ts: ExportRow gains `isCapped?: boolean`.
- tests/core/export.spec.ts: 3+ new test cases for G-04 + G-07 BLOCKER (both directions).
</objective>
