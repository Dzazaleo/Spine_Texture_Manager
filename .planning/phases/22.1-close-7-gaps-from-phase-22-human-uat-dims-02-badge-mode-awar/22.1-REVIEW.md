## Critical (CR-)

### CR-001: `computeExportDims` in export-view.ts does NOT apply the override source-ratio conversion, diverging from buildExportPlan

**File:** `src/renderer/src/lib/export-view.ts:168-171`

**Finding:**
`computeExportDims` computes `rawEffScale` for the override path as:
```ts
const rawEffScale =
  override !== undefined
    ? applyOverride(override).effectiveScale  // ← just percent/100
    : peakScale;
```

But `buildExportPlan` (both in `src/core/export.ts:176-178` and its byte-identical copy in `export-view.ts:229-232`) applies the source-ratio adjustment for drifted rows:
```ts
const rawEffScale =
  overridePct !== undefined
    ? applyOverride(overridePct).effectiveScale * (row.sourceW / (row.canonicalW ?? row.sourceW))
    : row.peakScale;
```

`computeExportDims` is called from `enrichWithEffective` in both panels to compute the "Peak W×H" column values shown to the user, and the docblock at `export-view.ts:121-136` claims it is a "single source of truth" whose output matches `buildExportPlan`. However, on drifted rows (where `sourceW !== canonicalW` because 22.1-01 set `sourceW = actualSourceW`), the override path in `computeExportDims` omits the `sourceW/canonicalW` multiplier. This means the Peak W×H column in both panels will display incorrect dims whenever a user sets an override on a drifted attachment — the displayed value will differ from what the export actually produces.

**Impact:** Users see wrong export dimensions in the panel "Peak W×H" column for overridden drifted attachments. They believe they are exporting at N×N pixels but the actual export uses a different size. This is a correctness failure in the user-facing display: the panel lies to the user for the exact rows that have the most complexity (both drifted AND overridden).

**Fix:**
```ts
// In computeExportDims, pass canonicalW/canonicalH alongside the existing
// actualSourceW/actualSourceH params, then mirror the buildExportPlan formula:
export function computeExportDims(
  sourceW: number,
  sourceH: number,
  peakScale: number,
  override: number | undefined,
  actualSourceW?: number,
  actualSourceH?: number,
  dimsMismatch?: boolean,
  canonicalW?: number,   // ADD: needed for override source-ratio adjustment
  canonicalH?: number,   // ADD: (unused in dim calc but keeps parity)
): { effScale: number; outW: number; outH: number } {
  const rawEffScale =
    override !== undefined
      ? applyOverride(override).effectiveScale * (sourceW / (canonicalW ?? sourceW))
      : peakScale;
  // ... rest unchanged
}
```
Then update callers in both panels to pass `row.canonicalW` and `row.canonicalH`. The existing cap math using `actualSourceW / sourceW` (where `sourceW === canonicalW` per the 22.1-01 unified model comment) remains correct once the override is also adjusted.

## Warning (WR-)

### WR-001: `safeScale` (ceil-thousandth) is NOT applied in `computeExportDims`, breaking the "parity with buildExportPlan" contract

**File:** `src/renderer/src/lib/export-view.ts:172`

**Finding:**
`computeExportDims` applies:
```ts
const downscaleClampedScale = Math.min(safeScale(rawEffScale), 1);
```
This matches the `buildExportPlan` formula for the case without an override. However, for the override case in `buildExportPlan`, `safeScale` is also applied:
```ts
const downscaleClampedScale = Math.min(safeScale(rawEffScale), 1);
```
where `rawEffScale = applyOverride(overridePct).effectiveScale * (row.sourceW / (row.canonicalW ?? row.sourceW))`.

In `computeExportDims`, the `rawEffScale` for the override case is `applyOverride(override).effectiveScale` (already a round fraction like `0.5`). `safeScale(0.5) = ceil(0.5 * 1000)/1000 = 0.5`. So for exact percentages this is a no-op. However, for the combined override+drift case (once CR-001 is fixed and the source-ratio multiplier is restored), `rawEffScale` can be an irrational decimal where `safeScale` makes a difference. The panels would then show different dims than the actual export at the sub-pixel level. This is a latent precision discrepancy that becomes visible once CR-001 is fixed.

**Impact:** Sub-pixel discrepancy between panel display and actual export output for overridden drifted rows. Currently hidden by CR-001 but will manifest once CR-001 is fixed.

**Fix:** No additional change needed if CR-001 is fixed correctly — the same `safeScale(rawEffScale)` is already applied via `downscaleClampedScale` in `computeExportDims`. The concern only applies to the source-ratio multiplier correction from CR-001.

### WR-002: Dead/unreachable code branch in `loader.ts` emits confusing comment ("defense-in-depth") for a structurally impossible path

**File:** `src/core/loader.ts:376-388`

**Finding:**
The branch at lines 376-388 checks `if (opts.loaderMode === 'atlas-less')` inside the D-05 fall-through path (reached only when `opts.atlasPath === undefined` AND `opts.loaderMode !== 'atlas-less'`). The comment acknowledges the branch is "structurally unreachable via the 4-way loader branch order (D-08 at line 303 intercepts loaderMode:'atlas-less' before this D-05 fall-through is reached)."

The comment is accurate — the condition `opts.loaderMode === 'atlas-less'` can never be true at that point because the `else if (opts.loaderMode === 'atlas-less')` branch at line 305 handles it first. Dead code that is structurally guaranteed unreachable carries zero risk of firing, but it muddies static analysis and can mislead maintainers into thinking the branch is a live code path. If the branch order ever changes (as the comment warns), it will fire — but it was placed to address that exact scenario, making this an intentional defense-in-depth. The concern is that TypeScript's flow analysis cannot flag it, so a future refactor may not notice it and remove it, leaving orphan logic.

**Impact:** Maintenance burden; no runtime consequence today. If the branch order changes and this branch becomes reachable, the error it throws (`MissingImagesDirError`) is correct for the scenario, so it would not produce wrong behavior.

**Fix:** Add a TypeScript assertion to make the intent explicit:
```ts
// This assertion documents that the condition is structurally unreachable.
// If TypeScript ever flags this as reachable, re-examine the branch order.
const _never: never = opts.loaderMode as never; // remove if loaderMode can be 'atlas-less' here
```
Or alternatively, simply remove the dead branch and rely on the existing D-08 intercept, with a comment pointing at line 305.

### WR-004: DimsBadge tooltip `tooltipId` is not unique when the same attachment appears multiple times (e.g., two panels or two cards)

**File:** `src/renderer/src/components/DimsBadge.tsx:45`

**Finding:**
```ts
const tooltipId = `dims-badge-tooltip-${row.attachmentName}`;
```
The docblock states "tooltipId must be unique per rendered DimsBadge instance to support multiple badges on the same page without ARIA collision." However, the same attachment can legitimately appear more than once — for example, `SQUARE` appears in the Global Max panel AND in every animation card in the Animation Breakdown panel. When both tabs are simultaneously rendered (they are not — the tab switch unmounts the inactive panel), or more practically when there are multiple animation cards all showing `SQUARE` in their rows, each `DimsBadge` for `SQUARE` produces the same `id="dims-badge-tooltip-SQUARE"`. This violates the HTML uniqueness constraint for `id` attributes and causes ARIA `describedby` to be ambiguous (the browser matches the first or last element with that `id`).

**Impact:** ARIA accessibility violation when multiple `DimsBadge` instances for the same attachment are simultaneously mounted. With the virtualized list this may rarely occur in practice (only one card is fully expanded at a time), but for short lists below the virtualization threshold all rows in all expanded cards are in the DOM simultaneously.

**Fix:** Use a ref-based counter or incorporate a more specific namespace:
```ts
const tooltipId = `dims-badge-tooltip-${row.attachmentKey}`;
```
`attachmentKey` (e.g., `default::SQUARE::SQUARE`) is unique per skin+slot+attachment combination and is available on every `DisplayRow` and `BreakdownRow`. This resolves the collision without any structural changes.

### WR-005: `validateExportPlan` in `ipc.ts` does not validate `passthroughCopies` array

**File:** `src/main/ipc.ts:280-299`

**Finding:**
`validateExportPlan` validates `plan.rows` element-by-element but does not check `plan.passthroughCopies`:
```ts
function validateExportPlan(plan: unknown): string | null {
  if (!plan || typeof plan !== 'object') return 'plan is not an object';
  const p = plan as { rows?: unknown; excludedUnused?: unknown; totals?: unknown };
  if (!Array.isArray(p.rows)) return 'plan.rows is not an array';
  if (!Array.isArray(p.excludedUnused)) return 'plan.excludedUnused is not an array';
  if (!p.totals || typeof p.totals !== 'object') return 'plan.totals is not an object';
  for (let i = 0; i < p.rows.length; i++) {
    const r = p.rows[i] as Record<string, unknown>;
    if (
      typeof r.sourcePath !== 'string' || r.sourcePath.length === 0 ||
      // ...
    ) {
      return `plan.rows[${i}] has invalid shape`;
    }
  }
  return null;
}
```
Since Phase 22.1 added `passthroughCopies` as a required field of `ExportPlan`, the trust-boundary check should also validate that field's presence and array-ness. A renderer (or attacker) sending a plan with `passthroughCopies` as a non-array would cause a runtime TypeError in `runExport` when the image-worker tries to iterate it.

**Impact:** A malformed or malicious renderer payload with `passthroughCopies` absent or not an array causes an unhandled runtime exception inside `handleStartExport`'s `runExport` call, which is caught by the outer `catch (err)` block and returned as a generic `kind: 'Unknown'` error. No data loss or security escalation — the error is surfaced to the renderer. However, the T-01-02-01 trust-boundary validation contract is incomplete.

**Fix:**
```ts
// After the existing p.totals check:
if (!Array.isArray((p as { passthroughCopies?: unknown }).passthroughCopies)) {
  return 'plan.passthroughCopies is not an array';
}
```

### IN-001: `isCapped` flag semantics vs. the dedup update: subtle edge case is structurally impossible but worth documenting

**File:** `src/core/export.ts:226-230` and `src/renderer/src/lib/export-view.ts:280-284`

**Finding:**
When dedup promotes a new winner (higher `effScale`), the new winner's `isCapped` replaces the accumulated `isCapped`. The promotion condition is `effScale > prev.effScale`. Because both `effScale` values are computed as `Math.min(downscaleClampedScale, sourceRatio)`, if the new `effScale` is higher than the previous, it means the new row's natural scale was higher AND it still exceeded the cap (otherwise it would equal or be below `sourceRatio`). Therefore, the new winner being capped implies `isCapped = true`, and `isCapped` after promotion is always correct. This was confirmed by analysis; noting it here for maintainer awareness.

**Fix:** A comment near the dedup update confirming the structural soundness would help future readers:
```ts
// isCapped from the new winner is always correct:
// if new effScale > prev effScale, the winner's natural scale was also higher,
// meaning it either also exceeded sourceRatio (→ isCapped=true), or the cap
// is inert (Infinity) for both rows (→ isCapped=false). No inconsistency is possible.
prev.isCapped = isCapped;
```

### IN-003: `fixture-atlas-source-drift.mjs` copies only file-level entries, silently skipping subdirectories

**File:** `scripts/fixture-atlas-source-drift.mjs:52-58`

**Finding:**
```js
for (const entry of fs.readdirSync(SOURCE)) {
  const src = path.join(SOURCE, entry);
  const dst = path.join(DEST, entry);
  if (fs.statSync(src).isFile()) {
    fs.copyFileSync(src, dst);
  }
  // directories are silently skipped
}
```
If the `SIMPLE_PROJECT/` fixture directory ever gains a subdirectory (e.g., an `images/` subfolder for combined atlas+images layout, or a nested skin folder), the script will silently omit it and the drifted fixture will be incomplete. The current fixture has no subdirectories, so this is not a current bug.

**Fix:** Use `fs.cpSync(SOURCE, DEST, { recursive: true })` (Node 16.7+, compatible with the `engines.node >= 18` requirement in `package.json`):
```js
fs.cpSync(SOURCE, DEST, { recursive: true });
// Then mutate the atlas file...
```
This replaces the manual loop entirely.

### IN-004: CR-002 withdrawn — `isCapped` dedup promotion is structurally sound

**File:** `src/core/export.ts:225-234` and `src/renderer/src/lib/export-view.ts:280-288`

**Finding:**
An initial hypothesis that `isCapped` could be wrong after dedup promotion was investigated and disproved. When a new row wins on `effScale > prev.effScale`, the scenario where the new winner has `isCapped = false` but `effScale` higher than a previous `isCapped = true` row is structurally impossible: if the new row's natural scale exceeds `sourceRatio` (the cap), it would also be capped. If it does not exceed `sourceRatio`, its `effScale` cannot exceed the previous capped row's `effScale = sourceRatio`. No action required. See IN-001 for the explanatory comment suggestion.

**Fix:** No code change required.
