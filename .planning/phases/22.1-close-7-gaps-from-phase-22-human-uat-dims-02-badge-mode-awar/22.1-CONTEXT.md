When `peakAlreadyAtOrBelowSource && !override-pushes-above-source`, render only the first sentence ("Source PNG (W×H) is smaller than canonical region dims (W×H)." in atlas-less; "Atlas region declares W×H but canonical is W×H." in atlas-source). - **G-04 (major)** — Generalized passthrough predicate. Replace the Phase 22 D-04 REVISED `dimsMismatch && (isCapped || peakAlreadyAtOrBelowSource)` formula with the simpler `outW === sourceW AND outH === sourceH` (after cap, post-override-resolution). ## Implementation Decisions

### G-01 Atlas-source mode behavior (LOCKED 2026-05-03)
- **D-01:** **B-strict + unified actualSource model.** In atlas-source mode the loader does NOT read `images/` PNG headers. `actualSourceW/H` is populated from `atlas.region.originalWidth/Height` (which spine-core 4.2 auto-backfills from `region.width/height` when no `orig:` line is present — see `loader.ts:424-435`). `sourcePaths` is NOT populated from `images/` in atlas-source mode; export pulls from atlas page extracts via the existing `image-worker.ts:148-162` fallback path. Result: drift is detected when atlas declares smaller orig than JSON canonical (TexturePacker scale<1 scenario); no spurious badge when atlas matches canonical but `images/` has stale shrunk PNGs.
- **D-02:** **Mode-aware tooltip wording (G-01 UX side).** Atlas-source variant: "Atlas region declares W×H but canonical is W×H. Optimize will cap at on-disk size." Atlas-less variant: "Source PNG (W×H) is smaller than canonical region dims (W×H). Optimize will cap at source size." (unchanged from Phase 22). Both variants gate the second sentence on `isCapped` per G-03 cap-binding-aware copy.
- **D-03:** **Load-time rotation rejection (G-01b).** Atlas-source mode scans `atlas.regions` for `region.degrees !== 0` during `loadCanonical`; on first match throws a typed `RotatedRegionUnsupportedError` BEFORE skeleton load completes. Renderer surfaces "Rotated atlas regions are not supported. Re-export from Spine with rotation disabled." User-stated invariant lock: rotated atlas regions are non-negotiable for this app's contract.

### G-02 Tooltip implementation (LOCKED 2026-05-03)
- **D-04:** **Replace native `title` with custom CSS or React-managed tooltip primitive.** Native `title` flake root cause is the combination of small hit area (w-4 h-4) + virtualizer remount + browser idle-trigger heuristic. New tooltip surfaces deterministically on every hover, allows multi-line content for cap-binding-aware wording (G-03), tested at 100/125/150% zoom and in dark mode (Phase 22 deferred items lift here). Implementation choice (CSS-only vs portal-mounted React) is Claude's discretion — pick whichever stays Layer-3-clean for the renderer side and avoids new heavy dependencies. No new tooltip library required if a small inline implementation suffices.

### G-07 Override semantics + partition (LOCKED 2026-05-03)
- **D-05:** **Override % = % of source-on-disk (REVISES Phase 22 D-02).** Phase 22 D-02 wording said "% of canonical" but code already interprets override as `effScale × row.sourceW`, and after G-01 D-01 `row.sourceW === row.actualSourceW` in BOTH modes (atlas-source: atlas region orig; atlas-less: PNG header). So 50% override on a drifted SQUARE (actual 670, canonical 1000) produces `outW = ceil(670 × 0.5) = 335`, NOT 500. This matches user intent and the existing math. **No code change to override math** — only D-02 semantic label revised. Canonical dims are purely informational (badge tooltip + drift detection only).
- **D-06:** **Generalized partition predicate POST-override (G-04 + G-07 unified).** Replace Phase 22 D-04 REVISED `dimsMismatch && (isCapped || peakAlreadyAtOrBelowSource)` with the simpler `isPassthrough = (outW === sourceW AND outH === sourceH)` evaluated AFTER cap and AFTER override resolution. Coverage: (a) TRIANGLE no-drift peakScale=1.0× → outW = sourceW → passthrough ✓; (b) drifted row re-Optimized without override → sampler computes new peakScale against shrunk source, naturally produces outW ≤ sourceW, ≤ 1.0 clamp + ceil math gives outW = sourceW → passthrough ✓; (c) drifted row with 50% override → outW = 335 ≠ 670 → resize ✓; (d) drifted row with 100% override → outW = sourceW → passthrough ✓. The Phase 22 D-04 REVISED predicate becomes vestigial after G-01's unified model (cap math reduces to no-op since `sourceW === actualSourceW` everywhere); simplification is safe.
- **D-07:** **Cap-binding signal surfaced in OptimizeDialog row.** When the cap clamps an override (or a natural peakScale would exceed source ratio), surface this in the OptimizeDialog row label too — not just on the panel badge. Exact rendering is Claude's discretion (e.g. small "(capped)" suffix, secondary badge, muted icon next to the dim label). User's directive: "I'd like to see in the optimization row as well. If it's too noisy we can remove it later." Keep it simple; revisit during HUMAN-UAT if it adds visual noise.

<specifics>
## Specific Ideas

- **The unified model is the big simplification.** Phase 22 had implementation-leaky semantics (override = "% of canonical" which only made sense in atlas-source mode where sourceW === canonical). Phase 22.1 makes the model mode-uniform: sourceW === actualSourceW in BOTH modes (atlas-source: from atlas region orig; atlas-less: from PNG header). Override is "% of source-on-disk" everywhere. Cap math becomes vestigial. Partition predicate simplifies. Badge predicate works without mode-gating.
- **TexturePacker scale<1 is the legitimate atlas-source drift case.** User's screenshot showed TexturePacker scale=0.67. The atlas region.originalWidth in that case = the scaled-down dim (670, not canonical 1000), because TexturePacker without "Strip whitespace" enabled does NOT emit `orig:` lines, and spine-core auto-backfills `originalWidth = packedWidth`. Drift is detectable from the .atlas file alone — no PNG read required.
- **The user's original UAT bug scenario (full-scale atlas + manually-shrunk images/) is FIXED by G-01 B-strict.** With atlas-source mode no longer reading images/, the stale shrunk PNGs in images/ are simply ignored — no spurious badge, no spurious cap, no spurious byte-copy of stale data.
- **Override semantic clarification matters more than code change.** Phase 22 D-02's "% of canonical" wording was misleading; users would expect 50% override on a drifted SQUARE (actual 670, canonical 1000) to produce 500×500, but the existing math actually produces 335×335 (= 50% of 670). Phase 22.1 D-05 corrects the documented semantic to match the actual code behavior. No `.stmproj` migration needed.
- **Rotation rejection moves left.** Existing Optimize-time rotation error becomes load-time rejection (G-01b D-03). User explicitly stated rotated atlas regions are non-negotiable for this app's contract — fail fast at load.
- **G-04 + G-07 are coupled.** The generalized predicate (G-04) and post-override partition (G-07) are the same restructuring — both about moving the partition decision to "after final outW/outH is known, including override." Plan them together.
- **G-02 tooltip primitive enables G-03.** Native `title` attribute can't render multi-line cap-binding-aware copy elegantly. The custom tooltip primitive unblocks the G-03 wording variants AND fixes the hover-flake AND lifts the Phase 22 deferred dark-mode/zoom UAT items.
- **Pre-flight is the user's last sanity check.** G-06 D-08 (source→target dim shape on passthrough rows) + G-07 D-07 (cap-binding signal in OptimizeDialog row) both serve this principle: the dialog should let the user verify override outcomes before clicking Start.
