### `electron-builder.yml` (M) — config / data

**Analog:** self. The file already has a `publish:` block (lines 19-23) and already has list-of-objects shape (`mac.target`, `win.target`, `linux.target`, `files`, `asarUnpack`). The 12.1 edit replaces the publish-block VALUE with `null` (single token, no nested keys) and APPENDS an `extraResources:` top-level key with one entry.

**Current state to replace** (lines 10-23):
```yaml
# Phase 12 (Plan 12-02): GitHub publish provider declared so electron-builder
# generates `app-update.yml` into bundled resources/ at build time. Without this
# block, electron-updater errors at runtime ("app-update.yml is missing") —
# see RESEARCH §Common Pitfalls (GH issues #8620 + #2667).
# The `--publish never` CLI suffix on build:mac/build:win/build:linux scripts
# (Phase 11 D-01 lock) is preserved as belt-and-braces — this YAML block ONLY
# enables the bundling of `app-update.yml`; it does NOT trigger auto-publish to
# GitHub at build time. Auto-update detection happens at runtime against the
# published GitHub Releases feed, populated by .github/workflows/release.yml.
publish:
  provider: github
  owner: Dzazaleo
  repo: Spine_Texture_Manager
  releaseType: release
```

**Target state** (per RESEARCH §Pattern 1 + §Code Examples):
```yaml
# Phase 12.1: revert Phase 12's publish: github block (it caused per-artifact
# upload race in CI — see 12.1-RESEARCH.md §Summary). Static app-update.yml
# under build/ is bundled via extraResources to satisfy electron-updater's
# runtime requirement that the file exists in resources/.
publish: null

extraResources:
  - from: build/app-update.yml
    to: app-update.yml
```

**`extraResources:` schema reference** — already present in this file's spirit (the `files:` block at lines 32-49 and the `asarUnpack:` block at lines 51-54 use the same `- glob` list shape). The `from`/`to` object form is the documented variant for renaming during copy (`[CITED: electron.build/configuration.html — extraResources is a glob relative to project directory; copies into Contents/Resources (mac) or resources/ (win+linux)]`).

**What to copy / what to change:**

| Element | Source | Action |
|---------|--------|--------|
| The whole `publish:` block (lines 19-23) | self | DELETE the 4 nested lines, replace with `publish: null` (single line) |
| Comment block (lines 10-18) | self | REWRITE to reflect the 12.1 fix (revert + extraResources) |
| `extraResources:` top-level key | NEW | INSERT after `publish: null` (or at any top-level position; YAML key ordering is convention-only) |
| `--publish never` CLI suffix on `build:mac/win/linux` scripts | `package.json` (already in place) | LEAVE ALONE — preserved per RESEARCH §Pitfall 1 |

**Anti-pattern guard (per RESEARCH §Pitfall 1):** After the edit, `grep "publish:" electron-builder.yml` MUST return exactly `publish: null` and nothing else. Any leftover `publish: github` re-introduces the race.

### `tests/integration/install-md.spec.ts` (M — 1 new `test()` block)

**Analog:** self. The existing file has 18 `test()` blocks across 4 `describe()` blocks; the existing `'Each screenshot file is recognized as PNG (magic bytes)'` test at lines 109-127 is the byte-for-byte template for the new test.

**Existing test to mirror** (lines 109-127):
```typescript
test('Each screenshot file is recognized as PNG (magic bytes)', () => {
  const expected = [
    'docs/install-images/macos-gatekeeper-open-anyway.png',
    'docs/install-images/windows-smartscreen-more-info.png',
    'docs/install-images/windows-smartscreen-run-anyway.png',
    'docs/install-images/linux-libfuse2-error.png',
  ];
  // PNG magic bytes: 0x89 0x50 0x4E 0x47 0x0D 0x0A 0x1A 0x0A (\x89PNG\r\n\x1a\n).
  const pngMagic = Buffer.from([0x89, 0x50, 0x4e, 0x47, 0x0d, 0x0a, 0x1a, 0x0a]);
  for (const rel of expected) {
    const full = resolve(REPO_ROOT, rel);
    const head = readFileSync(full).subarray(0, 8);
    expect(head.equals(pngMagic)).toBe(true);
    // Sanity: file is non-empty (≥ pngMagic length); 1×1 placeholders are
    // ~67 bytes, real captures will be larger but still well under the
    // 500 KB acceptance limit.
    expect(statSync(full).size).toBeGreaterThanOrEqual(pngMagic.length);
  }
});
```
