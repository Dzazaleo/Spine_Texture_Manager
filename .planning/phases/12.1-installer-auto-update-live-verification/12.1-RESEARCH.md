# Phase 12.1: Installer + auto-update live verification - Research

**Researched:** 2026-04-28
**Domain:** electron-builder 26.x publish-race fix + live runtime verification of auto-update + INSTALL.md screenshot capture + Windows local-build documentation
**Confidence:** HIGH (publish-race root cause + 3 fix options); HIGH (live verification sequencing); MEDIUM-HIGH (screenshot capture protocols); HIGH (winCodeSign workaround already enumerated by user todo)

## Summary

Phase 12.1 is **primarily a live-verification + docs phase, not feature dev.** The auto-update code surface is already shipped and 433/433 vitest passing on macOS. The phase exists to (a) unblock the CI tag-push pipeline by fixing the electron-builder 26.x publish race that has burned 3 CI runs, (b) execute the deferred Plan 12-01 Spike Runbook against a real published rc2 to flip the 9 `human_needed` items in `12-VERIFICATION.md` to `passed`, (c) capture the 4 INSTALL.md screenshots (binary-only PNG swap, no markdown changes per Plan 12-06 design), and (d) document the local Windows `winCodeSign-2.6.0.7z` symlink-extract papercut.

**Root cause of the publish race is verified.** From PublishManager.ts source reading and the project's own SPIKE OUTCOME block: `publish: github` in YAML wins over `--publish never` on the CLI for **per-artifact** publisher construction — `--publish never` only suppresses the post-build `publish()` invocation, not `PublishManager.artifactCreatedWithoutExplicitPublishConfig` which constructs publishers per-artifact and uploads as artifacts emit. Two failure modes were observed: (i) no `GH_TOKEN` → publisher constructor throws "GitHub Personal Access Token is not set"; (ii) `GH_TOKEN` present → 3 parallel matrix legs race to upload to the same draft release, second/third hit HTTP 422 `asset_already_exists`. This is a known longstanding electron-builder pattern (issues #4546, #5463, #3559, #2094, #2337) — community consensus is that electron-builder's per-artifact auto-upload bypasses any orchestrated single-job publish step.

**Three fix architectures are viable**, ranked by simplicity and risk: (a) flip `electron-builder.yml` `publish: null` and inject `app-update.yml` via `extraResources` (reverts the Plan 12-02 wiring that introduced the race; minimal change to release.yml because softprops/action-gh-release@v2.6.2 already uploads all 6 assets atomically); (b) add `--config.publish=null` CLI override on `build:mac/win/linux` scripts (overrides the YAML at build time without touching the file; surgical); (c) keep `publish: github` and unset `GH_TOKEN` from the build job env (publisher constructor fails fast with no upload attempt; must verify electron-builder doesn't abort the entire build on this throw — risk: rebuilds the original failure mode (i)). Option (a) is recommended — it is the architecture that was working through Phase 11 Plan 11-02 close. The seam is: bake `app-update.yml` as a static file under `build/` and reference it via `extraResources`, OR write a tiny `electron-builder` `afterPack` hook that synthesizes it. Either way, `--publish never` becomes truly never. The existing `softprops/action-gh-release@v2.6.2` step at release.yml:151 already uploads 6 assets (3 installers + 3 `latest*.yml` feed files) atomically; this surface is correct and preserved.

**Live verification sequencing** is gated on the publish-race fix landing first. Recommended order: tag rc2 → wait for green CI → install rc2 on macOS + Linux + Windows (capture screenshots while installing) → tag rc3 → observe each installed copy detect rc3 → optionally tag rc4 (or rely on rc3-already-installed) → manual "no newer version" check on Help → Check for Updates. Total elapsed time: ~30 minutes of CI wait + ~15 minutes of per-OS install/observe.

**Primary recommendation:** Land Option (a) for the publish-race fix in a small autonomous wave (single plan, 2-3 atomic commits: revert `publish: github` to `publish: null` in `electron-builder.yml`; add `extraResources` entry for static `app-update.yml` OR an `afterPack` hook; remove the polluted-rc2 cleanup tax). Then a separate live-verification wave with HUMAN-UAT checkpoints for each `human_needed` item, structured as a runbook the user (Leo) executes locally with each OS install.

### Pattern 1: Static `app-update.yml` via extraResources

**What:** When `electron-builder.yml` has `publish: null`, the auto-generation of `app-update.yml` (the file electron-updater reads at runtime to know which feed to query) is suppressed. Inject a hand-written one via `extraResources`.

**When to use:** When the project wants electron-updater to work but doesn't want electron-builder to attempt any publishing.

**Example:**
```yaml
# electron-builder.yml — Phase 12.1 fix
publish: null   # revert from publish: github (the race source)

extraResources:
  - from: build/app-update.yml
    to: app-update.yml
```

```yaml
# build/app-update.yml — static, hand-authored, committed to git
provider: github
owner: Dzazaleo
repo: Spine_Texture_Manager
updaterCacheDirName: spine-texture-manager-updater
```

`[CITED: electron-builder docs at electron.build/configuration.html — extraResources is a glob relative to project directory; copies into Contents/Resources (mac) or resources/ (win+linux)]`
`[CITED: GH issue #4975 + #4233 + #2667 — confirms electron-updater throws "app-update.yml is missing" at runtime if file is absent; only path forward when publish:null is the static-file workaround]`

### Pattern 2: HUMAN-UAT Runbook Checkpoint

**What:** A plan that produces no code commits but executes a sequence of manual steps with explicit "stop-and-record-evidence" gates. Output is a transcript appended to a per-plan VERIFICATION.md (or back-folded into 12-VERIFICATION.md `human_verification` table).

**When to use:** Live-runtime verification of behavior that cannot be automated (real installer + real GitHub feed + real OS dialog).

**Example:** PLAN 12.1-03 (macOS live verification):
```markdown
## Runbook
1. STOP — confirm v1.1.0-rc2 is published as a non-draft release. Evidence: `gh release view v1.1.0-rc2 --json isDraft -q .isDraft` returns false.
2. Download `Spine-Texture-Manager-1.1.0-rc2-arm64.dmg` from the release page on the macOS test host.
3. Double-click the .dmg, drag app to Applications.
4. Right-click the app → Open. CAPTURE the Gatekeeper "Open Anyway" dialog → save as docs/install-images/macos-gatekeeper-open-anyway.png. Click Open Anyway.
5. Confirm app launches. Verify Help → About shows v1.1.0-rc2.
6. STOP — wait for v1.1.0-rc3 to be tagged + CI to succeed + draft to be published.
7. With rc2 still running, observe whether the UpdateDialog appears within 10s of next launch (3.5s startup check + network). If yes: UPD-01 verified. Click "Download + Restart". After download + restart, confirm app reports v1.1.0-rc3.
8. Test "Later": dismiss the prompt on a fresh launch; relaunch; confirm prompt does not re-appear for the same version.
9. Test offline: disable Wi-Fi, relaunch app. Confirm: no error dialog, no crash, normal startup. UPD-05 verified.
10. Append transcript line to 12-VERIFICATION.md flipping the macOS row from `human_needed` → `passed`.
```

### Anti-Patterns to Avoid

- **DO NOT add `GH_TOKEN` back to the per-build-job env.** This was the proximate cause of failure mode (ii). The build jobs do not need GitHub write access — only the publish job does, and it gets `permissions: contents: write` from release.yml:128-129.
- **DO NOT attempt to "fix" the race with retry logic in softprops.** softprops/action-gh-release@v2.6.2 already does the right thing — the race is upstream in electron-builder's per-artifact upload, not in the orchestrated publish step.
- **DO NOT capture screenshots on the dev macOS box.** macOS trusts locally-built apps even with quarantine attribute set; the Gatekeeper dialog state captured will not match what testers see. Captures MUST happen on a fresh download from GitHub Releases.
- **DO NOT introduce markdown changes when swapping placeholder PNGs.** Plan 12-06 explicitly designed for binary-only swap. The 4 markdown image references at INSTALL.md lines 27, 54, 60, 106 are correct as-is.
- **DO NOT push tags before the publish-race fix lands.** Per `deferred-items.md`: "DO NOT push tags until 12.1 lands the publish-race fix — workflow will fail and pollute releases."

## Common Pitfalls

### Pitfall 1: Re-introducing the publish-race by leaving `publish: github` after a partial revert

**What goes wrong:** Planner edits `electron-builder.yml` to add `extraResources` but forgets to also revert `publish: github` → `publish: null`. Both surfaces try to provide `app-update.yml`; electron-builder's auto-generated one wins; the static file is overwritten; AND electron-builder still attempts per-artifact upload because `publish: github` is still present.

**Why it happens:** The two changes look like "additions" but one is a reversal. Easy to miss the revert step.

**How to avoid:** Plan 12.1-PLAN-01 must explicitly require BOTH changes in the same atomic commit, with a verification grep: `grep "publish:" electron-builder.yml` should return exactly `publish: null` and nothing else.

**Warning signs:** Local validation build still emits "GitHub Personal Access Token is not set" — this is the smoking gun that `publish: github` is still active.

### Pitfall 2: Capturing screenshots on the wrong OS or wrong app build

**What goes wrong:** Tester captures the Gatekeeper dialog from a locally-built `.dmg` instead of the GitHub-downloaded one. macOS trusts locally-built apps; the dialog text/state differs from what real testers see.

**Why it happens:** Convenience — local .dmg is right there.

**How to avoid:** The runbook for each OS MUST start with "Download installer from GitHub Releases page" as Step 1, not "Use the .dmg already on your desktop." The capture happens AFTER the download dialog clears, on the freshly-installed app.

**Warning signs:** The captured PNG shows file paths under `~/Documents/WORK/CODING/Spine_Texture_Manager/release/` — that's the local build, not the downloaded one.

### Pitfall 3: Tagging rc3 before rc2 is installed on all test hosts

**What goes wrong:** Tester rushes ahead, tags rc3 immediately after rc2 publish. Now rc2 will detect rc3 on its first check — but if they haven't installed rc2 yet on the Windows host, they install rc3 directly and never observe the auto-update path from rc2 → rc3.

**Why it happens:** "Just push another tag" feels lower-friction than "wait for the human loop."

**How to avoid:** Sequence the runbook explicitly: (1) tag rc2; (2) wait CI; (3) install rc2 on macOS + Linux + Windows; (4) confirm Help → About reads rc2 on all three; (5) ONLY THEN tag rc3.

**Warning signs:** Help → About shows rc3 instead of rc2 on a "freshly installed rc2" host.

### Pitfall 4: macOS Gatekeeper "Open Anyway" wording differs from documented bypass

**What goes wrong:** macOS Sequoia (14.x → 15.x evolution) repositioned Gatekeeper UI to System Settings → Privacy & Security in 14.0. Earlier "right-click → Open" is still supported but the dialog wording changed in 14.0.

**Why it happens:** OS UI evolves; INSTALL.md was authored against documented historical wording.

**How to avoid:** During capture, transcribe the exact dialog wording. If it differs from the INSTALL.md text at lines 24-25, edit the markdown text in the SAME commit as the binary swap. This is the only Phase 12.1 case where INSTALL.md gets a markdown change beyond the prescribed binary swap.

**Warning signs:** Captured PNG shows wording that, when read, doesn't match INSTALL.md's italicized step text.

`[CITED: WebSearch 2026-04-28 + Apple Community thread on Sequoia "Allow Apps from Anywhere"]`

### Pitfall 5: Windows SmartScreen evolution between Win10 and Win11

**What goes wrong:** SmartScreen wording on Win10 22H2 vs Win11 23H2/24H2/25H2 differs subtly. The "More info" → "Run anyway" pattern is the same, but the dialog title may be "Microsoft Defender SmartScreen" (Win11) vs "Windows protected your PC" (Win10).

**Why it happens:** Microsoft updated SmartScreen UI multiple times since 2024.

**How to avoid:** The Windows test host should be the Windows version most testers will use (per project context: 22000 is Win10/11 dev box per `winCodeSign` todo). Document the OS version of the capture host inline with the screenshot for provenance.

**Warning signs:** Screenshot title matches one OS version; INSTALL.md text describes the other.

### Pitfall 6: AppImage runs successfully on Ubuntu 24.04 with libfuse2t64 INSTALLED — can't capture the error

**What goes wrong:** Tester wants to capture the libfuse2t64 error PNG, but their Ubuntu 24.04 already has libfuse2t64 installed (e.g., from prior testing).

**Why it happens:** The error only fires on a fresh Ubuntu 24.04 install with no FUSE library.

**How to avoid:** (a) `sudo apt remove libfuse2t64` before the capture attempt (then re-install for normal verification); OR (b) capture in a Docker container/VM with a clean Ubuntu 24.04 base; OR (c) use a VM snapshot if one exists.

**Warning signs:** AppImage launches normally — no error dialog to capture.

`[CITED: WebSearch 2026-04-28 — AppImage docs + Cursor Community Forum + Syllo/nvtop issue #292; exact error string is "dlopen(): error loading libfuse.so.2 — AppImages require FUSE to run."]`

### Pitfall 7: SmartScreen blocks the auto-update spawn on Windows (latent — only matters if SPIKE_PASSED is later flipped to true)

**What goes wrong:** Even if a tester runs rc2's auto-update flow (which is currently disabled — Windows defaults to manual-fallback per D-04), the spawned rc3 installer hits SmartScreen because it is itself unsigned. UPD-04 "Restart" appears to succeed but actually shows a SmartScreen block.

**Why it happens:** A spawned process inherits the trust level of its parent only when the parent is signed; an unsigned parent (rc2) spawning an unsigned child (rc3 installer) gets SmartScreened.

**How to avoid:** This is OUT OF SCOPE for 12.1 — Windows ships the manual-fallback variant by default per D-04, and the spike runbook is to verify the FALLBACK, not the auto-update. If a future phase flips `SPIKE_PASSED` to true, this pitfall becomes load-bearing.

**Warning signs:** Tester reports "Restart button doesn't actually relaunch the app." Confirm SPIKE_PASSED is still `false` on win32 — it should be.

3. Restart the terminal.

\`\`\`

### Known Threat Patterns for {Electron + GitHub Releases + auto-update stack}

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Malicious release feed (replaced `latest.yml`) | Tampering | electron-updater verifies SHA-512 of installer against feed; feed itself is served from GitHub Releases (TLS); only repo collaborators can publish — already mitigated by GitHub's access model |
| Replay attack on draft release | Spoofing | Drafts only visible to repo collaborators; tags require write access to repo; `softprops/action-gh-release@v2.6.2` SHA-pinned (supply chain) |
| Symlink extraction privilege escalation (winCodeSign) | Elevation | Workaround documents Developer Mode (per-user) or Admin (audited) — neither escalates beyond user's already-granted privileges; the extracted symlinks are macOS-only dylibs that have no effect on Windows runtime behavior |
| Unsigned Windows installer execution | Spoofing | SmartScreen "Run anyway" + INSTALL.md walkthrough — explicit user consent to run unsigned binary; per v1.1 milestone scope (no EV cert) |
