<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| repo → installer bundle | electron-builder copies build/app-update.yml into installer resources/; tampering with build/app-update.yml in a malicious commit could redirect updates to a hostile feed |
| installer → GitHub Releases feed | electron-updater reads the bundled app-update.yml at runtime to determine which feed to query; the feed itself is served via TLS from GitHub |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-12.1-01-01 | Tampering | build/app-update.yml | mitigate | Committed to git; reviewed in PR; the 4 fields are tightly constrained (any drift is visible in `git diff`); no interpolation/templating — the file IS the source of truth |
| T-12.1-01-02 | Tampering | electron-builder.yml extraResources entry | mitigate | Single-source-of-truth — the only line referencing build/app-update.yml; reviewed in PR; covered by Task 2's grep acceptance criteria |
| T-12.1-01-03 | Spoofing | GitHub Releases feed query | accept | TLS terminated at github.com; GitHub access model controls who can publish; outside this plan's threat surface |
| T-12.1-01-04 | Information Disclosure | (none) | accept | No new secrets, no PII; provider/owner/repo are public information |

**Security improvement note:** This plan REMOVES a token surface — the prior `publish: github` configuration would attempt to construct GitHub publishers per-artifact (requiring GH_TOKEN in the build env, which was attempted and reverted in commit 44bd03b). Reverting to `publish: null` means no token is ever required during the build phase, reducing token-exposure risk in CI logs / matrix legs.
</threat_model>
