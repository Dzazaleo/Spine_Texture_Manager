## Why this is blocking

`.github/workflows/release.yml` lines 73-79 (build-mac job):

```yaml
- uses: actions/upload-artifact@…
  with:
    name: installer-mac
    path: |
      release/*.dmg
      release/latest-mac.yml          # ← required by glob
    if-no-files-found: error          # ← fails the job if missing
```

Identical structure exists for `build-win` (line 99: `release/latest.yml`) and `build-linux` (line 120: `release/latest-linux.yml`).

If we tag `v1.1.0-rc2` with the current `679cffc` config, all 3 build legs will succeed at `electron-builder` step but FAIL at `actions/upload-artifact` because the `latest-*.yml` files no longer exist on disk. **rc2 becomes the 4th burned tag** at a different failure point than the 3 prior runs.

## Rework options (not selected — replanning required)

These are candidate paths to the real fix. None has been validated; pick one via `/gsd-discuss-phase 12.1` or a small spike before re-attempting Plan 12.1-01.

### Option A — `publish: never` (instead of `publish: null`)

Change `electron-builder.yml` line `publish: null` → `publish: never`. `never` is electron-builder's special value meaning "do not publish, but still generate publish artifacts." Plausible no-code-change fix.

- **Pros:** Single-character edit. Keeps `extraResources` static-yaml architecture.
- **Risk:** Behavior of `publish: never` vs `--publish never` CLI flag is not clearly documented. May still construct the GitHub publisher (re-introducing the auth race) or may not — needs empirical test.
- **Validation:** Same Task 3 pre-flight, verifying both bundled `app-update.yml` byte-match AND `latest-mac.yml` re-emergence.

### Option B — Restore `publish: github` + use `--publish never` CLI flag + ensure no `GH_TOKEN` in build envs

Revert Task 2 partly: keep the `provider: github / owner / repo` block in YAML (so latest-*.yml gets written), but rely on each `npm run build:*` script's existing `--publish never` suffix to suppress the upload race. **This is what Phase 11 era was doing successfully.**

- **Pros:** Restores known-working architecture from before the Phase 12 race.
- **Risk:** Re-introduces split source-of-truth between YAML feed pointer and bundled `app-update.yml`. Need to confirm the actual root cause of the 3 prior failed CI runs — was it the `publish: github` block alone, or was it `publish: github` + an environment factor (GH_TOKEN presence, electron-builder version)?
- **Validation:** Empirically reproduce one of the 3 prior failure logs (25017095851 / 25017351602 / 25017624868) to confirm the root cause is what we think it is, not what we assume.

### Option C — `publish: null` + post-build script that synthesizes `latest-*.yml`

Keep `679cffc` as-is. Add a post-`electron-builder` step (in package.json scripts) that reads the .dmg, computes its SHA-512 + size + sha512 + version, and writes `release/latest-mac.yml` (and equivalents for win/linux) by hand. electron-updater only consumes ~6 fields from these files (`version`, `files`, `path`, `sha512`, `size`, `releaseDate`).

- **Pros:** Fully decouples feed-file generation from electron-builder's publisher logic. Auditable.
- **Risk:** New script surface. Schema-drift if electron-updater requires fields we don't write. More moving parts on each rc/v tag-push.
- **Validation:** Hand-write a `latest-mac.yml`, install rc2 manually, point at a test feed, confirm the update flow works.

### Option D — Use `provider: generic` instead of `github`

`provider: generic` in electron-builder is an HTTP/HTTPS-served feed (any URL). It generates `latest-*.yml` without needing GitHub auth. Bundled `app-update.yml` would also need to switch to `provider: generic` with a stable URL (e.g., GitHub Pages or raw.githubusercontent.com).

- **Pros:** Keeps publish artifacts generated; no token surface; widely-used pattern.
- **Risk:** Changes the runtime feed URL. Different security model (no GitHub asset auth). Likely heaviest rework.
