## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-52-01 | Tampering (destructive FS) | D-03 `fsRm(outDir, {recursive:true, force:true})` in exportOneVariant rollback | mitigate | `outDir` is an APP-DERIVED path: `join(parentDir, `${NAME}@${formatScaleToken(s)}x`)` where `NAME = basename(skeletonPath).replace(/\.json$/i,'')` is rejected if empty or contains `:` (variant-export.ts:142-143, T-49-DIR), and `formatScaleToken` emits a clean numeric token. The recursive rm targets ONLY this derived leaf folder, never parentDir. The only-if-empty `readdir(outDir).length === 0` guard means a pre-existing NON-EMPTY folder (overwrite=true re-export with user content) is NEVER deleted — the rm fires solely on a folder this run freshly mkdir'd and then failed to populate. force-rm swallows ENOENT. No path component is attacker-influenced beyond the basename, which is sanitized upstream. |
| T-52-02 | Denial of Service (batch abort) | D-01 dup handling | mitigate | The old whole-batch abort was itself a self-DoS (one bad row killed all rows); D-01 downgrades it to a per-row failure so a single duplicate cannot deny the rest of the batch. No new surface. |

**No HIGH-severity threat.** This is local desktop file I/O on app-derived paths. The recursive force-rm is bounded to a sanitized, run-created, verified-empty leaf directory; it cannot escape into parentDir or destroy pre-existing user content. ASVS L1 file-access expectations are met by the basename sanitization (already shipped) + the only-if-empty guard (added here).
