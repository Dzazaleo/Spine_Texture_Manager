<objective>
Eliminate the false-positive green "savings" readout shown for reopened exported variants (and the latent same bug on master rigs with `peakScale>1`), purely as a renderer-side read-model / DISPLAY fix.

A reopened variant's geometry JSON is sized source-based (`width/height × s`, `scale-bake.ts`) while its PNGs are sized peak-based (`ceil(canonicalW × s·peakScale)`, `export.ts`). For art drawn smaller than it renders (`peakScale>1`, e.g. GRAND 0.846×, L_SKIRT 0.877×) the two disagree, so on reopen the display path computes Peak `= ceil(canonical × min(peakScale,1))` `< actualSource` and tints the cell green — false savings. The premature `≤ 1.0` clamp in the DISPLAY path is the root cause (it discards the `peakScale>1` signal). The fix computes the displayed Peak demand as `min(canonical × peakScale, actualSource)` (the locked `project_peak_anchored_invariants` "Peak = world demand"), rewrites the green tint to compare the two rendered integers (D-03), and rebases the savings chip onto the same per-row demand so chip ≡ Σ row states.

Purpose: make the readout tell the truth — variant textures are deliberately sharp (sized to `s × peak`); that is a feature of a per-resolution exporter, not waste. NO exported bytes change; do NOT re-optimize/shrink variants (that under-sizes `peakScale>1` art → blurry).

Output: edited renderer files (export-view, enrich-overrides, GlobalMaxRenderPanel, AppShell), 1 extracted pure helper (`row-state.ts`), a coupled tooltip-copy update (both panels + the icon docblock + the existing tooltip spec), and 1 new synthetic-row regression spec. `src/core/` untouched (Layer-3 pure).

This is a single wave (one plan, 3 tasks): the change is one renderer read-path with no internal parallelizable boundary. Task 1 (Wave 0) creates the test + the extracted helper first; Task 2 lands the display-demand math; Task 3 wires the panel + chip + the coupled tooltip-copy change.
</objective>
