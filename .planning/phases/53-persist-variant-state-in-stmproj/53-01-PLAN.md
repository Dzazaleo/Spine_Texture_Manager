## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `.stmproj` file on disk → core validator | A user-owned local `.stmproj` JSON is parsed and fed to `validateProjectFile`. The file may be hand-edited or copied from another machine — it is untrusted *input shape* even though it is the user's own file. |
| core materialized data → renderer (IPC) | `MaterializedProject.variantRows` crosses the main→renderer IPC boundary as the Open response. |

No NEW IPC channel is added; no NEW fs path is dereferenced (SC#3 reuses `lastOutDir` as a
hint only, never opening/statting it).

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-53-01 | Tampering | hand-edited `.stmproj` `variantRows` is a non-array (e.g. `5`, `"x"`, `{}`) | mitigate | `validateProjectFile` returns `ok:false` `invalid-shape` for any non-array `variantRows` before it reaches export math; the Open path surfaces the typed validation error rather than crashing. |
| T-53-02 | Tampering / Denial of Service | element `scale` is non-finite (`NaN`/`Infinity`) — `JSON.stringify` turns these to `null` on re-save and they would propagate into `s × master_peak` export arithmetic | mitigate | validator rejects any element whose `scale` is not `Number.isFinite` (mirrors the `overridesAtlasLess` finite-number guard); non-finite values can never enter the materialized state or be re-serialized. |
| T-53-03 | Tampering | element `scale` is negative, zero, or absurdly large; or the array is enormous | accept | Out of phase scope: the validator enforces finite-number shape only. Negative/zero/huge scales are a *value* concern handled by the existing renderer scale-input clamp (Phase 50/52) and the export-time `VariantScaleError` path; persistence stores whatever finite number the UI produced. Low risk — local single-user file, no privilege boundary; rows are re-edited in the dialog before any export. |
| T-53-04 | Denial of Service | saved `lastOutDir` points at a missing/inaccessible folder | accept (by design, D-02/SC#3) | No fs check is performed on `lastOutDir` at load; the variant flow always opens the native picker, so a stale dir is a harmless start hint and can never hard-fail the load. A grep-guard test asserts no new `existsSync`/`access`/`stat` is keyed on it. |
