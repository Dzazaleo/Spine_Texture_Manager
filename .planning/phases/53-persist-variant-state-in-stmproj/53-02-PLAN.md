<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| main → renderer (IPC Open response) | `MaterializedProject.variantRows` (already validated/back-filled by the 53-01 core validator) crosses into `AppShell` and seeds the lifted `variantRows` state. |
| user interaction → renderer state | The user edits scale rows in the Export Variant dialog; AppShell tracks them in `variantRows` and the dirty derivation. |

This plan adds NO IPC channel and NO fs access. It consumes the already-validated
`variantRows` from the materialized project and writes scales back through the existing save
payload (which flows to the 53-01 serializer).

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-53-06 | Tampering | a malformed `variantRows` reaches the renderer | mitigate (upstream) | By the time `MaterializedProject.variantRows` reaches AppShell it has already passed the 53-01 pure-core validator (non-array / non-finite-scale rejected; missing → default). The renderer trusts a validated shape; the `?? [{ scale: 0.5 }]` defence-in-depth on both restore paths covers any future path that bypasses the validator. |
| T-53-07 | Tampering / Repudiation | regenerated row ids cause a silent false-dirty, prompting an unnecessary quit-guard or letting an edit go unnoticed | mitigate | The dirty derivation compares an order-sensitive `number[]` scale projection (D-03), never the `{ id, scale }` objects — ids never enter the equality test, so a fresh-id load is NOT dirty and a real scale edit IS dirty. Locked by the Task 2 not-dirty-on-open + dirty-on-edit tests. |
| T-53-08 | Information Disclosure | persisted scales leak data | accept | Scales are numeric factors only (no paths/PII/secrets) — same disposition as T-53-05 in 53-01. |
| T-53-09 | Denial of Service | a stale `lastOutDir` breaks the variant flow | accept (by design, D-01/D-02/SC#3) | The reused `lastOutDir` is a picker pre-fill/start hint only; the variant flow always opens the native picker, so a missing folder is harmless. No fs check is added in the renderer (acceptance criterion greps for none). |

No `high`-severity unmitigated threat remains. The renderer-specific risk (id-driven
false-dirty, T-53-07) is mitigated by the scale-projection compare and locked by tests; all
shape-validation risk is handled upstream by the 53-01 validator.
</threat_model>
