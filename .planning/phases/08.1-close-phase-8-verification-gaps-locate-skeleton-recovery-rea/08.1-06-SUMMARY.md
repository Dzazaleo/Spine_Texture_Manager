### Patterns established / reaffirmed

1. **Discriminated-union typed-error envelope** â€” when an error variant needs to carry kind-specific payload, promote the flat `SerializableError` interface to a tagged union; the kind-specific arm declares the payload fields directly; consumers narrow via `error.kind === 'KindLiteral'` or `Extract<UnionType, { kind: 'KindLiteral' }>`. Fields appear in `state.error.*` reads without `as` casts. T-Tampering / wrong-arm-field-reads threats mitigated at compile time.
2. **Wave-0 RED-scaffolding pattern** (Plan 08-01 + 08.1-01 precedent) â€” type contract + grep-anchored RED specs land FIRST; downstream plans drive each spec to GREEN. Inter-plan dependencies become typecheck breaks at deliberate construction sites that subsequent plans MUST resolve, locking the integration path.
3. **`Exclude<Union['kind'], PayloadArmKind>` narrowing at forwarder cast sites** â€” zero runtime cost, catches future error-class additions at compile time. Used at 3 SpineLoaderError forwarder sites in Plan 08.1-02.
4. **React useRef callback-ref bridge for cross-component coupling** â€” when a stable callback identity must be passed to a third-party API (DropZone.onBeforeDrop) but the impl depends on child-only state (isDirty), hold `useRef<Fn|null>(null)` at the parent + stable `useCallback` wrapper that dereferences `.current` + child `useEffect` that registers/unregisters with cleanup that nulls the ref so post-unmount calls fall through to a safe default. Avoids prop-drilling and avoids the alternative of hoisting `isDirty` into the parent.
5. **Optional state-slot extension for new dispositions** â€” when an existing flow has 'do nothing' for a hook point (e.g., `'quit'` `onCancel` = no-op since Electron handles the abort), make the hook optional and let new flows opt in; existing call sites compile unchanged. One line of TypeScript instead of a full discriminated union.
6. **React 19 testing pattern** â€” under React 19's concurrent rendering, state updates from async event handlers flush via MessageChannel/setTimeout (macrotasks), NOT microtasks; use `await screen.findByX(...)` (testing-library `waitFor` with default 1000 ms / 50 ms polling) for any async-driven DOM, never `await Promise.resolve()` chains. Same Rule 3 fix applied in Plan 08.1-04 + Plan 08.1-05.
7. **Test-fixture target validity check** â€” when a fixture mutates state through the UI, verify the fixture data actually contains the mutation target; add explicit assertions (not silent `.catch` fallthroughs) so a missing target fails LOUDLY at the fixture line rather than producing a misleading downstream assertion failure.
8. **Compile-time `Extract<SerializableError, { kind: ... }>` for kind-specific payload access** in render branches â€” exposes the 7 cached recovery fields directly inside `state.error.*` reads without per-site narrowing.
9. **Renderer state slot field-name mapping** â€” when a renderer call site populates a state slot whose field names diverge from the IPC envelope's field names (e.g. `cached*` prefix on the renderer side), prefer direct field-read at the assignment (`{ cachedSamplingHz: resp.error.samplingHz }`) over intermediate destructuring with rename aliases â€” the single-line-per-field shape mirrors the state-slot declaration verbatim.

## Discovered out-of-scope issue â†’ Phase 08.2

**One adjacent issue surfaced during the manual UAT â€” flagged for the next planner pickup.**

**Symptom.** When the app is in an error state (e.g. the "skeleton not found" recovery banner is visible OR a "Not a .json or .stmproj file: skeleton.png" toast is still showing), pressing **Cmd+O** does NOT open the file picker. The user must either:

1. Dismiss the existing error banner / wait for the toast to clear, OR
2. Drop a clean/working .json/.stmproj first to clear the error state,

â€¦before Cmd+O becomes responsive again.

**Scope classification.** Keyboard-accelerator gating issue. **Adjacent to** VR-02's domain (toolbar Open + Cmd+O) but **NOT a regression** of the VR-02 fix itself â€” VR-02 specifically tested the data-correctness path (threaded `projectPath` instead of empty string), which is GREEN both in automated specs and in manual UAT. The issue is at a different layer: the keydown listener registration / suppression logic when an error mount is active on App.tsx (the new `projectLoadFailed` variant from Plan 08.1-04, OR the toast surface from Phase 8).

**Likely root cause hypothesis** (for Phase 08.2 planner â€” not investigated here): the `Cmd+O` keydown listener probably lives inside the AppShell mount path with a `role="dialog"` suppression check. When App.tsx renders `projectLoadFailed` instead of `projectLoaded`, AppShell does NOT mount â†’ its keydown listener is not registered â†’ Cmd+O has no handler. The toast variant has a similar shape: the toast surface may be on App.tsx but AppShell still owns the Cmd+O listener.

**Phase 08.2 planning notes:**
- Either hoist the Cmd+O listener to App.tsx so it survives AppShell-unmount errors states, OR add an App.tsx-level keydown listener for the `projectLoadFailed` variant + toast surface.
- Define expected behavior: should Cmd+O dismiss the error first then open the picker? Or should it open the picker AND leave the error visible (since the user is presumably trying to recover)?
- VR-02 spec coverage stays as-is â€” this is a NEW spec category (keyboard-accelerator-during-error-state), not a regression of VR-02.

**Disposition.** Routed to **Phase 08.2** for planner pickup. Per orchestrator decision documented in this close-out: do NOT extend Phase 08.1 scope. Phase 08.1 closes here with a clean signoff.
