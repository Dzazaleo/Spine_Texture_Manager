<objective>
Establish the CI delivery surface and runtime dependency that 12-01 (auto-update wiring) consumes. Three coordinated changes:
1. Expand the `release.yml` `test` job to a 3-OS matrix (`ubuntu-latest`, `windows-2022`, `macos-14`) so future Windows-only regressions like F1 are caught at PR time, not Phase-N+1 install smoke (D-22, D-23).
2. Add per-platform update-feed file uploads (`latest.yml`, `latest-mac.yml`, `latest-linux.yml`) to the `publish` job alongside installers — the missing piece that lets `electron-updater` poll our GitHub Releases for newer versions (D-11, D-12).
3. Flip `electron-builder.yml` `publish: null` → `publish: { provider: github, ... }` so `app-update.yml` gets baked into `resources/` at build time. Without this, electron-updater errors at runtime ("app-update.yml is missing"). The `--publish never` belt-and-braces in `package.json` build scripts is preserved (Phase 11 D-01 lock).

Purpose: This plan is the **CI/build-tool wave**. Lands in Wave 1 because Plan 12-01's spike (post-execution) requires both `latest.yml` files in the live release feed AND `app-update.yml` in the installed app's resources. Without 12-02, the spike cannot run.

Output: A CI workflow that produces auto-updater-ready installers on every tag push, plus the runtime dependency installed for 12-01 to import.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions runner → npm registry | `npm install` over HTTPS; npm-side integrity via `package-lock.json` SHA-512 + npm registry signing |
| GitHub Actions runner → GitHub Releases API | `softprops/action-gh-release@v2.6.2` (SHA-pinned) authenticated via `GITHUB_TOKEN` (workflow-scoped, not user PAT); writes to draft release |
| GitHub Releases (HTTPS) → installed app | Future `electron-updater` reads `latest*.yml` over HTTPS — Plan 12-01 wires this; this plan only stages the data on the server side |
| Build tool (`electron-builder`) → bundled `app-update.yml` | `publish: { provider: github, ... }` block in `electron-builder.yml` becomes the trust anchor — installed app reads this file at runtime to know which feed URL to poll |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-12-02-01 | Tampering | `latest*.yml` integrity in flight (CI runner → GH Releases) | mitigate | `softprops/action-gh-release@v2.6.2` is SHA-pinned (Phase 11 D-04 discipline preserved); upload happens over HTTPS to api.github.com; GitHub stores the file on its CDN with their signing |
| T-12-02-02 | Spoofing | Malicious actor publishes a fake `latest.yml` to a different repo, tricks the app | accept | `app-update.yml` (bundled into installer at build time per Task 2) hard-codes `owner: Dzazaleo, repo: Spine_Texture_Manager`. The installed app polls ONLY this repo's feed. To spoof, attacker would need to compromise the Dzazaleo GitHub account itself — out of v1.1's threat model (Apple/Win code-signing is the v1.2+ defense layer). |
| T-12-02-03 | Tampering | `latest.yml` uploaded by a malicious GHA workflow run from a forked PR | mitigate | `release.yml` is gated on `tags: ['v*.*.*']` push events to `main` (Phase 11 D-04). Forked PR runs cannot push tags to the upstream repo. Workflow only runs from `main` branch's tag refs. |
| T-12-02-04 | Information disclosure | `electron-updater` runtime dependency carries 80+ transitive deps; supply-chain attack surface | accept | `electron-updater@6.8.3` is the canonical electron-builder companion (RESEARCH-VERIFIED npm registry, latest stable, 2026-04-27). Same supply-chain trust posture as `electron-builder` itself (already a devDependency). `package-lock.json` SHA-pinning provides bit-for-bit reproducibility. |
| T-12-02-05 | Denial of service | CI matrix expansion 3× the test minutes; could exhaust GitHub Actions free tier | accept | Test job is ~1-2 min; 3× = ~5 min. GHA free tier = 2000 min/month. At ≤10 tag pushes/month this is well within budget. Sole project; no concurrent contributors. |
| T-12-02-06 | Elevation of privilege | `app-update.yml` baked into the installer is mutable post-install (sits in `resources/`); a local attacker could rewrite it to point to a malicious feed | accept | Local-attacker threat is out of v1.1 scope (per CONTEXT.md "no signing in v1.1"). NSIS installer's per-user install location (`%LOCALAPPDATA%`) means user can rewrite anything in their own profile anyway; the `app-update.yml` swap is not a privilege escalation, just one of many possible self-attacks an unsigned app cannot defend against. |
| T-12-02-07 | Repudiation | If a malicious release is published to `Dzazaleo/Spine_Texture_Manager`, no audit trail exists in installed apps | accept | GitHub Releases page provides the audit trail (release author, timestamp, commit SHA). Phase 13 (TEL) will add crash-reporting metadata that includes installed app version; if testers report compromise, version will narrow it down. Out of Phase 12 scope. |
