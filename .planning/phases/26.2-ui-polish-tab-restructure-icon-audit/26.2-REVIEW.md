# Phase 26.2: Code Review Report

**Reviewed:** 2026-05-04
**Depth:** standard
**Files Reviewed:** 6
**Status:** issues_found

## Summary

Phase 26.2 introduced a 3-tab system (`global` / `unused` / `animation`), lifted the `query` prop from `UnusedAssetsPanel` to `AppShell`, added a conditional Unused tab with a danger badge count, and converted fill-based warning triangle SVGs to stroke style in three panel files. The structural changes are sound and the D-03 query-lift is correctly implemented. Four warnings and three info items were identified.

The most significant warning is a dead code path in the `rowState` function ŌĆö the `'unused'` state variant is fully coded (type definition, function branch, styling in two `clsx` blocks) but is **never reachable** because every call site hard-codes `isUnused = false`. This is a logic gap: the intent to colour orphaned-attachment rows red in the Global panel is present in the code structure but silently bypassed. It does not crash, but it means the visual contract described in comments is not honored.

The remaining warnings are: a project-level typography contract violation (`font-medium` on tab buttons, which is forbidden by the two-weight rule in `AnimationBreakdownPanel.tsx` and `GlobalMaxRenderPanel.tsx` doc headers), an `onQueryChange` prop that is accepted but intentionally never called in two panels with no suppression annotation outside `UnusedAssetsPanel`, and a dead `closeBothDialogs` callback kept alive only via `void closeBothDialogs`.

## Warnings

### WR-01: `'unused'` RowState is unreachable ŌĆö orphaned-attachment rows never receive the danger left-accent

**File:** `src/renderer/src/panels/GlobalMaxRenderPanel.tsx:955` and `:1070`

**Issue:** `rowState` has a complete `'unused'` branch (`isUnused` check at line 176, `bg-danger` left-accent at line 435, `bg-danger/10 text-danger` tint at line 526). The comments inside `RowProps` (line 357) and the `rowState` doc (lines 166-172) describe this as the state for orphaned/unused attachments. However, **every call site passes `isUnused = false` literally**:

```ts
// line 955 (virtualized path)
const state = rowState(row.effectiveScale, false, row.isMissing);
// line 1070 (flat-table path)
const state = rowState(row.effectiveScale, false, row.isMissing);
```

There is no code path where `isUnused` is derived from `effectiveSummary.orphanedFiles` or any per-row flag on `DisplayRow`. The `DisplayRow` type does not have an `isUnused` field (confirmed in `src/shared/types.ts`). The same dead pass exists in `AnimationBreakdownPanel.tsx` lines 851 and 896 (though unused state is explicitly documented as always-false there ŌĆö that is intentional and consistent with the panel's design).

For `GlobalMaxRenderPanel`, this is a logic gap: the visual design intent (red danger accent on rows whose attachment name appears in `orphanedFiles`) is coded structurally but never connected to data. Orphaned-attachment rows currently render as `'neutral'`.

**Fix:** Either connect `isUnused` to real data by computing a `Set<string>` from `effectiveSummary.orphanedFiles` before the render loop and passing it to `rowState`, or remove the `'unused'` branch from `GlobalMaxRenderPanel`'s local `rowState` if the feature is intentionally deferred. Leaving the dead branch without a comment invites future confusion.

```ts
// Option A ŌĆö connect to data (if feature is intended now)
const orphanedNames = useMemo(
  () => new Set((summary.orphanedFiles ?? []).map((f) => f.filename)),
  [summary.orphanedFiles],
);
// ... in render:
const state = rowState(row.effectiveScale, orphanedNames.has(row.attachmentName), row.isMissing);

// Option B ŌĆö remove dead branch (if feature is deferred)
// Delete lines 176 + 435 + 526 in GlobalMaxRenderPanel.tsx
// and simplify RowState to 'under' | 'over' | 'neutral' | 'missing'
```

### WR-02: `font-medium` used in `TabButton` violates the two-weight typography contract

**File:** `src/renderer/src/components/AppShell.tsx:1787` and `:1794`

**Issue:** The project enforces a two-weight contract: only `font-normal` (400) and `font-semibold` (600) may appear. This is documented in the `AnimationBreakdownPanel.tsx` file header ("Two-weight typography contract: only font-normal (400) + font-semibold (600) appear. Weight 500 is forbidden project-wide.") and is mirrored in `GlobalMaxRenderPanel.tsx`.

`TabButton` uses `font-medium` (weight 500) in two places:

```tsx
// line 1787 ŌĆö active tab label
isActive ? 'font-medium text-accent' : 'font-normal text-fg-muted hover:text-fg',

// line 1794 ŌĆö badge pill
className="bg-danger/20 text-danger rounded-full text-xs font-medium px-1.5 py-0.5 ml-1.5"
```

Both are Phase 26.2 additions. The `TabButton` comment at line 1761-1763 explicitly documents this as `font-medium text-accent (weight 500)`, acknowledging it knowingly departs from the two-weight rule without explaining why an exception is justified.

**Fix:** Replace `font-medium` with `font-semibold` on both the active tab label and the badge pill to comply with the project-wide typography rule, or add a rationale comment explaining why the tab strip is an approved exception.

```tsx
// line 1787
isActive ? 'font-semibold text-accent' : 'font-normal text-fg-muted hover:text-fg',

// line 1794
className="bg-danger/20 text-danger rounded-full text-xs font-semibold px-1.5 py-0.5 ml-1.5"
```

### WR-03: `onQueryChange` prop is accepted but intentionally never called in `UnusedAssetsPanel` ŌĆö contract mismatch with zero documentation in the type

**File:** `src/renderer/src/panels/UnusedAssetsPanel.tsx:31` and `:34`

**Issue:** `UnusedAssetsPanel` declares `onQueryChange: (q: string) => void` as a required prop (not optional), yet renames it to `_onQueryChange` on destructure and never calls it:

```ts
export function UnusedAssetsPanel({ orphanedFiles, query, onQueryChange: _onQueryChange }: UnusedAssetsPanelProps)
```

The panel has no SearchBar ŌĆö the query is fully controlled from `AppShell`. The `onQueryChange` prop is accepted purely to satisfy the same-shape convention as `GlobalMaxRenderPanel` and `AnimationBreakdownPanel`. In those two panels, `onQueryChange` is also destructured but never called for the same reason (the SearchBar lives in the sticky header, not inside the panel). However, those two panels have a long inline comment explaining the design (e.g., `AnimationBreakdownPanel.tsx:313-318`: "`onQueryChange` is destructured to satisfy the props contract but never invoked from this panel"). `UnusedAssetsPanel` has only a brief comment at the interface level with no explanation of the suppression.

The deeper issue is that the prop is required but mechanically useless. If a future caller passes `onQueryChange` expecting the panel to call it when the search bar is used, they will be silently ignored. The `_` prefix convention suppresses the unused-variable lint warning but does not communicate the deliberate no-call intent to downstream consumers.

**Fix:** Either mark `onQueryChange` optional (`onQueryChange?: (q: string) => void`) to signal it is not load-bearing, or add a comment in `UnusedAssetsPanelProps` matching `AnimationBreakdownPanel`'s existing documentation that explicitly calls out the intentional no-call contract:

```ts
export interface UnusedAssetsPanelProps {
  orphanedFiles: OrphanedFile[];
  /** Phase 26.2 D-03 ŌĆö query lifted to AppShell (single SearchBar drives all panels). */
  query: string;
  /**
   * Phase 26.2 D-03 ŌĆö accepted for shape-parity with GlobalMaxRenderPanel /
   * AnimationBreakdownPanel. Never invoked from this panel ŌĆö the sticky-bar
   * SearchBar in AppShell calls setQuery directly. No SearchBar lives inside
   * this panel.
   */
  onQueryChange: (q: string) => void;
}
```

### WR-04: `closeBothDialogs` kept alive via `void closeBothDialogs` ŌĆö dead production code

**File:** `src/renderer/src/components/AppShell.tsx:637-641`

**Issue:** `closeBothDialogs` (defined at line 587) was superseded by the more specific `onConflictCancel` handler. It is no longer wired to any button or event. The code suppresses the TypeScript "declared but never read" error with:

```ts
void closeBothDialogs;
```

This is a deliberate workaround to keep the symbol "live" so the typechecker does not warn, while the function is never actually called in any code path. The comment at line 637-641 acknowledges this ("keep the symbol live for the typechecker"). This pattern is fragile: if `closeBothDialogs` is ever inadvertently reconnected to a handler, it will silently skip the `pendingConfirmResolve` resolution that `onConflictCancel` performs ŌĆö leaving `OptimizeDialog` awaiting a promise that never resolves.

**Fix:** Remove `closeBothDialogs` and its `void` suppression entirely. The function's logic lives in `onConflictCancel` (which also resolves the pending confirmation promise). If the function is referenced from a comment for documentation purposes, a prose comment is sufficient:

```ts
// Note: Cancel routes through onConflictCancel (not a shared closeBothDialogs
// helper) because Cancel must also resolve the pending OptimizeDialog promise.
```

### IN-01: D-02 auto-redirect `useEffect` dependency array uses an expression, not a variable ŌĆö React lint may warn

**File:** `src/renderer/src/components/AppShell.tsx:438`

**Issue:** The dependency array for the D-02 redirect effect contains a computed expression rather than a stable value reference:

```ts
}, [(effectiveSummary.orphanedFiles ?? []).length, activeTab]);
```

`(effectiveSummary.orphanedFiles ?? [])` allocates a new array reference when `orphanedFiles` is `undefined`, meaning the expression itself is not stable across renders ŌĆö only the `.length` primitive is. React's rules-of-hooks linter (`react-hooks/exhaustive-deps`) will flag this because the expression is not a hook-dependency-safe primitive or stable reference. The comment at line 432-433 anticipates this ("Watches a primitive (.length) not the array reference ŌĆö avoids infinite re-render") but the linter may still flag it depending on eslint config. The code is functionally correct, but a named intermediate variable would make the intent clearer and silence the lint warning:

**Fix:**
```ts
const orphanedCount = (effectiveSummary.orphanedFiles ?? []).length;
useEffect(() => {
  if (activeTab === 'unused' && orphanedCount === 0) {
    setActiveTab('global');
  }
}, [orphanedCount, activeTab]);
```

### IN-02: Test spec (i) does not cover the collapsed + query case

**File:** `tests/renderer/unused-assets-panel.spec.tsx:118-138`

**Issue:** Test (i) verifies that re-rendering with a new `query` prop filters rows correctly. However it does not cover the case where the panel is **collapsed** (user clicked "Hide details") and then a query is applied. In that state, `filteredOrphans` is computed (hooks run unconditionally) but the table is not rendered ŌĆö both the filtered rows and the "(no matches)" message are hidden from the user. This is technically correct behavior (the filter doesn't show anything when the table is collapsed), but it means the test does not verify:

- That `filteredOrphans` computation is correct when `expanded = false`
- That a non-empty query does not inadvertently force the panel to expand

This is a coverage gap, not a bug. Adding a test case that collapses the panel, applies a query, and then verifies the panel remains collapsed (no table rows visible) would protect against a future regression where a `useEffect` or other side effect accidentally re-expands the panel on query change.

### IN-03: `TabButton` tab strip lacks `aria-controls` / `role="tabpanel"` wiring required by WAI-ARIA tabs pattern

**File:** `src/renderer/src/components/AppShell.tsx:1780-1807`

**Issue:** `TabButton` renders `role="tab"` with `aria-selected` but omits `aria-controls` (linking each tab to its panel) and the corresponding `role="tabpanel"` / `aria-labelledby` on the content regions. The WAI-ARIA Tabs pattern requires:

- Each `role="tab"` button has `aria-controls="<panelId>"`
- Each panel has `role="tabpanel"` and `aria-labelledby="<tabId>"`

Without these, assistive technologies cannot navigate from the active tab to the tab panel content. The existing code works visually and the three panels render correctly, but screen-reader users cannot discover the tab/panel association.

**Fix (minimal):** Add `id` props to each `TabButton` and `aria-controls` pointing to the panel `id`, and add `role="tabpanel"` + `aria-labelledby` to the wrapping elements around each panel's content. Example:

```tsx
<TabButton
  id="tab-global"
  aria-controls="panel-global"
  isActive={activeTab === 'global'}
  ...
>Global</TabButton>

// and in <main>:
{activeTab === 'global' && (
  <div id="panel-global" role="tabpanel" aria-labelledby="tab-global">
    <GlobalMaxRenderPanel ... />
  </div>
)}
```
