# Phase 26.2: Code Review Report

**Reviewed:** 2026-05-04
**Depth:** standard
**Files Reviewed:** 6
**Status:** issues_found

## Summary

This phase implements tab restructuring (adding an "Unused" tab, tab-strip icons, badge count) and query-state lifting for the UnusedAssetsPanel. The structural changes are sound — tab-gating, auto-redirect on empty orphans, and the `onQueryChange` lift are implemented correctly. However, one critical correctness bug was found: the `onQueryChange` callback is received as a required prop but immediately discarded via a dead-alias (`_onQueryChange`), with no search-input element ever being rendered in the panel. This makes the prop contract dishonest and breaks any future caller who expects to receive query change events. Four additional warnings cover real behavioral gaps and maintainability risks.

## Critical Issues

### CR-01: `onQueryChange` prop accepted but permanently discarded — SearchBar is never rendered in UnusedAssetsPanel

**File:** `src/renderer/src/panels/UnusedAssetsPanel.tsx:35`

**Issue:** The component signature declares `onQueryChange` as a required prop in the public interface (`UnusedAssetsPanelProps`), but then immediately aliases it to `_onQueryChange` (underscore prefix = intentionally unused). No `<SearchBar>` or `<input>` element is rendered inside the panel. The prop is inert dead weight.

This is a correctness failure under the Phase 26.2 D-03 contract. D-03 says "query lifted to AppShell (single SearchBar drives all panels)". The intent is that AppShell's SearchBar calls `onQueryChange` → updates the `query` prop → filters the panel. That chain is correctly wired in AppShell (`onQueryChange={setQuery}` at line 1585), so AppShell's SearchBar *does* filter this panel through the `query` prop. So filtering itself works.

However, the panel's own contract advertises `onQueryChange` as a required callable that consumers are expected to provide. Nothing inside the panel will ever invoke it. If the caller passes a function expecting to be notified of query changes originating inside this panel (e.g. a typeahead input inside the panel), that callback silently does nothing. More concretely: the public interface is a lie. It declares a required callback that is never called. Any code that adds a search input *inside* this panel in a future iteration will discover the aliased-to-unused pattern and either leave the alias or be confused about why the prop exists.

The underlying logic is correct (query filtering via the `query` prop works), but the prop signature must be honest.

**Fix:** Remove `onQueryChange` from `UnusedAssetsPanelProps` entirely, since this panel never emits query changes. AppShell already manages query state centrally; this panel only needs to *receive* `query`, not report it back. Update the interface and all call sites:

```typescript
// UnusedAssetsPanelProps — remove onQueryChange
export interface UnusedAssetsPanelProps {
  orphanedFiles: OrphanedFile[];
  query: string;
  // onQueryChange removed — panel is read-only consumer of query (D-03)
}

// Component signature
export function UnusedAssetsPanel({ orphanedFiles, query }: UnusedAssetsPanelProps) {
```

```tsx
// AppShell.tsx line ~1582 — drop onQueryChange from the JSX prop
<UnusedAssetsPanel
  orphanedFiles={effectiveSummary.orphanedFiles ?? []}
  query={query}
/>
```

```tsx
// tests/renderer/unused-assets-panel.spec.tsx — remove onQueryChange={() => {}} from all render calls
render(<UnusedAssetsPanel orphanedFiles={ONE_FILE} query="" />);
```

If the intention is to keep `onQueryChange` for future use (e.g., a per-panel search input that feeds back up), it should be marked `optional?: (q: string) => void` and the comment should explain why it is not called today. Required props that are structurally dead cause confusion and test noise.

## Warnings

### WR-01: `useEffect` dependency on `.length` of a freshly-created array expression causes unexpected re-runs

**File:** `src/renderer/src/components/AppShell.tsx:258`

**Issue:** The auto-redirect effect is declared as:

```typescript
useEffect(() => {
  if (activeTab === 'unused' && (effectiveSummary.orphanedFiles ?? []).length === 0) {
    setActiveTab('global');
  }
}, [(effectiveSummary.orphanedFiles ?? []).length, activeTab]);
```

The dependency `(effectiveSummary.orphanedFiles ?? []).length` evaluates `?? []` on every render. When `effectiveSummary.orphanedFiles` is `undefined`, this produces a new `[]` array literal on each render whose `.length` is always `0`. React's hook machinery reads the primitive `0` from `.length`, so the dep comparison is `0 === 0` — stable. This happens to work numerically, but the `?? []` construct inside the dep array is unnecessarily convoluted and fragile; a future reader or linter rule might hoist it to a variable and break the primitive comparison. The comment "Watches the primitive .length ... to avoid infinite re-render loops" justifies the approach but the code pattern is subtle.

More importantly: if `orphanedFiles` is always either `undefined` or a proper array (which the `??` guard suggests), then when it is `undefined` this effect fires with `.length === 0` on every re-render of AppShell while the Unused tab is active — which would continuously redirect away. That is the correct behavior, but the mechanism is fragile.

**Fix:** Extract to a local variable before the effect:

```typescript
const orphanedCount = (effectiveSummary.orphanedFiles ?? []).length;

useEffect(() => {
  if (activeTab === 'unused' && orphanedCount === 0) {
    setActiveTab('global');
  }
}, [orphanedCount, activeTab]);
```

This makes the dependency explicit, survives linter rules that flag inline expressions in dep arrays, and removes the hidden `?? []` double-evaluation on each render check.

### WR-02: Tab buttons use `role="tab"` without `aria-controls` — incomplete ARIA tab pattern

**File:** `src/renderer/src/components/AppShell.tsx:1783-1808`

**Issue:** `TabButton` renders a `<button role="tab" aria-selected={isActive}>`. The parent `<nav role="tablist">` is correctly established (line 1338). However, the WAI-ARIA Tabs pattern requires that each `role="tab"` element have an `aria-controls` attribute pointing to the associated `role="tabpanel"` element, and each tabpanel must carry `role="tabpanel"` with `aria-labelledby` pointing back to its tab.

Currently, the tab panel content (the `<div className="...">` wrapping `GlobalMaxRenderPanel` / `UnusedAssetsPanel` / `AnimationBreakdownPanel`) carries no `role="tabpanel"` or `id`. Screen readers using a tabs interaction model will not be able to navigate to the associated panel from the tab button.

This is a real accessibility regression introduced by Phase 26.2 (the tab strip was added in this phase).

**Fix:** Assign stable IDs to tabs and panels:

```tsx
// TabButton
function TabButton({ id, panelId, isActive, onClick, children, icon, badge }: {
  id: string;
  panelId: string;
  // ... rest of existing props
}) {
  return (
    <button
      id={id}
      role="tab"
      type="button"
      aria-selected={isActive}
      aria-controls={panelId}
      onClick={onClick}
      // ... existing classNames
    >
```

```tsx
// In AppShell render — wrap each conditional panel in a tabpanel div
{activeTab === 'global' && (
  <div id="tabpanel-global" role="tabpanel" aria-labelledby="tab-global">
    <GlobalMaxRenderPanel ... />
  </div>
)}
{activeTab === 'unused' && (
  <div id="tabpanel-unused" role="tabpanel" aria-labelledby="tab-unused">
    <UnusedAssetsPanel ... />
  </div>
)}
{activeTab === 'animation' && (
  <div id="tabpanel-animation" role="tabpanel" aria-labelledby="tab-animation">
    <AnimationBreakdownPanel ... />
  </div>
)}
```

### WR-03: `closeBothDialogs` is defined, then silenced with `void closeBothDialogs` — dead-code side-effect

**File:** `src/renderer/src/components/AppShell.tsx:588-642`

**Issue:** `closeBothDialogs` is a `useCallback` hook defined at line 588. The comment at line 638-641 explains it is "no longer wired to any handler" and the `void closeBothDialogs` expression at line 642 is used purely to prevent TypeScript's `no-unused-locals` error. This is a deliberate dead-code pattern that conceals a cleanup debt.

The problem: using `useCallback` for a function that is never actually called creates a hook allocation on every render (React still allocates the memoized slot even if nobody calls the function). More importantly, the `void` trick is a signal that the symbol should have been removed but wasn't, and future readers will see a named function that appears to be wired but is actually dead.

**Fix:** Remove `closeBothDialogs` entirely. The two code paths that previously might have used it (`onConflictCancel` and `onConflictPickDifferent`) both inline their own cleanup logic. If future work needs a combined-close helper, it can be re-added then.

```typescript
// Delete lines 588-592 and 638-642:
// const closeBothDialogs = useCallback(() => { ... }, []);
// ...
// void closeBothDialogs;
```

### WR-04: `UnusedAssetsPanel` header count and total bytes reflect ALL orphaned files, but the table shows only filtered results — count/bytes mismatch when a query is active

**File:** `src/renderer/src/panels/UnusedAssetsPanel.tsx:42-44, 87-93`

**Issue:** The header always renders:

```tsx
<span className="font-semibold text-danger">
  {count} orphaned file{plural}   // count = orphanedFiles.length (ALL files)
</span>
{totalBytes > 0 && (
  <span className="text-fg-muted"> · {formatBytes(totalBytes)}</span>  // ALL bytes
)}
```

`count` and `totalBytes` are derived from the full `orphanedFiles` array (lines 42-44), not from `filteredOrphans`. So when a query is active (e.g. `query="STALE"`), the header says "3 orphaned files · 3.5 KB" while the table shows only 1 row. This is misleading — the user sees a count that doesn't match the visible rows.

The GlobalMaxRenderPanel and AnimationBreakdownPanel handle this more carefully: the card header shows "M / N ... — filtered" when a query is active (AnimationBreakdownPanel line 481-488). UnusedAssetsPanel has no such indicator.

**Fix:** Derive header display values from `filteredOrphans` when a query is active, or add a "filtered" suffix:

```tsx
const displayCount = query.trim() !== '' ? filteredOrphans.length : count;
const displayBytes = query.trim() !== ''
  ? filteredOrphans.reduce((acc, f) => acc + f.bytesOnDisk, 0)
  : totalBytes;
const filteredSuffix = query.trim() !== '' && filteredOrphans.length !== count
  ? ` — ${filteredOrphans.length} / ${count} shown`
  : '';
```

Then render `{displayCount}` and `{formatBytes(displayBytes)}` in the header, with the suffix appended to the label text.

### IN-01: Duplicate `rowState` function defined in both `GlobalMaxRenderPanel.tsx` and `AnimationBreakdownPanel.tsx`

**File:** `src/renderer/src/panels/GlobalMaxRenderPanel.tsx:174-180`, `src/renderer/src/panels/AnimationBreakdownPanel.tsx:176-182`

**Issue:** The `rowState` helper is identical in both files (same signature, same body, same `RowState` type definition). The comments acknowledge the duplication explicitly ("intentional duplication to keep this panel self-contained"). This is an accepted trade-off per Plan 19-04, but it is a maintenance liability: if a new state variant is added (e.g. `'overridden'`), both files must be updated in sync.

**Fix:** Extract to a shared renderer utility, e.g. `src/renderer/src/lib/row-state.ts`. Both panels import from there. This respects the Layer 3 invariant (renderer-only import, not from `src/core/*`).

### IN-02: `onQueryChange` prop in `AnimationBreakdownPanel` is received but never invoked — same structural dead code as `UnusedAssetsPanel`

**File:** `src/renderer/src/panels/AnimationBreakdownPanel.tsx:305, 313-318`

**Issue:** `onQueryChange` is destructured at line 305 and a comment at lines 313-318 explains "the panel-internal SearchBar JSX element is GONE ... `onQueryChange` is destructured to satisfy the props contract but never invoked from this panel." This is the same pattern as `UnusedAssetsPanel`'s `_onQueryChange`, just without the underscore prefix. The prop is accepted, documented as unused, and silently dropped.

Unlike the critical finding (CR-01) on `UnusedAssetsPanel` (which declares `onQueryChange` as required and creates API surface for callers), `AnimationBreakdownPanel` at least documents why the prop exists — the D-03 lift removed the internal SearchBar but kept the prop for contract stability. However, there is no test or mechanism preventing someone from calling `onQueryChange` inside the panel and getting confused when AppShell's `query` state doesn't respond.

**Fix:** If the prop is intentionally retained for forward-compatibility, mark it as optional with a comment:
```typescript
/** @deprecated Phase 19 D-04 — no longer invoked; AppShell's SearchBar is the source of truth. May be removed in a future phase. */
onQueryChange?: (q: string) => void;
```
This documents intent without breaking existing callers and avoids the "required but unused" API lie.

### IN-03: `TabButton`'s active-state underline uses `font-medium` (weight 500) which violates the project's two-weight typography contract

**File:** `src/renderer/src/components/AppShell.tsx:1789`

**Issue:** The two-weight contract documented in `AnimationBreakdownPanel.tsx` and throughout phase docs is `font-normal` (400) + `font-semibold` (600). Weight 500 (`font-medium`) is explicitly forbidden project-wide. The active tab renders `font-medium text-accent` at line 1789, which violates this invariant.

```tsx
isActive ? 'font-medium text-accent' : 'font-normal text-fg-muted hover:text-fg',
```

**Fix:** Change `font-medium` to `font-semibold`:

```tsx
isActive ? 'font-semibold text-accent' : 'font-normal text-fg-muted hover:text-fg',
```

The project comment at `AppShell.tsx:1765` even mentions "Active branch: font-medium text-accent (weight 500)." — this comment codifies the violation rather than flagging it. Correct both the code and the comment.
