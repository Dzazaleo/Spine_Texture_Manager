## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| renderer→preload | `window.api.requestPendingUpdate()` call site — sandboxed renderer code |
| preload→main IPC | `ipcRenderer.invoke('update:request-pending')` — crosses sandbox boundary |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-14-02-01 | Tampering | renderer call site passing arguments | mitigate | The bridge signature has zero parameters (`requestPendingUpdate: ()`) — TypeScript at the call site rejects any inbound argument at compile time. At runtime, even if a malicious renderer pushed extra args via `ipcRenderer.invoke`, the main handler in Plan 01 ignores all positional arguments and returns the slot unconditionally. |
| T-14-02-02 | Information Disclosure | bridge exposing more surface than needed | accept | The bridge exposes ONLY the channel name `update:request-pending`; no other channels are reachable via this method. The contextBridge whitelist enforces that `window.api.requestPendingUpdate` is the only renderer-visible reference to this channel. |
| T-14-02-03 | Spoofing | renderer invoking via raw `ipcRenderer.invoke` | accept | `ipcRenderer` is NOT exposed to renderer-land via contextBridge; the renderer only sees `window.api.*` methods. Calling `update:request-pending` outside the bridge would require sandbox escape, which is out of scope for this hotfix. |
| T-14-02-04 | XSS via returned payload | renderer rendering `summary` markdown | accept (existing) | The returned `summary` field passes through `extractSummary()` in main (Phase 12 D-09 — strips HTML, returns plain text). UpdateDialog renders it via `<pre>` (D-09 plain-text-only contract). Phase 14 reuses these contracts unchanged. No new XSS surface introduced by this plan. |

**Block on:** None. All threats accepted with existing mitigations.
