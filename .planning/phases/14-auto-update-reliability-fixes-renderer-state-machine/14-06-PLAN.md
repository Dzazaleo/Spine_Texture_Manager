<objective>
Close gap WR-01 from Phase 14's verification report (`14-VERIFICATION.md` frontmatter `gaps[0]`, also surfaced as G-1 in `14-HUMAN-UAT.md`): wire two production callers of the already-exported `clearPendingUpdateInfo()` into the existing `update:download` and `update:dismiss` IPC handlers so the sticky `pendingUpdateInfo` slot is emptied on every renderer-driven dismissal/download event, then add two regression tests that lock the new behavior.

Purpose: Plan 03's frontmatter must-have truth #11 promised "Clicking 'Later' in UpdateDialog calls `window.api.dismissUpdate(version)` AND clears the sticky slot via the new IPC (defense-in-depth)." Plan 03 shipped without that production caller — `clearPendingUpdateInfo` is exported, JSDoc-documented (auto-update.ts:111-115 explicitly names "the existing `update:dismiss` IPC + Phase 14 Plan 03 download-click handler" as the clearance path), and unit-tested, but `grep -rn "clearPendingUpdateInfo" src/` returns zero production callers. Cold-start auto-update is unaffected (slot is in-memory only, rebuilt every launch per CONTEXT.md D-Discretion-2), but the dangling contract is a latent footgun: any future in-session remount path (HMR, React StrictMode dev cycle, "Reset session" affordance, vitest unmount/remount paths) would let `requestPendingUpdate()` re-deliver an already-dismissed payload because the slot never empties. This plan closes that gap with a ~10-LoC change + 2 regression assertions before Phase 15 ships v1.1.2.

Output: Modified `src/main/ipc.ts` (two existing handlers extended with a destructured `clearPendingUpdateInfo` import + synchronous pre-delegation call) and modified `tests/main/auto-update-dismissal.spec.ts` (two new `it(...)` cases appended to the existing "Phase 14 D-03 update:request-pending sticky slot" describe block — labels `(14-l)` and `(14-m)` continue the existing `(14-a)..(14-k)` convention without collision).

This is the recommended Option A from `14-REVIEW.md` WR-01 (preferred per VERIFICATION's recommendation — keeps the renderer ignorant of main's module state and centralizes the cleanup at the IPC trust boundary rather than inside `dismissUpdate`/`downloadUpdate` themselves). Option B (clearing inside the auto-update.ts functions) was explicitly NOT chosen because it couples persistence-layer functions to the in-memory slot lifecycle.
</objective>
