<objective>
QA-01: Convert `handleToggleRow` and `handleRangeToggle` in `GlobalMaxRenderPanel.tsx` from closure-capturing `setSelected(new Set(selected))` form to the functional `setSelected((prev) => ...)` form. Eliminates a latent stale-closure race where two state updates batched into the same render window would lose the first update's mutation.

Purpose: React state-update correctness — closure over `selected` means the handler reads `selected` at the time it was created, not at the time it runs. Fast keyboard events (Shift+Down repeat, holding Space across multiple rows) can fire two toggles before React flushes the first state update; the second handler still sees the pre-update `selected` and overwrites the first. Functional updater form passes the latest queued state via `prev`.

Output: Two refactored callbacks, plus a vitest regression spec that simulates the rapid-toggle race and asserts the cumulative selection.

Scope discipline: This is a pure refactor — observable behavior is unchanged for sequential clicks. The only scenario where new behavior surfaces is the previously-broken rapid-fire path, which now produces correct cumulative selection. No other handlers, no other components, no UI markup, no styling. Wave 1 because no Phase 26 carry-forwards land in this file.

**SelectAllCheckbox surface disambiguation (verified at planning time):** `SelectAllCheckbox` at lines 568-585 contains `const next = new Set(selected); ... onBulk(next);`. It does NOT call `setSelected(new Set(selected))` — it calls `props.onBulk(next)` (a parent-supplied bulk setter). The grep gate `setSelected\(new Set` therefore catches ONLY the closure-capture anti-pattern in handleToggleRow / handleRangeToggle and never matches SelectAllCheckbox. SelectAllCheckbox is intentionally NOT in QA-01 scope.
</objective>
