QA-02: Disable the Apply button when the input field is empty or whitespace-only. Today, an empty input falls through to `Number('') === 0`, which the AppShell silently clamps to 1% (the dialog has `min={1}` on the input but the keyboard path lets the user clear the field, hit Enter, and submit `0`). We surface the invalidity at the dialog layer instead of relying on downstream clamps.

QA-04: Remove the unreachable `if (!props.open) return null` early-return guard and its associated `open` prop from `OverrideDialogProps`. AppShell already mounts `OverrideDialog` conditionally via `{dialogState !== null && (<OverrideDialog ... />)}` (AppShell.tsx:1616-1626) so the gate inside the dialog is dead code — the component never renders with `open === false`.

Locked decisions (per planning_context):
- D-01 (QA-02): Use **disabled Apply button**, not inline validation message. Matches existing dialog idioms in the project (OptimizeDialog, ConflictDialog have no inline form-level validation banners) and is the simpler patch.

The conditional guard `dialogState !== null && (...)` is the runtime mount/unmount switch — it is preserved verbatim. The dialog's lifetime is now governed entirely by React's mount/unmount of the JSX, not by an internal `open` flag.

**Why useFocusTrap can take `true` literally:** The hook's signature treats the second arg as the "is-active" predicate; once the dialog mounts conditionally, it is by definition always active. Passing `true` is the canonical "always-active" idiom.

**Why useEffect's dep array becomes []:** The select() call should run exactly once on mount (the focus + auto-select behavior). With the old `open` prop, the effect re-ran when `open` flipped true→true→false; in the new mount-driven model the effect runs once on mount and is correct.
