## Typography

The drop-zone copy at App.tsx:621 already declares `text-fg-muted font-mono text-sm`. Phase 32 keeps this paragraph styling and inserts a single inline `<span>` whose only typographic delta is `font-bold`.

| Role | Size | Weight | Line Height | Usage in this phase |
|------|------|--------|-------------|---------------------|
| Body (drop-zone idle copy) | 14px (`text-sm`) | 400 (regular, default for `font-mono`) | Tailwind default ≈ 1.25 (`text-sm` line-height) | **Unchanged.** Outer `<p className="text-fg-muted font-mono text-sm">` |
| Inline emphasis (the `v4.2` token) | 14px (`text-sm`, inherited) | **700 (`font-bold`)** | inherited | **NEW.** Inline `<span className="font-bold text-danger">v4.2</span>` |
| Error banner prefix ("Skeleton not found:") | 12px (`text-xs`) | 600 (`font-semibold`) | inherited | **Unchanged** — App.tsx:675 |
| Error banner body (carries `state.error.message`) | 12px (`text-xs`) | 400 | inherited | **Unchanged** — App.tsx:671/676 |

**Weight contrast intent:** the `v4.2` token uses `font-bold` (700) — a deliberate one-step bump above the project's default emphasis pattern of `font-semibold` (600) seen at App.tsx:675 ("Skeleton not found:"), MissingAttachmentsPanel.tsx:63/73, and AnimationBreakdownPanel.tsx:765. **This bump is REQUIREMENTS-locked** per CONTEXT.md D-04 + REQUIREMENTS.md L14. The intent: `v4.2` is the single load-bearing constraint a user must absorb before dropping a file; bolder weight earns the eye more aggressively than the in-banner `font-semibold` prefix.

**Stack discipline (3 weights total across both surfaces):**
- 400 — paragraph body (default `font-mono`)
- 600 — error banner prefix ("Skeleton not found:") — **pre-existing**
- 700 — `v4.2` inline emphasis — **new**

3 weights edges the soft "max 2 weights" guidance, but the third weight (`font-semibold`) is pre-existing on a non-modified surface; Phase 32 itself only introduces `font-bold` on top of the project's existing two-weight (400/600) baseline. Acceptable.

## Color

Project uses a dark-surface chromatic palette with three semantic accents. Phase 32 introduces no new colors. The `v4.2` token reuses `text-danger` (#e06b55, terracotta), the same color already proven on three surfaces (`App.tsx:675`, `MissingAttachmentsPanel.tsx:63/73`, `panels/AnimationBreakdownPanel.tsx:765`).

| Role | Value | Usage in this phase |
|------|-------|---------------------|
| Dominant (60%) | `--color-surface` = `#1f1f1e` (dark warm gray) | App background, drop-zone background |
| Secondary (30%) | `--color-panel` = `#1f1f1e` (same as surface in current theme); `--color-modal` = `#232731` (dark blue) for elevated layers | Error banner background (`bg-panel`); not directly modified |
| Accent (10%) | `--color-accent` = `#f97316` (orange-500, Spine-adjacent) | NOT USED on Phase 32 surfaces. Reserved for drag-over ring + focus outlines + button hover affordance. |
| Destructive / warning | `--color-danger` = `#e06b55` (terracotta) | **The `v4.2` inline token (NEW).** Pre-existing usage: error banner accent bar (`bg-danger`) + "Skeleton not found:" prefix (`text-danger`). |

**Accent reserved for (project-wide):**
- Drag-over ring on the drop zone
- Focus outline (`outline-accent`)
- Button hover affordance (`hover:border-accent hover:text-accent`) on banner buttons (App.tsx:681, 688)
- Phase 4 over-1.0× scale row pills (`AnimationBreakdownPanel`)

**`text-danger` reserved for (project-wide):**
- Inline error/warning prefixes ("Skeleton not found:" — App.tsx:675)
- Missing-attachments panel headers (`MissingAttachmentsPanel.tsx:63/73`)
- Over-budget signals in panels (`panels/AnimationBreakdownPanel.tsx:765`)
- **NEW (Phase 32):** the `v4.2` token in the drop-zone idle copy — signals "this is a constraint you must satisfy before the load will succeed"

**WCAG contrast (a11y):**
- `text-danger` (#e06b55) on `--color-surface` (#1f1f1e) ≈ **5.33:1** — passes WCAG AA for normal text (≥ 4.5:1) and AAA for large text (≥ 4.5:1). Same surface-pairing as the existing "Skeleton not found:" usage at App.tsx:675; explicitly proven in `index.css:60-66` comment block.
- `font-bold` (700) makes the token effectively "large bold" (still 14px so AA-normal applies; passes regardless).
- The `v4.2` token's contrast envelope is a STRICT SUPERSET of the existing `font-semibold text-danger` neighbor — bolder weight only improves perceived contrast, never degrades it.

## Copywriting Contract

Phase 32 modifies one user-facing string and introduces one new user-facing error message. Both are byte-locked.

### Drop-zone idle copy (MODIFIED — App.tsx:621-623)

**Before (current, 2026-05-10):**
```tsx
<p className="text-fg-muted font-mono text-sm">
  Drop a <code>.spine</code> JSON file anywhere in this window
</p>
```

**After (Phase 32 lock — verbatim from CONTEXT.md D-04):**
```tsx
<p className="text-fg-muted font-mono text-sm">
  Drop a Spine <span className="font-bold text-danger">v4.2</span> <code>.spine</code> JSON file anywhere in this window
</p>
```

**Token-by-token diff:**
- `Drop a` → `Drop a Spine` (insert " Spine")
- `<span className="font-bold text-danger">v4.2</span> ` (insert before `<code>`)
- `<code>.spine</code>` — unchanged
- `JSON file anywhere in this window` — unchanged

**Spacing semantics:** single ASCII space before `<span>`, single ASCII space after `</span>`. Matches App.tsx:675 inline-span pattern.

### Error message (NEW string surfaced via existing banner — App.tsx:660-697)

**Source:** `src/core/errors.ts` `SpineVersionUnsupportedError` constructor (CONTEXT.md D-03 — branch by `detectedVersion`).

**4.3+ branch message (REQUIREMENTS.md L13 verbatim — byte-for-byte lock):**
```
This app currently supports Spine v4.2. Re-export from your 4.3 editor as Version 4.2 (supported downgrade) and try again.
```

**Pre-4.2 branch message (UNCHANGED — preserves Phase 12 F3 contract):**
```
This file was exported from Spine ${detectedVersion}. Spine Texture Manager requires Spine 4.2 or later. Re-export from Spine 4.2 or later in the editor.
```

**How it surfaces in the UI:** the `projectLoadFailed` branch (App.tsx:660-697) renders `state.error.message` verbatim inside the existing banner alongside the `text-danger` "Skeleton not found:" prefix. **NO renderer-side template changes.** The Locate skeleton… button is irrelevant for this error class (no on-disk path to relocate); user sees both buttons but the actionable button is **Dismiss**.

### Copywriting summary

| Element | Copy | Source / lock |
|---------|------|---------------|
| Primary CTA (drop-zone idle) | "Drop a Spine **v4.2** `.spine` JSON file anywhere in this window" | CONTEXT.md D-04 verbatim; `v4.2` is `font-bold text-danger` |
| Empty state | n/a — this phase has no empty-state surface (drop-zone idle IS the empty state and is covered above) | — |
| Error state (4.3+ detection) | "This app currently supports Spine v4.2. Re-export from your 4.3 editor as Version 4.2 (supported downgrade) and try again." | REQUIREMENTS.md L13 byte-lock |

| Destructive confirmation | n/a — phase introduces no destructive actions | — |

**Tone discipline:** both error strings follow the project's "problem, then explicit next step" pattern (also seen in `RotatedRegionUnsupportedError`, `AssetNotFoundError`). The 4.3+ message names the supported version, identifies the downgrade as "supported", and ends with "try again" — no jargon, no engineering terms ("schema", "constraints array"), no editor-version numbers other than 4.2 / 4.3.

**Help-dialog copy (`HelpDialog.tsx:158`):** explicitly OUT of scope per CONTEXT.md `<deferred>`. Today's wording ("Drop a `.json` file (or its parent folder) onto the app window…") does not reference a specific Spine version; the inconsistency is acknowledged and deferred.

**Elevated-fallback advisory (`App.tsx:615-619`):** explicitly OUT of scope per CONTEXT.md D-04 §"Symmetric callsite check". COMPAT-02 only locks the `idle && !isElevated` branch. The elevated branch is a separate concern (DnD blocked at OS level); no v4.2 callout required.

## Interaction Contract

No new interactions. The drop-zone's drag-over ring, focus outline, and drop handler are unchanged. The error banner's two buttons (Locate skeleton… / Dismiss) are unchanged. The `v4.2` `<span>` is non-interactive plain text — no hover state, no tooltip, no click behavior.

**Accessibility:**
- The `v4.2` `<span>` is plain inline text — no `role` attribute needed; screen readers will read it inline as part of the parent `<p>`.
- The bold-color combination is **redundant cueing** — the word "Spine" preceding it provides semantic context, and `v4.2` itself reads correctly without color (the color is the *visual* emphasis; the *semantic* emphasis is carried by the literal "v4.2" token in the sentence). Color-blind users (deuteranopia / protanopia) lose the red-vs-muted-gray contrast but retain the bold-weight differentiation; AAA-compliant.
- The error banner already has `role="alert"` (App.tsx:670) — assistive tech announces the new COMPAT-01 message automatically when the projectLoadFailed state activates. No additional ARIA needed.
