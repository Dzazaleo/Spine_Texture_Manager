# Requirements (v1.0 — final state)

All checkboxes reflect end-of-milestone status. Outcomes annotated where the implementation diverged from the original wording.

## Functional requirements

### F1 — Skeleton loading
- [x] F1.1 Load a Spine 4.2+ skeleton JSON via `spine-core` `SkeletonJson.readSkeletonData`. — `src/core/loader.ts`
- [x] F1.2 Auto-detect companion assets next to the JSON: `.atlas` file and/or an `images/` folder. — atlas auto-discovery wired in Phase 8 D-152
- [x] F1.3 Use a stub `TextureLoader` so PNG decoding is not required for the math phase. — `StubTexture` subclass in `loader.ts`
- [x] F1.4 Surface a clear error when neither atlas nor images folder is available. — typed `SerializableError` envelope (8 kinds)
- [ ] F1.5 Future: support Spine 4.3+ via versioned loader adapters. — **deferred post-v1.0** (out of scope marker)

### F2 — Per-attachment peak render-scale computation
- [x] F2.1 For each `(attachment, skin)` pair, compute the peak world-space AABB across all animations at a configurable sampling rate (default 120 Hz). — `sampleSkeleton` in `src/core/sampler.ts`
- [x] F2.2 When an attachment has no animation timeline touching it, report its setup-pose peak instead. — setup-pose pass per skin
- [x] F2.3 Support `RegionAttachment` (4 world vertices) and `MeshAttachment`/`VertexAttachment` (N world vertices, including weighted meshes). — `bounds.ts`
- [x] F2.4 Correctly reflect bone-chain scale, slot scale, weighted-mesh bone influences, IK constraints, TransformConstraints, PathConstraints, Physics constraints (4.2), DeformTimeline offsets. — verified via golden tests
- [x] F2.5 Report `scaleX`, `scaleY`, and `scale = max(scaleX, scaleY)` separately. — `PeakRecord` shape
- [x] F2.6 Record source animation name, source frame/time, and source skin for every peak. — `PeakRecord.sourceAnimation/frame/skin`
- [x] F2.7 Resolve each attachment's source dimensions via atlas `orig`, PNG metadata, or `RegionAttachment.width/height`. Warn on mismatch. — `loader.ts` provenance check

### F3 — Global Max Render Source panel
- [x] F3.1 Sortable table: Asset, Original Size, Max Render Size, Scale, Source Animation/SetupPose, Frame. — `GlobalMaxRenderPanel`
- [x] F3.2 Search field filters rows by attachment name. — `SearchBar` component
- [x] F3.3 Per-row checkbox for batch operations. — batch-select wiring

### F4 — Animation Breakdown panel
- [x] F4.1 Collapsible per-animation cards, each showing unique asset count (or "No assets referenced"). — `AnimationBreakdownPanel`
- [x] F4.2 "Setup Pose (Default)" shown as its own top card. — Phase 3 layout
- [x] F4.3 Per-row Bone Path shown, plus `source → scale → peak → frame`. — `boneChainPath` module
- [x] F4.4 Per-row "Override Scale" button opens the override dialog. — D-69 unlock in Phase 4

### F5 — Scale overrides
- [x] F5.1 Double-click any peak scale → dialog accepting a percentage (100% = source dimensions). — D-91 semantics rewrite (supersedes original D-78/D-79)
- [x] F5.2 `< 100%` shrinks; `> 100%` clamps to 100% (source dimensions are the absolute maximum). — D-91
- [x] F5.3 Overrides visually badged on affected rows across all panels. — Phase 4 Plan 03
- [x] F5.4 Overrides persist in saved project state. — Phase 8 `.stmproj` v1 schema

### F6 — Unused attachment detection
- [x] F6.1 Flag attachments defined in skins that are never rendered (active slot, non-zero alpha) in any animation in any skin. — `src/core/usage.ts`
- [x] F6.2 Surface as its own panel section. — terracotta-header `<section aria-label="Unused attachments">` above peak table

### F7 — Atlas Preview modal
- [x] F7.1 Before/after side-by-side atlas visualization using `maxrects-packer`. — `AtlasPreviewModal` + `core/atlas-preview.ts`
- [x] F7.2 Show dimensions and estimated file-size delta. — **Reinterpreted per CONTEXT D-127**: dims + page count + per-page efficiency only — no bytes shown (file-size delta would require encoding every region, contradicting the Lanczos quality preservation contract).

### F8 — Optimize Assets (image export)
- [x] F8.1 Export button opens a folder picker. — `OptimizeDialog`
- [x] F8.2 For each attachment, `sharp.resize(W, H, { kernel: 'lanczos3', fit: 'fill' })` and write to `<out>/images/<path>.png`. — `image-worker.ts`
- [x] F8.3 Preserve directory structure of the source `images/` layout. — `runExport` per-row pipeline
- [x] F8.4 Never modify original source files. — atomic Pattern-B write (`.tmp` + `fs.rename`)
- [x] F8.5 Progress UI with per-file error surfacing. — `ExportProgressEvent` IPC channel

### F9 — Save/Load project state
- [x] F9.1 Session JSON contains: skeleton path, atlas/images root, overrides, settings (sampling rate). — `.stmproj` v1 schema
- [x] F9.2 Load restores overrides and settings; recomputes peaks. — Phase 8 9-step Open chain; Phase 8.1 closes 3 reachability gaps (VR-01/VR-02/VR-03)

## Non-functional requirements

### N1 — Correctness
- [x] N1.1 Every `core/` function has golden unit tests driven by `fixtures/SIMPLE_PROJECT/`. — 331 vitest passing at close
- [x] N1.2 Simple leaf-bone attachment: peak matches hand-computed `source × scale`. — golden test
- [x] N1.3 Bone-chain attachment: peak matches `source × product(chain scales) × slot scale`. — golden test
- [x] N1.4 Weighted-mesh attachment: peak matches weighted-sum formula. — differential test (CHAIN_5 doubled → 1.78× CIRCLE worldW)
- [x] N1.5 TransformConstraint-driven bone: peak reflects the constraint automatically. — constrained-vs-unconstrained delta on SQUARE
- [x] N1.6 PhysicsConstraint: deterministic peak across repeated sampler runs (given `Physics.reset` at start). — locked tick lifecycle

### N2 — Performance
- [x] N2.1 Simple rig samples in < 500 ms. — measured ~24 ms on SIMPLE_TEST.json at 120 Hz
- [x] N2.2 Complex rig (~80 attachments, ~16 animations) samples in < 10 s on the main thread; move to worker if slower. — **Phase 9: 606 ms on `fixtures/Girl/TOPSCREEN_ANIMATION_JOKER.json` via `worker_threads` Worker — ~17× under the 10 s contract**
- [x] N2.3 Sampler hot loop does zero filesystem I/O (stub TextureLoader). — Layer 3 arch.spec invariant locked

### N3 — Quality preservation
- [x] N3.1 Optimize Assets uses Lanczos3 resampling, PNG compression level 9, alpha preserved. — `image-worker.ts:runExport` pipeline
- [x] N3.2 Output files must be visually indistinguishable from Photoshop-Lanczos output at the same target dims. — verified in Phase 6 manual UAT

### N4 — Portability
- [ ] N4.1 Ships as signed `.dmg` (macOS) and `.exe` (Windows) at minimum. — **deferred post-v1.0**: `electron-builder` config landed (asarUnpack for sharp), unsigned dev builds work; signing + notarization not in v1.0 scope.
- [x] N4.2 No native compilation required for end users beyond what Electron + sharp bundles provide. — sharp ships prebuilt binaries; electron-builder config wired
