<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Releases → Linux test host | TLS download of AppImage; integrity via GitHub TLS + (later) SHA-512 from latest-linux.yml during auto-update |
| AppImage launch → libfuse2 dlopen | OS dynamic linker — failure mode is well-characterized and not a security concern (graceful error, no crash) |
| Docker container → host filesystem | Read-only AppImage download in container; PNG capture happens on host (terminal screenshot); no container-host file mount needed |
| Linux Help → Installation Guide | Same allow-list path as macOS (ipc.ts:174 SHELL_OPEN_EXTERNAL_ALLOWED) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-12.1-04-01 | Tampering | docs/install-images/linux-libfuse2-error.png | mitigate | PNG magic-bytes + ≥5KB size both gated by tests/integration/install-md.spec.ts |
| T-12.1-04-02 | Spoofing | Docker container as capture host | accept | Docker is acceptable per RESEARCH §Pitfall 6 fallback; provenance recorded in transcript (which path used + Ubuntu version) |
| T-12.1-04-03 | Information Disclosure | terminal screenshot may capture username/hostname | mitigate | Tester clears terminal scrollback before capture; capture only the relevant error block; review PNG before commit |
| T-12.1-04-04 | Tampering | AppImage hot-swap during auto-update | accept | electron-updater handles AppImage replacement via documented API; no privilege escalation; runs at user's own privilege level |
| T-12.1-04-05 | Denial of Service | libfuse2 dependency on Linux | mitigate | INSTALL.md documents the prerequisite; the dlopen error IS the user-facing signal to install libfuse2; graceful degradation by design |
</threat_model>
