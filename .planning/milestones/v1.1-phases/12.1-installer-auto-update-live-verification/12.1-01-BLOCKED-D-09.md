## Root cause

electron-builder's YAML `publish:` field accepts:
- `null` — suppress publisher configuration entirely (what `679cffc` did; falsified by Assumption A6 in `12.1-01-BLOCKED.md` because it ALSO suppresses `latest-*.yml` feed-file generation).
- A built-in publisher type name: `github`, `s3`, `bintray`, `generic`, `snapStore`, `keygen`, `space-store`, `bittorrent`. Each maps to a built-in publisher class that constructs feed-file metadata AND can upload.
- An array of the above.
- A custom publisher name. electron-builder will try to dynamically `require()` `build/electron-publisher-<name>.{mjs,js,cjs}` (this is the convention for user-defined publisher plugins, e.g., `electron-publisher-custom-s3.js`).

The literal string `never` matches none of the built-in types. electron-builder therefore treats it as a custom publisher name and looks for a plugin module named `electron-publisher-never.{mjs,js,cjs}` in `build/`. Since no such file exists, the build aborts with:

```
⨯ unable to find publish provider in build resources  path=build template=electron-publisher-never extensionsChecked=["mjs","js","cjs"]
```

This is independent of (and not suppressed by) the `--publish never` CLI flag. The CLI flag controls what happens AFTER electron-builder has constructed the publisher chain (it skips the upload step). The YAML field controls WHICH publisher chain is constructed in the first place. The two mechanisms operate at different layers of the build pipeline.

D-09 conflated these two mechanisms — the hypothesis assumed `never` was a YAML keyword equivalent to the CLI flag's behavior. It is not.
