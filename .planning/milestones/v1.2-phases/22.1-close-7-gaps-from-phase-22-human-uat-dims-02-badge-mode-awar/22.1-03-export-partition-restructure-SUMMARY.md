## What Changed

### `src/core/export.ts`

**Acc interface** (dedup accumulator): removed `isPassthrough: boolean` (decision moved to emit loop). Added `isCapped: boolean` (true when `downscaleClampedScale > sourceRatio`).

**Dedup loop** (cap math): deleted `peakAlreadyAtOrBelowSource` and pre-computed `isPassthrough` (3 lines). Kept `sourceRatio`, `cappedEffScale`, `downscaleClampedScale`, `isCapped` as defense-in-depth.

**Emit loop** (new predicate at lines ~254-288):
```typescript
const outW = Math.ceil(acc.row.sourceW * acc.effScale);
const outH = Math.ceil(acc.row.sourceH * acc.effScale);
const isPassthrough = outW === acc.row.sourceW && outH === acc.row.sourceH;
```
`isCapped` spread onto ExportRow when cap binds (`...(acc.isCapped ? { isCapped: true } : {})`).

## Override Application Site (consumed by plans 22.1-04 and 22.1-05)

Override is applied **INSIDE** `buildExportPlan`, at the dedup accumulator loop:
```typescript
const overridePct = overrides.get(row.attachmentName);
const rawEffScale =
  overridePct !== undefined
    ? applyOverride(overridePct).effectiveScale
    : row.peakScale;
```
This runs BEFORE cap math AND before the `isPassthrough` predicate. The post-cap predicate IS automatically post-override. Plans 22.1-04 and 22.1-05 can rely on the ExportRow's `effectiveScale` reflecting any user override.

## Cap Math Kept as Defense-in-Depth

After 22.1-01's unified actualSource model, in atlas-source mode `actualSourceW === sourceW` (both from `region.originalWidth`) so `sourceRatio = 1.0` and the cap is a no-op. However, in atlas-LESS mode with halved PNGs, `sourceW (PNG header) != canonicalW (JSON dims)` so `dimsMismatch = true` and `sourceRatio ≈ 0.5`. The cap still fires. Keeping the cap math preserves the `outW ≤ sourceW always` invariant regardless of future loader edits (T-22.1-09).
