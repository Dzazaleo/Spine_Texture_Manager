### D-158: VR-02 fix — WR-02 path (a) [LOCKED]
Extend `SerializableError` in `src/shared/types.ts` so the `'SkeletonNotFoundOnLoadError'` kind carries the cached recovery payload and the other kinds keep their existing shape `{ kind, message }`. Convert the type from a single interface to a discriminated union:

```ts
export type SerializableError =
  | {
      kind: 'SkeletonNotFoundOnLoadError';
      message: string;
      // Cached recovery payload — fed to handleProjectReloadWithSkeleton
      // when the user picks a replacement skeleton (D-149 chain).
      projectPath: string;
      originalSkeletonPath: string;
      mergedOverrides: Record<string, number>;
      samplingHz: number;
      lastOutDir: string | null;
      sortColumn: string | null;
      sortDir: 'asc' | 'desc' | null;
    }
  | {
      kind:
        | 'SkeletonJsonNotFoundError'
        | 'AtlasNotFoundError'
        | 'AtlasParseError'
        | 'ProjectFileNotFoundError'
        | 'ProjectFileParseError'
        | 'ProjectFileVersionTooNewError'
        | 'Unknown';
      message: string;
    };
```

**Rationale:** WR-02's recommended fix (a) — "principled, both Open paths get parity recovery data". WR-02's fix (b) — "hide the Locate button when path is empty" — is rejected because it papers over the bug instead of fixing it; users on the toolbar Open path would still hit the missing-skeleton dead end.

**Wire effect:** purely renderer-IPC envelope. On-disk `.stmproj` v1 schema unchanged (D-171).

### D-163: VR-03 fix — callback-ref bridge [LOCKED]
App.tsx holds a `useRef<((fileName: string, kind: 'json' | 'stmproj') => Promise<boolean>) | null>(null)` and passes a stable wrapper to `DropZone.onBeforeDrop`:

```ts
const beforeDropRef = useRef<((fileName: string, kind: 'json' | 'stmproj') => Promise<boolean>) | null>(null);

const handleBeforeDrop = useCallback(async (fileName: string, kind: 'json' | 'stmproj') => {
  return beforeDropRef.current?.(fileName, kind) ?? true;
}, []);
```

AppShell receives a new optional prop `onBeforeDropRef?: React.MutableRefObject<((fileName: string, kind: 'json' | 'stmproj') => Promise<boolean>) | null>` and registers/unregisters its dirty-guard impl in a `useEffect`:

```ts
useEffect(() => {
  if (!onBeforeDropRef) return;
  onBeforeDropRef.current = (fileName, kind) =>
    new Promise<boolean>((resolve) => {
      if (!isDirty) { resolve(true); return; }
      const reason = kind === 'json' ? 'new-skeleton-drop' : 'new-project-drop';
      setSaveQuitDialogState({
        reason,
        pendingAction: () => resolve(true),
        cancelAction: () => resolve(false),
      });
    });
  return () => {
    onBeforeDropRef.current = null;
  };
}, [isDirty, onBeforeDropRef]);
```

**Rationale (vs. lifting state):** dirty derivation depends on overrides + lastSaved + samplingHz, all of which live in AppShell (AppShell:103-137, 450-460). Lifting them to App.tsx is a much bigger refactor and undoes Phase 8's plan that AppShell owns Save/Load state. The ref-bridge is the smallest correct change.

### D-167: SaveQuitDialog 'new-skeleton-drop' semantics — Don't Save discards overrides
When the user clicks **Don't Save** during a `new-skeleton-drop` / `new-project-drop` confirmation, the resolver fires `resolve(true)`, DropZone proceeds, the existing skeleton is replaced (App.tsx remounts AppShell), and the in-memory overrides Map dies with the unmount. **Save** awaits `onClickSave` first, then resolves true on success (or stays open on save failure — same pattern as `'quit'` reason). Behavior matches the existing 'quit' flow exactly except for the cancel path.
