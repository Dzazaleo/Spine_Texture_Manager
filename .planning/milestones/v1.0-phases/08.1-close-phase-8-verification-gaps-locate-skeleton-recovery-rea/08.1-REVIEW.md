## Warnings

### WR-01: `KNOWN_KINDS` set is narrower than the type alias `KnownErrorKind`

**File:** src/main/ipc.ts:69-75

**Issue:** `KnownErrorKind` is `Exclude<SerializableError['kind'], 'SkeletonNotFoundOnLoadError'>`, evaluating to a 7-member union: `'SkeletonJsonNotFoundError' | 'AtlasNotFoundError' | 'AtlasParseError' | 'ProjectFileNotFoundError' | 'ProjectFileParseError' | 'ProjectFileVersionTooNewError' | 'Unknown'`. But the runtime `KNOWN_KINDS: ReadonlySet<KnownErrorKind>` only contains the first three. The check at line 259 (`KNOWN_KINDS.has(err.name as KnownErrorKind)`) therefore only accepts the three skeleton/atlas kinds and silently routes everything else (including the three `ProjectFile*Error` kinds and the literal `'Unknown'`) through the `kind: 'Unknown'` fallback at line 269. Works today only because `handleSkeletonLoad` uses `loadSkeleton` directly (not project-file parsing), so `ProjectFile*Error` kinds genuinely cannot escape it. A future refactor that adds project-file paths to `handleSkeletonLoad` would silently degrade typed errors to `'Unknown'`.

**Fix:** Either tighten `KnownErrorKind` to match the runtime set (preferred — compiler-enforced):
```ts
type KnownErrorKind = 'SkeletonJsonNotFoundError' | 'AtlasNotFoundError' | 'AtlasParseError';
const KNOWN_KINDS: ReadonlySet<KnownErrorKind> = new Set<KnownErrorKind>([
  'SkeletonJsonNotFoundError',
  'AtlasNotFoundError',
  'AtlasParseError',
]);
```
or document the asymmetry explicitly with a comment block above `KNOWN_KINDS` calling out that `handleSkeletonLoad` only forwards SpineLoaderError subclasses defined in `core/errors.ts`.

### WR-02: `handleProjectReloadWithSkeleton` does not validate `mergedOverrides` value types in rescue branch

**File:** src/main/project-io.ts:521-526, 549-562

**Issue:** When `loadSkeleton` throws `SkeletonJsonNotFoundError` on the user-picked replacement skeleton (line 542), the rescue branch threads `a.mergedOverrides as Record<string, number>` (line 556) into the new envelope without re-validating that the values are finite numbers. The successful path at lines 595-601 *does* validate (`typeof percent !== 'number' || !Number.isFinite(percent)` skip), but the failure path bypasses this. A misbehaving caller could leak `NaN` / `Infinity` / non-numeric values back to the renderer through the recovery envelope. In practice, the renderer's only caller (AppShell.onClickLocateSkeleton + App.handleLocateSkeleton) sources the value from a previously-validated envelope, so this is defense-in-depth rather than an active bug.

**Fix:** Sanitize once at the top of the function (after the `typeof a.mergedOverrides !== 'object'` check at line 521) and use the sanitized copy in both the rescue branch and the success branch:
```ts
const sanitizedOverrides: Record<string, number> = {};
for (const [k, v] of Object.entries(a.mergedOverrides as Record<string, unknown>)) {
  if (typeof v === 'number' && Number.isFinite(v)) sanitizedOverrides[k] = v;
}
```

### IN-02: Duplicated locate-skeleton recovery flow in App.tsx and AppShell.tsx

**Files:** src/renderer/src/App.tsx:129-163 and src/renderer/src/components/AppShell.tsx:596-619

### IN-03: 7-field recovery payload pattern duplicated across 4 sites

**Files:** src/shared/types.ts:506-519, src/main/project-io.ts:367-377, src/main/project-io.ts:551-561, src/renderer/src/components/AppShell.tsx:178-187

The 7 recovery fields are spelled out as object literals at four sites. The shared types docblock at line 480 explicitly forbids extracting an `atlasSource` named type ("precedent is duplication"), but the recovery payload is more semantically coupled — each field is meaningful only as part of the complete payload.

**Fix:** Introduce a named type in `src/shared/types.ts`:
```ts
export interface SkeletonNotFoundRecoveryPayload {
  projectPath: string;
  originalSkeletonPath: string;
  mergedOverrides: Record<string, number>;
  samplingHz: number;
  lastOutDir: string | null;
  sortColumn: string | null;
  sortDir: 'asc' | 'desc' | null;
}
```
Stylistic preference; leave as-is if "duplication over abstraction" precedent is intentional.

### IN-05: Test fixture `makeSummary()` peaks lack required `DisplayRow` fields

**File:** tests/renderer/save-load.spec.tsx:31-42

The `peaks` array uses `as any` casts on each row and the whole return value uses `as unknown as SkeletonSummary` because the synthetic rows omit fields `DisplayRow` requires. The 8.1-VR-03a/03b tests then depend on the renderer formatting `peakScale: 0.5` as `"0.500×"` — a contract currently locked by D-45/D-46 in `core/analyzer.ts`, NOT the renderer. If a future refactor moves the formatting responsibility, these tests would fail in a confusing way.

**Fix:** Either populate `scaleLabel: '0.500×'` and `scaleLabel: '1.000×'` on the synthetic rows so the test depends on its own fixture, or build the fixture via a real `core/analyzer.ts` round-trip helper.
