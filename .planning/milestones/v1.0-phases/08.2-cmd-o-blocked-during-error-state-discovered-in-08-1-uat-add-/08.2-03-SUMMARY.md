`registerIpcHandlers()` gains one new registration AFTER the existing `project:reload-with-skeleton` handler at line 538:

```ts
ipcMain.on('menu:notify-state', async (_evt, state: unknown) => {
  if (!state || typeof state !== 'object') return;
  const s = state as Record<string, unknown>;
  if (typeof s.canSave !== 'boolean') return;
  if (typeof s.canSaveAs !== 'boolean') return;
  if (typeof s.modalOpen !== 'boolean') return;
  const next = { canSave: s.canSave, canSaveAs: s.canSaveAs, modalOpen: s.modalOpen };
  const { applyMenu, getMainWindow, setCurrentMenuState } = await import('./index.js');
  setCurrentMenuState(next);
  void applyMenu(next, getMainWindow());
});
```

Three `typeof` boolean checks + non-object guard form the trust-boundary gate (T-08.2-03-01 mitigation; mirror of `validateExportPlan` discipline at ipc.ts:106-126).

## Patterns lifted

- **Trust-boundary input validation at IPC entry** — `validateExportPlan` (ipc.ts:106-126) + `handleProjectSave` inline checks (project-io.ts:97-112) → applied verbatim to the new `menu:notify-state` handler. Three `typeof` boolean checks + a non-object guard, all returning silently (one-way channel — no envelope shape to fill).
- **One-way fire-and-forget IPC** — `'export:cancel'` (ipc.ts:496-500) + `'project:confirm-quit-proceed'` (index.ts:91-97) → applied to `'menu:notify-state'`. `ipcMain.on` (NOT `ipcMain.handle`); main returns void; preload uses `ipcRenderer.send` (NOT `.invoke`).
- **Pitfall 9 listener-identity preservation** — `onExportProgress` (preload:126-132) + `onCheckDirtyBeforeQuit` (preload:220-226) → applied verbatim to all 4 `onMenu*` methods. The `wrapped` const is captured BEFORE `ipcRenderer.on`; the unsubscribe closure references the SAME `wrapped` so `removeListener` reaches the same reference.
- **Dynamic import for cycle deferral** — first use in this codebase. The pattern: when module A imports from module B, and module B imports from module A, eager imports tighten the cycle in ways that make module-load order fragile (especially under vitest mocking). `await import('./index.js')` inside a function body defers the module evaluation until the function is first called, by which time the cycle has fully resolved. Production safe because index.ts is the app entry point.
- **Side-effect swallow on non-critical failures** — `recent.ts loadRecent` (Plan 01) → applied to the addRecent + applyMenu call sites. The user's Save As / Open already succeeded; a missing recent-list bump or stale menu state is not worth surfacing as an error envelope (would clobber the legitimate ok:true returns).
- **Wire side effect after success arm** — `setStaleOverrideNotice(null)` after successful save at `AppShell.tsx:520` → applied to project-io.ts: `if (resp.ok) { try { await addRecent(...); } catch { ... } ... }`.

## Threat model — mitigations applied

- **T-08.2-03-01** (Tampering: malicious `'menu:notify-state'` payload): mitigated by the field-by-field `typeof` validation gate in the handler. Non-object → silent return. Missing or non-boolean canSave / canSaveAs / modalOpen → silent return. The `as Record<string, unknown>` narrowing does NOT walk prototypes (the three `typeof` checks read own properties only). Verified by `tests/main/ipc.spec.ts` case 2 (three sub-cases: missing field, non-boolean, null).
- **T-08.2-03-02** (Spoofing: compromised renderer lies about modalOpen): accepted per the threat register — even if the renderer pushes `{canSave:true, canSaveAs:true, modalOpen:false}` while a modal is open, clicking File→Save fires `menu:save-clicked` → renderer's `onMenuSave` handler routes through AppShell's `onClickSave`, which is itself defended by AppShell's modal-state derivation. No privileged action results from a stale menu state.
- **T-08.2-03-03** (Tampering: malicious recent.json fed via `menu:open-recent-clicked`): mitigated end-to-end by Plan 01's `validateRecentFile` (rejects non-string entries) combined with `window.api.openProjectFromPath` routing through `handleProjectOpenFromPath`'s existing `.stmproj` extension validation (Phase 8 D-145) + absoluteness check + sibling-path discipline (Phase 1 D-09). Three layers of defense between recent.json and an actual `.stmproj` load.
- **T-08.2-03-04** (DoS: notifyMenuState flood): accepted per CONTEXT discretion — menu rebuild is cheap (template construction + Menu.setApplicationMenu); AppShell's `useEffect` keying on a small array of state slots produces O(state-changes) calls, not O(time).
- **T-08.2-03-05** (Information Disclosure: paths leak via console.log): accepted — no logging hook added in this plan.
- **T-08.2-03-06** (Tampering: malicious recent.json points at `.exe`): mitigated by `handleProjectOpenFromPath`'s `.stmproj` extension check (Phase 8 D-145) — non-stmproj entries surface as the typed-error envelope flow in App.tsx; the user sees the recovery banner, not a silent execution.
- **T-08.2-03-07** (Repudiation: Save As bumps without confirmation): accepted — Save As is its own confirmation; matches macOS Pages / Keynote / etc. behaviour.

**1. [Rule 3 — Blocking issue] Switched from eager to dynamic `import('./index.js')` in `src/main/ipc.ts` AND `src/main/project-io.ts`**

- **Found during:** Task 2 (full suite run) and Task 3 (first project-io.spec.ts run).

The static module graph no longer pulls index.ts at module-load time; spec files that don't traverse the menu code path see the import as an inert reference until the function is invoked. Production load order is safe because index.ts is the app entry point — fully loaded before any handler runs.
