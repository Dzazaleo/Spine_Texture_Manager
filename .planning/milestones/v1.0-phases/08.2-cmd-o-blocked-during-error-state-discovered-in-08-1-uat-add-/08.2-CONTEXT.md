## Phase Boundary

Phase 08.2 ships a **native Electron application Menu** in [src/main/index.ts](src/main/index.ts) and fixes the Cmd+O gating bug discovered in Phase 08.1 UAT (Cmd+O does not fire when AppShell is unmounted — i.e., in `idle`, `error`, or `projectLoadFailed` AppState). Both pieces touch the same Electron `Menu` + IPC accelerator surface, so they ship together.

The new Menu carries the standard macOS top-level layout (App-name / File / Edit / View / Window / Help) so the app stops feeling "broken" on macOS and so Cut/Copy/Paste continue working inside text inputs (OverrideDialog input, Save As filename, etc.) once the default Electron menu is replaced. The custom File menu items are: **Open** (Cmd+O), **Open Recent →** (submenu of up to 10 paths + Clear Menu), **Save** (Cmd+S), **Save As…** (Cmd+Shift+S), **Close Window** (Cmd+W via role).

Menu accelerators fire **regardless of renderer AppState** — that's the entire point of moving the keyboard shortcut into the application menu instead of an `AppShell`-mounted `keydown` listener. The 08.1 UAT discovery (Cmd+O blocked while recovery banner / "Not a .json or .stmproj file" toast is showing) is closed by D-173/D-176.

Phases 0–7 LOCKED. Phase 8 / 8.1 LOCKED (the recovery banner surface from 8.1 D-161/D-162 is the *consumer* of menu-driven Cmd+O in error states — exactly what 08.2 fixes). Sampler stays untouched. CLI stays byte-for-byte (Phase 5 D-102). `.stmproj` v1 schema unchanged (Phase 8 D-145, Phase 8.1 D-171). Layer 3 invariant intact: `src/core/*` gets no new code; `recent.json` read/write lives in main only.

## Implementation Decisions

### Menu strategy / Cmd+O fix (Area 1)

- **D-173: Native Electron Menu via `Menu.buildFromTemplate` + `Menu.setApplicationMenu` in `src/main/index.ts`.** Cmd+O / Cmd+S / Cmd+Shift+S live as menu accelerators on File menu items. Accelerators fire regardless of renderer AppState — fixes the 08.1 UAT discovery (Cmd+O blocked while AppShell unmounted) in one shot, and ships the File menu in the same commit. Rejected: lift the keydown listener up from AppShell to App.tsx (defers the File menu work — Phase goal explicitly names the menu); both menu + listener (overkill — Electron's native menu accelerator is the canonical solution and works in every state on its own).

- **D-174: Full standard macOS template — App-name / File / Edit / View / Window / Help.** Menu structure:
  - **App menu** (macOS-only positioning via `role: 'appMenu'`): About, Hide / Hide Others / Show All, Quit Spine Texture Manager (Cmd+Q).
  - **File menu** (custom items per D-173, D-185, plus role: 'close' for Close Window):
    - Open… (Cmd+O)
    - Open Recent ► (submenu — D-180)
    - ── separator ──
    - Save (Cmd+S)
    - Save As… (Cmd+Shift+S)
    - ── separator ──
    - Close Window (Cmd+W via `role: 'close'`)
  - **Edit menu** (via `role: 'editMenu'` per D-188): Undo / Redo / Cut / Copy / Paste / Paste and Match Style / Delete / Select All. Free standard accelerators that work in all text inputs (OverrideDialog input, Save As filename input, planned future Settings modal inputs).
  - **View menu** (custom — only in dev mode): Reload / Force Reload / Toggle DevTools (DevTools-related items gated on `!app.isPackaged` per [src/main/index.ts:137](src/main/index.ts#L137)).
  - **Window menu** (via `role: 'windowMenu'`): Minimize / Zoom / standard window items.
  - **Help menu** (via `role: 'help'`): empty / placeholder for now (Phase 9 may add documentation links).
  Rejected: File-menu only (strips macOS App/Edit menus — Cmd+C/V stops working in modal text inputs, Cmd+Q has no UI surface, app feels broken); File + Edit only (covers must-haves but skips standard Window/View — non-conventional for desktop apps).

- **D-175: Menu click → one-way IPC to renderer → renderer triggers existing flow.** Menu Open click sends `'menu:open-clicked'` (no payload); Open Recent click sends `'menu:open-recent-clicked'` with the picked absolute path; Save / Save As click sends `'menu:save-clicked'` / `'menu:save-as-clicked'`. App.tsx (always mounted) listens via `window.api.onMenuOpen` / `onMenuOpenRecent` / `onMenuSave` / `onMenuSaveAs`. Each handler routes to the same `window.api.openProject()` / `window.api.openProjectFromPath(path)` / AppShell's `onClickSave` / AppShell's `onClickSaveAs` the existing toolbar/drag-drop entry points use. **Reuses the existing renderer-side state machine; menu is just another trigger.** Rejected: menu → main calls handler directly → push state to renderer (bypasses the renderer's state machine, distributes Open/Save logic across main and renderer, makes dirty-guard duplication likely).

- **D-176: Remove the `AppShell.tsx:626-643` `keydown` listener entirely.** Native menu accelerator is the single source of truth for Cmd+O / Cmd+S / Cmd+Shift+S. No renderer-side keydown listener for save/open shortcuts. The `[role="dialog"]` modal-suppression heuristic moves into the `notifyMenuState` push (D-184): when a modal is open, the menu-state IPC marks `modalOpen: true` and main rebuilds the menu with File items disabled. Avoids double-firing risk. Rejected: keep the listener as fallback (two paths to the same action; risk of double-firing unless one stops propagation; more code surface).

### Open Recent persistence (Area 2)

- **D-177: Persist recent paths at `app.getPath('userData')/recent.json`.** Hand-rolled read/write in `src/main/recent.ts` matches the project's hand-rolled-over-deps discipline (Phase 8 D-156, Phase 4 D-81). Atomic-write via `<path>.tmp` + `fs.rename` mirrors Phase 6 image-worker + Phase 8 project-io patterns. Schema:
  ```ts
  type RecentFileV1 = {
    version: 1;
    paths: string[]; // ordered front=newest
  };
  ```
  `validateRecentFile(unknown): { ok: true; file: RecentFileV1 } | { ok: false }` is a hand-rolled type guard (no `zod`). Missing / corrupt / unknown-version files load as `[]` silently — recent.json is non-critical UX state and must not block app launch on any malformed-state. Rejected: store inside `.stmproj` (recent is user/machine-global, not per-project); store via `electron-store` (adds dependency, inconsistent with discipline); defer to Phase 9 (phase goal explicitly names Open Recent persistence).

- **D-178: 10 recent slots.** macOS standard for File → Open Recent across most document apps (Pages, Keynote, Sketch, Spine Editor itself). New entries push to the front; oldest evicted when length > 10. Rationale: comfortable balance between menu clutter and history depth.

- **D-179: Click-and-recover for missing files.** Show every recent unconditionally — no `fs.access` at menu-build time. The menu Open Recent submenu lists every path verbatim (basename label, full path tooltip per discretion below). On click, the existing project-open path runs (`window.api.openProjectFromPath(path)`); if the file is gone, the existing typed-error envelope flow surfaces in App.tsx as the **Phase 8.1 D-161 / D-162 recovery banner**. Item stays in the recent list until user clicks Clear Menu. Rejected: auto-prune on launch (loses the diagnostic signal — user may not realize a moved folder); probe-on-menu-build (adds I/O per menu render; macOS rebuilds menus often; perf cost for low value).

- **D-180: Lifecycle — add path on Open OR successful Save As; Clear Menu item at submenu bottom.** A path is added to recent when:
  - User successfully Opens a `.stmproj` (drag-drop, toolbar Open, menu Open, menu Open Recent → re-bumps to front).
  - User successfully Saves As (after the new path is written; the new path goes to the front).
  Save (overwrite at existing path) does NOT change ordering — already at front from the original Open. Submenu structure when paths.length > 0:
  ```
  Open Recent ►
    1. RigA.stmproj
    2. RigB.stmproj
    …
    ── separator ──
    Clear Menu
  ```
  Submenu when paths.length === 0:
  ```
  Open Recent ►
    (empty placeholder, disabled)
    ── separator ──
    Clear Menu (disabled — nothing to clear)
  ```
  Or simpler: hide the entire submenu when empty (planner's call — see Discretion). Clear Menu fires `clearRecent()` in main and rebuilds the menu. **No confirmation dialog** — single-click clear matches macOS convention. Rejected: add only on Open (Save As-as-new should appear in Recent, matches user expectation); add on every Save event (over-bumping that doesn't change content).

### Menu item enabled/disabled rules + dirty-guard (Area 3)

- **D-181: Renderer pushes `'menu:notify-state'` IPC on change; main rebuilds + reapplies the menu on every notify.** Channel payload:
  ```ts
  type MenuStateNotify = {
    canSave: boolean;       // summary loaded
    canSaveAs: boolean;     // summary loaded (same as canSave today)
    modalOpen: boolean;     // any [role="dialog"][aria-modal="true"] mounted
  };
  ```
  Recent list is fetched by main from `recent.ts` directly on each rebuild — renderer doesn't push it. AppShell calls `window.api.notifyMenuState(...)` in a `useEffect` keyed on `[summary, dialogState, optimizeDialogOpen, atlasPreviewOpen, saveQuitDialogState]`. App.tsx calls it in the `idle` / `error` / `projectLoadFailed` branches with `{ canSave: false, canSaveAs: false, modalOpen: false }`. Main handler:
  ```ts
  ipcMain.on('menu:notify-state', async (_e, state: MenuStateNotify) => {
    const recent = await loadRecent();
    Menu.setApplicationMenu(buildAppMenu({ ...state, recent }));
  });
  ```
  Rejected: always-enabled (greyed-when-unavailable is a strong UX signal users expect on macOS; clicking Save with nothing loaded shouldn't fire-and-no-op silently); two static templates (loaded vs not-loaded — not granular enough to capture modal-open).

- **D-182: Save and Save As enabled when summary loaded; no dirty-check.** Save when clean is a no-op — matches Photoshop / Pages / VSCode / Spine Editor behavior. Avoids the IPC churn of notifying main on every override flip. The dirty marker (`•` prefix on the filename chip per Phase 8 D-144) is the visual signal for "you have unsaved work"; the menu Save item being enabled even when clean is acceptable. Rejected: Save enabled only when isDirty (extra IPC traffic per override; cleaner visual but the dirty marker already conveys the signal); Save As enabled with no project (fringe edge case — writing an empty untitled defaults to a chosen path is non-standard).

- **D-183: Menu-driven Open / Open Recent honors the same `SaveQuitDialog` dirty-guard as drag-drop and toolbar Open.** The flow reuses Phase 8 D-143 + Phase 8.1 D-163's `onBeforeDropRef` ref-bridge: App.tsx's `onMenuOpen` / `onMenuOpenRecent` handlers call `handleBeforeDrop(displayName, kind)` (same wrapper DropZone uses) before dispatching the open. When `isDirty`, AppShell's already-registered ref-impl mounts `SaveQuitDialog` with `reason='new-project-drop'` (planner may add a new `'menu-open'` reason to differentiate body copy if desired — see Discretion). Cancel resolves false → menu Open aborts. Save → resolves true after persisting. Don't Save → resolves true and discards. Same exact flow that DropZone uses for `.json` / `.stmproj` drops. Rejected: no gate (silent data-loss footgun, diverges from drag-drop behavior); gate only Cmd+O but not File→Open menu (inconsistent across menu entry points).

- **D-184: While any modal is open, File menu items (Open / Open Recent / Save / Save As) are disabled.** Menu-state push includes `modalOpen: boolean` — derived from existing renderer state (`!!dialogState || !!optimizeDialogOpen || !!atlasPreviewOpen || !!saveQuitDialogState`). When `modalOpen === true`, `buildAppMenu` greys all four File items via `enabled: false`. Edit menu (Cut/Copy/Paste/SelectAll/Undo/Redo) stays enabled — must work in modal text inputs. View / Window / Help / App-menu also unaffected. Rejected: keep menu items enabled and let modals defensively no-op (menu visually misrepresents what's clickable on macOS; users would file it as a bug); suppress only Save/Save As (Open should also be suppressed — opening a new project under an open OverrideDialog would orphan unsubmitted dialog input).

### Save As shortcut + Win/Linux visibility (Area 4)

- **D-185: Save As accelerator = `CmdOrCtrl+Shift+S`.** macOS / Windows / Linux standard for Save As across nearly every document app (Pages, Photoshop, VSCode, Spine Editor itself). Lives entirely in the application Menu — no renderer-side keydown listener. Electron's `'CmdOrCtrl+Shift+S'` accelerator string handles cross-platform conversion automatically (Cmd on macOS, Ctrl on Win/Linux). Rejected: menu-only (loses muscle memory); `CmdOrCtrl+Option+S` / `CmdOrCtrl+Alt+S` (used by some IDEs but uncommon for document apps).

- **D-186: Keep `autoHideMenuBar: true` on Windows/Linux (Phase 1 default).** macOS unaffected — the menu lives in the system menu bar regardless of this flag. On Win/Linux, the application menu stays hidden by default; users access it via Alt or F10 (standard Win/Linux convention for hidden menu bars). Cmd+O / Cmd+S / Cmd+Shift+S accelerators work regardless of menu visibility. Rationale: the app already has a toolbar with Save/Open buttons + filename chip + Optimize Assets + Atlas Preview; an always-visible menu bar adds ~28px of redundant chrome without UX win. Diverging from this is a Phase 9 polish call if Win/Linux user testing flags it. Rejected: `autoHideMenuBar: false` on all platforms (always-visible menu bar on Win/Linux is more chrome for low marginal value); platform-specific (`process.platform === 'win32' ? false : true`) — violates Phase 1 D-23 zero-Windows-specific-code rule.

- **D-187: In `status='projectLoadFailed'` (App.tsx Phase 8.1 D-161 recovery banner), File menu state = Open enabled, Open Recent enabled, Save disabled, Save As disabled.** App.tsx's `projectLoadFailed` branch registers a `useEffect` that calls `window.api.notifyMenuState({ canSave: false, canSaveAs: false, modalOpen: false })`. Cmd+O routes through App.tsx's `onMenuOpen` listener (no dirty-guard needed — there's no dirty session in this state; `handleBeforeDrop` resolves true via the `?? true` fallback when AppShell isn't mounted to register its ref-impl). Recovery banner is **NOT** a `[role="dialog"]` modal, so the `modalOpen` suppression rule does not apply. **This is exactly the bug Phase 08.2 fixes** — Cmd+O must work here, and post-08.2 it does, end-to-end. Rejected: disable everything until banner is dismissed (forces a friction click; contradicts the entire phase goal).

- **D-188: Use Electron `role`-based menu items where standard accelerators exist.** Specifically: `{ role: 'appMenu' }` (App-name with About / Hide / Quit), `{ role: 'editMenu' }` (Undo / Redo / Cut / Copy / Paste / Paste and Match Style / Delete / Select All), `{ role: 'viewMenu' }` (Reload / Force Reload / Toggle DevTools / Reset Zoom / Zoom In / Zoom Out / Toggle Fullscreen — but DevTools-related entries gated on `!app.isPackaged` per [src/main/index.ts:137](src/main/index.ts#L137)), `{ role: 'windowMenu' }` (Minimize / Zoom / standard items), `{ role: 'help' }` (placeholder). Custom (`label` + `accelerator` + `click`) is used only for the File menu's project-specific items (Open / Open Recent submenu / Save / Save As). Rationale: free standard shortcuts, platform-correct labels, zero drift risk from manual implementation. Rejected: manually wire Edit/Window/View items (more code, easy drift); skip non-File menus entirely (Cmd+C/V stop working in text inputs once we replace the default Electron menu).

### Claude's Discretion (not locked)

- **Open Recent label format** — basename only (`MyRig.stmproj`) vs basename + parent dir (`SIMPLE_PROJECT/MyRig.stmproj`) vs full absolute path. Recommend: **basename label + full absolute path as `toolTip`** (Electron MenuItem supports `toolTip`). Falls back gracefully on macOS where tooltips are less common.
- **Empty Open Recent submenu rendering** — disabled placeholder vs hide the entire submenu vs always show a "(empty)" item. Recommend: show submenu always; when empty, render single disabled `(No recent projects)` item + separator + disabled `Clear Menu`. Consistent affordance.
- **Whether menu-driven Open uses a new `reason='menu-open'` SaveQuitDialog discriminator vs reusing `'new-project-drop'`.** Recommend: reuse `'new-project-drop'` — body copy fits both ("This will discard your unsaved overrides"). Only fork if user testing flags the wording as confusing for a menu-initiated open.
- **Menu-rebuild debouncing** — if `notifyMenuState` fires frequently (e.g., during a modal open/close burst or dialog state churn), is debounce needed? Recommend: don't debounce until profiled; menu rebuild is cheap (template construction + `Menu.setApplicationMenu`).
- **Clear Menu confirmation** — single-click clear vs confirm dialog. Recommend: single-click (matches macOS Pages/Keynote/Finder behavior — none of them confirm).
- **IPC channel naming** — `'menu:open-clicked'` vs `'menu:trigger-open'` vs `'menu:dispatch'` with subtype payload. Recommend: per-action channels (`'menu:open-clicked'` etc) — matches Phase 6/7/8 IPC naming style, easier to grep, no payload-discriminator overhead.
- **Where the App.tsx menu-event listener registers its handlers** — top-level `useEffect` in App.tsx vs custom hook (e.g., `useMenuEvents()`). Recommend: top-level `useEffect` for now (Phase 4/8 precedent); promote to hook only if the renderer test file gets unwieldy.
- **`appShellMenuRef` shape** — single ref carrying `{ onClickSave, onClickSaveAs }` vs two refs vs imperative-handle (`useImperativeHandle`). Recommend: single ref carrying both handlers, parallel to `beforeDropRef` from Phase 8.1 D-163 (consistent with project-local pattern).
- **Whether `View` menu in production hides DevTools entirely vs disables it** — Recommend: hide entirely. Production users have no need for DevTools; Reload / Force Reload likewise hidden in packaged builds. View menu becomes empty in production — planner may collapse/omit it conditionally if empty.
- **Platform-specific menu labels** — macOS expects "Quit Spine Texture Manager" in App menu; Win/Linux convention is "Exit" in File menu. The `role: 'appMenu'` macOS App-menu auto-handles "Quit" correctly. For Win/Linux, planner may add an optional File → Exit item via `role: 'quit'` as a polish item, or leave Quit accessible only via Cmd+Q / window close.
- **`recent.ts` testing strategy** — real-fs in a tmpdir vs mocked `node:fs/promises`. Inherits Phase 6 + Phase 8 hybrid (mocked unit cases + one real-bytes integration spec). Planner picks.
- **Menu rebuild on `recent.json` change** — if user successfully Saves As (renderer notifies main of the new path via existing `'project:save-as'` handler), main now needs to rebuild the menu (recent list changed). Wire-up: `addRecent` in `handleProjectSaveAs` after successful write, then rebuild menu via the same `Menu.setApplicationMenu` call.
