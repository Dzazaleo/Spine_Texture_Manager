# Phase 08.2 Plan 04: Renderer Wiring Summary

The renderer side of Phase 08.2 — App.tsx now subscribes to all four menu IPC channels (onMenuOpen / onMenuOpenRecent / onMenuSave / onMenuSaveAs) at the always-mounted root, AppShell.tsx pushes notifyMenuState whenever its modal-state slots change, and the legacy keydown listener (the workaround that broke the 08.1 UAT bug) is deleted end-to-end. After this plan, the canonical Cmd+O reproducer (Cmd+O on the projectLoadFailed recovery banner) is structurally fixed — the menu accelerator dispatches into App.tsx's onMenuOpen handler, which calls window.api.openProject() regardless of AppState because App.tsx is always mounted. Plan 05 verifies via tests; this plan rests on the existing 8.1 reproducer specs continuing to pass.

## What was built

### `src/renderer/src/components/AppShell.tsx` (TOUCHED, +77 net lines after -23 deletion)

**Task 1 — `onClickSaveAs` callback extraction (D-175 split, +32 lines):**

Added a dedicated `onClickSaveAs` useCallback after the existing `onClickSave` (which keeps its smart-save semantics — Save when currentProjectPath set, Save As otherwise per Phase 8 D-141). The new callback ALWAYS calls `window.api.saveProjectAs(...)` regardless of currentProjectPath; updates currentProjectPath + lastSaved on resp.ok; clears stale-override notice on success. Lifted verbatim from PATTERNS.md §"Note: onClickSaveAs does not exist yet" lines 656-676. The toolbar's single Save button (Phase 8 D-141) is preserved unchanged — the new callback is consumed only by the native menu's File→Save As… item via appShellMenuRef.

**Task 2 — keydown listener DELETED + 2 new useEffects + AppShellProps extension (D-176 / D-181 / D-184 / D-175):**

1. **DELETE keydown listener (D-176):** The `useEffect` at lines 626-643 (pre-edit) that registered a `window.addEventListener('keydown', ...)` for Cmd/Ctrl+S/O is removed end-to-end. Replaced by a docstring explaining the deletion rationale (single source of truth = native menu accelerator). Verified by grep: `metaKey`, `ctrlKey`, `window.addEventListener('keydown'`, `window.removeEventListener('keydown'` all absent from the file.

2. **AppShellProps + function signature (D-175):** `AppShellProps` gains an optional `appShellMenuRef?: MutableRefObject<{ onClickSave: () => Promise<SaveResponse>; onClickSaveAs: () => Promise<SaveResponse> } | null>` (parallel to the existing `onBeforeDropRef`). The function signature destructures `appShellMenuRef` alongside `onBeforeDropRef`. `MutableRefObject` and `SaveResponse` were already imported in this file (used by `onBeforeDropRef` and `SaveResponse` references in handlers).

3. **ADD notifyMenuState push useEffect (D-181 + D-184):** Inserted between the existing before-quit useEffect and the onBeforeDropRef registration useEffect. Keyed on `[dialogState, exportDialogState, atlasPreviewOpen, saveQuitDialogState]`. canSave + canSaveAs are always `true` here (summary is non-null while AppShell is mounted); modalOpen is `dialogState !== null || exportDialogState !== null || atlasPreviewOpen || saveQuitDialogState !== null`. Confirmed all four useState slot names against the actual file declarations during execution.

4. **ADD appShellMenuRef registration useEffect (D-175):** Inserted after the existing onBeforeDropRef registration useEffect. `if (!appShellMenuRef) return;` guard mirrors the onBeforeDropRef shape; sets `appShellMenuRef.current = { onClickSave, onClickSaveAs }`; cleanup nulls the ref. Deps: `[onClickSave, onClickSaveAs, appShellMenuRef]`.

### `src/renderer/src/App.tsx` (TOUCHED, +100 net lines)

**Task 3 — appShellMenuRef + 4 menu-event subscriptions + per-AppState push (D-175 / D-183 / D-187):**

1. **SaveResponse import:** Added to the existing `from '../../shared/types.js'` import block alongside SkeletonSummary / SerializableError / LoadResponse / OpenResponse / MaterializedProject.

2. **Declare `appShellMenuRef` (D-175):** Inserted after the existing beforeDropRef + handleBeforeDrop block. Type `useRef<{ onClickSave: () => Promise<SaveResponse>; onClickSaveAs: () => Promise<SaveResponse> } | null>(null)`. Parallel to beforeDropRef shape EXACTLY.

3. **Menu-event subscription useEffect (D-175 + D-183 + the canonical Cmd+O fix):** Inserted after the console.log echo useEffect. Subscribes to all four channels:
   - `onMenuOpen` → `await handleBeforeDrop('', 'stmproj')` (dirty-guard via D-163 ref-bridge); on `proceed === true`, calls `window.api.openProject()` and routes through `handleProjectLoad(resp, '(menu)')`.
   - `onMenuOpenRecent(path)` → `await handleBeforeDrop(path, 'stmproj')`; on proceed, calls `window.api.openProjectFromPath(path)` and routes through `handleProjectLoad(resp, basename)`.
   - `onMenuSave` → `void appShellMenuRef.current?.onClickSave()` (silent no-op when AppShell unmounted).
   - `onMenuSaveAs` → `void appShellMenuRef.current?.onClickSaveAs()` (silent no-op when AppShell unmounted).
   - Cleanup function calls all four unsubs (Pitfall 15 listener-identity discipline).
   - Deps: `[handleBeforeDrop, handleProjectLoad]`.

4. **Per-AppState notifyMenuState push useEffect (D-187 — the recovery banner state push):** Inserted after the menu-event subscription useEffect. Keyed on `[state.status]`. When status is `'idle'` / `'error'` / `'projectLoadFailed'`, pushes `{ canSave: false, canSaveAs: false, modalOpen: false }`. The recovery banner is a `div` with `role="alert"` (NOT `[role="dialog"][aria-modal]`), so File→Open stays enabled — that's the Cmd+O fix end-to-end.

5. **Both AppShell mount points forward `appShellMenuRef`:** The `loaded` branch (line 222) and the `projectLoaded` branch (line 234) both gain `appShellMenuRef={appShellMenuRef}` alongside their existing `onBeforeDropRef={beforeDropRef}`.

### `tests/renderer/save-load.spec.tsx` (mock extended — Rule 3 adapter)

The existing `vi.stubGlobal('api', { ... })` block at lines 44-62 was extended with the 5 new menu-surface mocks: `notifyMenuState: vi.fn()`, plus `onMenuOpen`, `onMenuOpenRecent`, `onMenuSave`, `onMenuSaveAs` — each `vi.fn(() => () => undefined)` to mirror the Pitfall 9 unsubscribe shape. Without these, AppShell's notifyMenuState push useEffect throws "window.api.notifyMenuState is not a function" at mount time.

This is Rule 3 (mechanical adapter) — same discipline Plan 03 used to extend three test fixtures' electron mocks when project-io.ts started eager-importing recent.ts.

## Patterns lifted

- **Imperative ref-bridge (App.tsx ↔ AppShell)** — Phase 8.1 D-163's `beforeDropRef` (App.tsx:178-188 declaration; AppShell.tsx:678-702 registration) → applied verbatim shape to `appShellMenuRef`. Same useRef-on-the-root, useEffect-registration-on-the-shell, dereference-at-call-time discipline.
- **Pitfall 9 listener-identity preservation (consumed)** — the preload's `onMenu*` methods (Plan 03) capture wrapped const for unsubscribe identity; this plan's App.tsx subscription useEffect consumes those returned unsubscribe functions and calls all four in cleanup.
- **Pitfall 15 cleanup discipline** — single useEffect with 4 subscriptions returns a cleanup function that calls all 4 unsubs. Pattern reference: AppShell.tsx:678-702 onBeforeDropRef cleanup.
- **State-derived modal-open** — replaces the deleted `document.querySelector('[role="dialog"]')` DOM heuristic with a useEffect-keyed-on-state-slots derivation. Same shape as the existing isDirty derivation (AppShell.tsx:483-493) that reads from the same useState slots.
- **Per-AppState push useEffect keyed on state.status** — first use in this codebase. Mirrors the existing state-branching pattern at App.tsx:207-289 (the AppState branches in the JSX) but in useEffect form.
- **Late-bind via `.current?.method()`** — defense-in-depth so menu Save / Save As clicks fall through to silent no-op when AppShell is unmounted (idle/error/projectLoadFailed). Main also disables those items in those states (D-181 / D-187), so the IPC shouldn't fire; the optional-chain is the second line of defense.

## Threat model — mitigations applied

- **T-08.2-04-01** (Spoofing: malicious recent.json feeds attacker-controlled path through `menu:open-recent-clicked`): mitigated end-to-end by Plan 03's `handleProjectOpenFromPath` validation (absoluteness, `.stmproj` extension Phase 8 D-145, file existence, sibling-path discipline Phase 1 D-09). App.tsx's onMenuOpenRecent handler forwards verbatim to `window.api.openProjectFromPath(path)` — no additional renderer-side trust assumption.
- **T-08.2-04-02** (Tampering: compromised renderer fakes notifyMenuState): accepted per CONTEXT — even if the renderer pushes false-positive enabled state, clicking File→Save fires `menu:save-clicked` → renderer's onMenuSave handler routes through `appShellMenuRef.current?.onClickSave()` which has its own modal-state defenses. Worst case is misleading menu state, no privileged action.
- **T-08.2-04-03** (DoS: notifyMenuState flood from rapid modal toggling): accepted per CONTEXT — useEffect is keyed on the small set of modal state slots; no debouncing applied.
- **T-08.2-04-04** (Information Disclosure: deleted keydown listener removal): accepted — keystrokes Cmd+O / Cmd+S in the renderer now fall through to default browser handling (no-op in Electron without the menu accelerator). The native Menu accelerator (Plan 02) is the only active path.
- **T-08.2-04-05** (Repudiation: menu Open with cancel): accepted — same UX as drag-drop Cancel (Phase 8 D-143).
- **T-08.2-04-06** (Elevation of Privilege: appShellMenuRef.current mutation): accepted — same generic React assumption as Phase 8.1 D-163's beforeDropRef. ref.current is set ONLY by AppShell's own useEffect; no IPC pathway, no DOM, no external surface mutates it.
