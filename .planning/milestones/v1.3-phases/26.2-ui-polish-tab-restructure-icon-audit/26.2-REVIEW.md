## Warnings

### WR-01: Sub-toolbar `<nav>` is not sticky — tabs scroll out of view

**File:** `src/renderer/src/components/AppShell.tsx:1421-1466`
**Issue:** The new sub-toolbar `<nav role="tablist">` is a sibling of `<header>` (correctly resolving AP-01), but the parent `<header>` has `sticky top-0 z-20` while the new `<nav>` has no sticky positioning. The previous design (tab strip inlined in the toolbar) inherited stickiness from `<header>`. After the relocation, when content scrolls vertically, the toolbar remains pinned but the tab strip slides off the top, leaving the user without tab context while reading rows. Sketch-001 variant A (referenced in the comment block) likely intends the sub-toolbar to stick directly below the main toolbar — otherwise the relocation introduces a UX regression vs. the prior behavior.

**Fix:** Either confirm the design intent matches (tabs scrolling away is acceptable) or pin the sub-toolbar:
```tsx
<nav
  role="tablist"
  className="sticky top-[var(--toolbar-height)] z-10 flex gap-1 items-center bg-panel border-t border-border px-6"
>
```
A simpler fix wraps both rows in a single sticky container so the nav inherits stickiness without needing to know the toolbar height. Phase 26.2 should explicitly UAT this scroll behavior before treating the diff as merged.

### WR-02: `<nav role="tablist">` lacks an `aria-label` to disambiguate the tablist

**File:** `src/renderer/src/components/AppShell.tsx:1421-1424`
**Issue:** A `tablist` with no accessible name (no `aria-label`, no `aria-labelledby`) gives screen-reader users only the generic announcement "tab list, 2 tabs". When the rest of the document grows additional landmarks/lists, this tablist becomes one of several anonymous regions. The pre-existing in-toolbar `<nav>` had the same gap, so this is not a 26.2 regression — but the diff is the perfect window to fix it because the element was just rewritten. Combined with the `<nav>` element's implicit role being **overridden** by the explicit `role="tablist"` (so neither nav-list landmark navigation nor tablist navigation announces a name), screen-reader UX here is poor.

**Fix:** Add an accessible name on the new `<nav>`:
```tsx
<nav
  role="tablist"
  aria-label="View"
  className="flex gap-1 items-center bg-panel border-t border-border px-6"
>
```
Or label by an existing visible element via `aria-labelledby="…"` if a heading exists.

## Info

### IN-01: `<nav role="tablist">` mixes ARIA semantics — `<div>` is more idiomatic

**File:** `src/renderer/src/components/AppShell.tsx:1421-1424`
**Issue:** Using a `<nav>` element with `role="tablist"` overrides the implicit `navigation` landmark role with `tablist`. The element is no longer a navigation landmark to assistive tech, so the choice of `<nav>` is purely cosmetic at the HTML level. The standard ARIA pattern is `<div role="tablist">`. Pre-existing pattern (carried over from the original code), but flagged here because the element was just rewritten.

**Fix (optional):**
```tsx
<div
  role="tablist"
  aria-label="View"
  className="flex gap-1 items-center bg-panel border-t border-border px-6"
>
```
No behavioral change; only HTML semantics cleaner.

### IN-02: Wrapping span carrying `aria-label="Missing PNG"` has no role and is not focusable

**File:** `src/renderer/src/panels/GlobalMaxRenderPanel.tsx:458-464` and `src/renderer/src/panels/AnimationBreakdownPanel.tsx:700-706`
**Issue:** Both missing-row sites render the warning-triangle icon inside a `<span aria-label="Missing PNG">`. Because the span has no `role` (defaulting to a generic-text/inline role) and is not focusable, most screen readers ignore `aria-label` on it. The `WarningTriangleIcon` itself sets `aria-hidden="true"`, so the icon is correctly hidden — but the intended "Missing PNG" announcement is lost. Pre-existing issue; not introduced by Phase 26.2 (the diff only swapped the inner SVG content). Flagged for the record so it's not lost in the next a11y pass.

**Fix:** Either give the span a generic role and make it focusable for screen readers, or use a `<span role="img" aria-label="Missing PNG">` wrapper:
```tsx
<span
  role="img"
  aria-label="Missing PNG"
  className="inline-flex items-center justify-center w-4 h-4 text-danger mr-1 flex-shrink-0"
>
  <WarningTriangleIcon className="w-4 h-4" />
</span>
```
`role="img"` on a wrapper span causes screen readers to announce the `aria-label` text as if it were `<img alt="…">`.

### IN-03: Module-doc comment in `UnusedAssetsPanel.tsx` still says "orphaned" while user-facing copy is now "unused"
